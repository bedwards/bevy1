# What is Bevy?

Before we can talk about Bevy, we need to talk about what a game engine actually is, because the term gets used so loosely that it can mean almost anything. At its most fundamental level, a game engine is a collection of software that handles the repetitive, difficult, low-level work that nearly every video game needs done. Think about what a game has to do, moment by moment: it has to draw images to the screen dozens of times per second, it has to detect when two objects collide, it has to play sounds at the right moments, it has to read input from a keyboard or a gamepad, and it has to do all of these things simultaneously without stuttering or falling behind. Every single game, whether it is a two-dimensional puzzle game or a sprawling open-world adventure, needs solutions to these same problems. A game engine exists so that each new game does not have to solve them from scratch.

The oldest game engines were proprietary, internal tools built by studios for their own titles, and they were never intended to be shared. Over time, some of these engines became products in their own right. Id Software's Quake engine, for example, was licensed to other studios in the late 1990s, and that model of engine licensing eventually gave rise to the commercial engines that dominate the industry today. Unity Technologies released Unity in 2005, Epic Games evolved its Unreal Engine into a publicly available tool, and more recently the Godot Engine emerged as a community-driven open source alternative. Each of these engines represents a different set of tradeoffs around cost, capability, openness, and philosophy, and understanding those tradeoffs is essential context for understanding where Bevy fits into the picture.

Unity and Unreal are the two most widely used game engines in the world. Unity became enormously popular because it was approachable: it provided a visual editor, it used C-sharp (C#) as its scripting language, and it made it relatively easy for a solo developer or a small team to get a game running on multiple platforms. Unreal, by contrast, has historically targeted higher-fidelity projects, offering a powerful rendering pipeline and using C++ as its primary language alongside a visual scripting system called Blueprints. Both engines are backed by large corporations, and both have source code that can be viewed under certain license terms, but neither is truly open source in the way that phrase is understood by the free software community. Their licenses come with revenue-sharing clauses, usage restrictions, or terms that can change at the discretion of the company that owns them, as Unity's controversial runtime fee announcement in 2023 made painfully clear to developers who had built their livelihoods on the platform.

Godot occupies a different position. It is a fully open source engine released under the Massachusetts Institute of Technology (MIT) license, which means anyone can use it, modify it, and distribute it without paying royalties or worrying about license changes. Godot uses its own scripting language called GDScript, along with optional support for C-sharp and other languages through bindings. It has a dedicated visual editor, a passionate community, and a development model driven by donations and grants rather than corporate revenue. For many developers, Godot represents freedom from the uncertainty of proprietary licensing, and its popularity has grown rapidly, particularly after the Unity pricing controversy pushed developers to seek alternatives.

Bevy enters this landscape as something genuinely different from all of the above. It is a game engine written in the Rust programming language, it is open source under a dual MIT and Apache 2.0 license, and it takes a radically modular, data-driven approach to engine architecture that sets it apart not just from Unity and Unreal but from Godot as well. To understand what makes Bevy distinctive, we need to look at three things in sequence: the language it is built in, the architectural pattern it is built around, and the community philosophy that shapes how it evolves.

## Why Rust Matters for Games

Rust is a systems programming language that first reached its stable 1.0 release in 2015. It was designed to provide the performance characteristics of C and C++ — direct control over memory, no garbage collector, predictable runtime behavior — while eliminating entire categories of bugs that have plagued software written in those older languages. The most famous of Rust's innovations is its ownership and borrowing system, a set of compile-time rules that guarantee memory safety without the need for a garbage collector. In practical terms, this means that many of the crashes, data races, and subtle corruption issues that C++ game engines have historically struggled with are simply impossible to express in valid Rust code. The compiler catches them before the program ever runs.

This matters enormously for game development. Games are among the most demanding software applications in existence. They must process physics simulations, artificial intelligence (AI) logic, audio mixing, network synchronization, and rendering commands all within the span of a single frame, which at sixty frames per second means roughly sixteen milliseconds of total budget. Languages with garbage collectors, like C-sharp or Java, periodically pause execution to reclaim unused memory, and those pauses can cause visible stutters in a game. Languages without memory safety guarantees, like C++, avoid those pauses but introduce the risk of use-after-free bugs, buffer overflows, and data races that can be extraordinarily difficult to diagnose. Rust threads the needle between these two worlds: it gives you the performance of C++ and the safety guarantees that are typically associated with much higher-level languages, and it does so at compile time rather than at runtime, meaning there is no performance cost to the safety.

Beyond memory safety, Rust brings a modern tooling ecosystem that is remarkably pleasant to work with. Its package manager and build system, called Cargo, handles dependency management, compilation, testing, and documentation generation in a unified workflow. Its crate ecosystem (a crate being Rust's term for a library or package) is centralized on crates.io, making it straightforward to find, evaluate, and integrate third-party code. And its compiler error messages are famously helpful, often telling you not just what went wrong but what you probably meant to do instead. For a game engine that aspires to be modular and extensible, this ecosystem is a tremendous asset, because it means that plugins and extensions can be distributed, versioned, and integrated using the same tools that every Rust developer already knows.

## The Entity Component System

The architectural heart of Bevy is a pattern called the Entity Component System, almost always abbreviated as ECS. To appreciate what ECS offers, it helps to first understand the approach it replaces. Traditional game engines, including Unity for most of its history, use an object-oriented model built around the concept of a game object. A game object is a thing in your game world — a player character, an enemy, a tree, a particle effect — and it carries with it a bundle of behaviors and data defined through inheritance or through components attached in an editor. In Unity, for instance, you might have a GameObject with a Transform component for its position, a MeshRenderer for its visual appearance, and a custom script that defines its behavior. This model is intuitive because it maps neatly onto how we naturally think about the world: there are things, and those things have properties and behaviors.

The problem with this model emerges at scale. When you have tens of thousands of entities, the object-oriented approach tends to scatter their data across memory in ways that are hostile to modern central processing unit (CPU) cache architectures. Each game object is its own allocation, its components may live in entirely different regions of memory, and iterating over all entities that share a particular set of components requires chasing pointers through a tangled web of references. This is not just an abstract concern; it has direct, measurable consequences for frame rates and the number of entities a game can support.

ECS takes a fundamentally different approach. Instead of bundling data and behavior together inside objects, ECS separates them into three distinct concepts. Entities are lightweight identifiers, essentially just numbers, that represent things in the game world. Components are plain data structures — a position, a velocity, a health value, a sprite reference — that can be attached to entities in any combination. Systems are functions that operate on entities based on which components they have. A movement system, for example, might query for all entities that have both a Position component and a Velocity component, and then update each position by adding the velocity to it. The system does not care whether those entities are players, enemies, or projectiles; it only cares about the presence of the relevant data.

This separation has profound consequences for performance and for the way you think about your game. Because components of the same type are stored together in contiguous arrays, iterating over them is extraordinarily cache-friendly, which translates directly into speed. Because systems declare exactly which components they read and write, the engine can automatically determine which systems are independent of one another and run them in parallel across multiple CPU cores without the programmer having to manage threads manually. And because behavior is defined by the intersection of data rather than by the identity of objects, the architecture naturally encourages composition over inheritance, making it easy to create new kinds of entities simply by combining existing components in novel ways.

Bevy's ECS implementation is one of the fastest in any language, and it is deeply integrated into every layer of the engine. Rendering, audio, input, user interface (UI) layout, and asset loading all flow through the same ECS world, which means there is no impedance mismatch between the engine's internals and the code you write for your game. Everything is a system, everything operates on components, and the engine schedules and parallelizes all of it for you. This uniformity is one of Bevy's most compelling qualities: once you understand the ECS pattern, you understand how the entire engine works, because there is only one pattern to learn.

## Modularity, Plugins, and the Open Source Model

Bevy is not a single monolithic program. It is a collection of loosely coupled crates, each providing a specific capability — rendering, audio, windowing, input, asset management, scene serialization, and so on — that are composed together through a plugin system. When you create a Bevy application, you start with an App builder and add plugins to it. The DefaultPlugins bundle gives you a fully featured engine with rendering, audio, and input, but you are free to replace any individual plugin with your own implementation, to omit plugins you do not need, or to add third-party plugins from the community. This is not modularity as a marketing term; it is modularity as a structural reality. The rendering backend is a plugin. The windowing system is a plugin. If you wanted to, you could strip Bevy down to nothing but its ECS scheduler and build an entirely different kind of application on top of it, and people have done exactly that for simulations, creative coding tools, and server-side game logic.

The plugin ecosystem that has grown up around Bevy is one of its most vibrant features. Community-maintained plugins provide capabilities ranging from physics engines like Avian (formerly known as bevy_rapier wrappers and then heron) to networking layers, tilemap renderers, debug inspectors, editor prototypes, and integrations with external tools. The Bevy Assets website catalogs hundreds of these plugins, and because they are distributed as ordinary Rust crates, integrating them is usually as simple as adding a line to your Cargo.toml file and calling app.add_plugins() with the new plugin. This composability is a direct consequence of both the plugin architecture and the Rust ecosystem's mature approach to dependency management.

Bevy's open source model goes beyond simply publishing the code under a permissive license. The project is developed in the open on GitHub, with a request for comments (RFC) process for significant changes, a robust continuous integration (CI) pipeline, extensive documentation efforts, and a welcoming community on Discord that regularly helps newcomers navigate the engine's learning curve. The project has hundreds of contributors, and its governance is structured around a small core team with a benevolent dictator for life (BDFL) model led by the engine's creator, Cart (Carter Anderson). Financial sustainability comes through sponsorships and donations, and the project has been transparent about its funding and its goals. This is a meaningfully different relationship between an engine and its users than the one offered by Unity or Unreal. There is no corporation that can unilaterally change the terms of your license. There is no runtime fee that might appear in a future update. The code is yours, permanently, under terms that will never change.

The rapid release cadence is both a strength and a source of friction. As of January 2026, Bevy is at version 0.18, with new releases arriving roughly every three months. Each release tends to bring substantial new features and improvements — better rendering capabilities, performance optimizations, API refinements, new platforms — but it also tends to bring breaking changes, because the engine has not yet committed to a stable application programming interface (API). This means that upgrading from one version to the next often requires updating your code to match new function signatures, renamed types, or restructured modules. The Bevy project publishes migration guides for each release, and the community is generally helpful with the transition, but it is a real cost that you should factor into your decision. The engine is moving fast precisely because it is not yet constrained by backward compatibility guarantees, and that speed of evolution is what allows it to improve as dramatically as it does from one release to the next.

This brings us to the central tradeoff of choosing Bevy today. You are choosing an engine that is not yet at version 1.0, which means you are accepting a degree of instability and incompleteness that you would not face with Unity or Unreal or even Godot. There are features that are still missing or experimental. The visual editor, a tool that most other engines consider essential, is still in development. Documentation, while improving steadily, does not yet match the depth of engines that have been around for a decade or more. Some third-party plugins lag behind the latest engine release, creating version compatibility headaches. And the pool of Bevy developers you could hire for a commercial project is smaller than the pool of Unity or Unreal developers, simply because the engine is younger and the language is less widely known.

And yet, there are compelling reasons to choose Bevy despite all of this. If you value performance and safety, Rust gives you both without compromise, and Bevy's ECS lets you exploit modern multicore hardware in ways that traditional engines make difficult. If you value ownership and freedom, the open source license and community governance model mean you will never be at the mercy of a corporate policy change. If you value architectural clarity, Bevy's commitment to a single, consistent ECS pattern means there is less magic, less hidden state, and less friction between what the engine does internally and what your code does on top of it. If you are building a game that needs to handle large numbers of entities — a simulation, a strategy game, a procedurally generated world — the data-oriented architecture gives you a performance ceiling that object-oriented engines struggle to match. And if you enjoy the process of building software as much as the end result, there is a particular satisfaction in working with an engine where you can read every line of the source code, understand exactly what it does, and contribute improvements back to the project that benefits everyone.

The difference between a compiled engine like Bevy and an engine that relies on interpreted or just-in-time compiled scripting languages is worth dwelling on for a moment. When you write game logic in Unity's C-sharp, that code is compiled to an intermediate language and then further compiled or interpreted at runtime by the Mono or IL2CPP (Intermediate Language to C++) backend. When you write GDScript in Godot, it is interpreted at runtime. There are performance implications to both approaches, but the more important distinction is about the development workflow. Interpreted and just-in-time compiled languages allow for faster iteration cycles — you can often change a script and see the result almost immediately, without waiting for a full recompilation. Rust, being an ahead-of-time compiled language, requires a compilation step that can take anywhere from a few seconds to over a minute depending on the size of your project and whether it is an incremental or full rebuild. Bevy and the Rust ecosystem have invested heavily in reducing compile times through techniques like dynamic linking during development, and the experience has improved substantially over the past few years, but it remains a meaningful difference in workflow compared to engines with scripting language support. The payoff is that once your code compiles, it runs at native speed with all optimizations applied, and the compiler has already verified a broad class of correctness properties that scripted languages can only catch at runtime, if they catch them at all.

Bevy is, in the end, a bet on a particular vision of what game development can be. It is a bet that the Rust language's combination of performance and safety will prove to be a lasting advantage, not just a novelty. It is a bet that data-driven architecture, and ECS in particular, represents a better foundation for interactive software than the object-oriented patterns that have dominated the industry for decades. It is a bet that an open source community, governed transparently and funded through voluntary support, can build tools that rival or exceed those built by corporations with hundreds of engineers and billions of dollars. And it is a bet that there are enough developers who care about these things — about understanding their tools deeply, about owning their tools completely, about building on a foundation that is fast and correct and free — to sustain and grow the engine into something remarkable. Whether that bet pays off is still an open question, but the trajectory so far is encouraging, the community is thriving, and the engine gets meaningfully better with every release. For the right kind of developer and the right kind of project, Bevy is not just a viable choice; it is an exciting one.
