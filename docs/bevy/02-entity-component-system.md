# The Entity Component System

There is a certain kind of pain that every programmer who has worked on a large object-oriented codebase knows intimately. It begins innocently enough. You create a class called `GameObject`. It has a position, maybe a name. Then you need a `Character` that can move and has health, so you extend `GameObject`. Then you need a `Player` that can receive input, so you extend `Character`. Then you need a `NonPlayerCharacter` that has dialogue, so you also extend `Character`. And for a while, this feels clean, even elegant. The hierarchy mirrors how you think about the world: a player is a kind of character, and a character is a kind of game object. The trouble is that the world does not actually organize itself into tidy inheritance trees, and the moment your game grows beyond a handful of entity types, the hierarchy begins to buckle under its own weight.

Suppose you want a turret that has health but cannot move. It is not a character, because characters can move, but it needs the health system that lives inside `Character`. You could pull health up into `GameObject`, but then every game object has health, including the ones that should not. You could create a `Damageable` interface or mixin, but now you are fighting the type system to compose behaviors that the inheritance hierarchy was never designed to support. In languages that permit only single inheritance, you run headlong into the diamond problem: if `FlyingCreature` extends `Character` and `SwimmingCreature` also extends `Character`, what does a `FlyingFish` extend? The standard object-oriented answer is either multiple inheritance, which introduces ambiguity about which parent's methods take precedence, or a web of interfaces and delegation that scatters related logic across dozens of files. Neither answer is satisfying, and both tend to produce code that is brittle, difficult to refactor, and almost impossible to reason about when the number of entity types exceeds a few dozen.

This is not a new observation. Game developers have been struggling with inheritance hierarchies since at least the 1990s, and by the early 2000s a different architectural pattern had begun to crystallize in the collective wisdom of the industry. The pattern is called the Entity Component System, often abbreviated ECS, and it represents a fundamental inversion of how you think about the objects in your game. Rather than defining what something is through a chain of parent classes, you define what something has through a collection of independent data attachments. Rather than encoding behavior inside the objects themselves, you extract behavior into standalone functions that operate on whichever objects happen to carry the relevant data. The shift is subtle in description but profound in practice, and it is the architectural foundation on which the Bevy game engine is built.

To understand the Entity Component System, you need to understand its three pillars separately before you can appreciate how they work in concert. The first pillar is the entity. An entity is not an object. It is not a struct with fields, not an instance of a class, not a node in a scene graph. An entity is, in the most literal and reductive sense, just an identifier, typically a number. You can think of it as a row number in a spreadsheet, or a primary key in a database table, or a wristband at a concert that says nothing about the wearer except which wristband they are wearing. The entity itself carries no data and no behavior. It is the anchor point to which everything else attaches, and its emptiness is precisely what makes it powerful. Because an entity has no inherent type, it can become anything, and it can change what it is at any moment during the program's execution, simply by adding or removing the data attached to it.

The second pillar is the component. A component is a piece of pure data, nothing more. It has no methods that operate on itself, no update function, no reference to the entity it belongs to. A component called `Position` might contain an x and a y coordinate. A component called `Health` might contain a current value and a maximum value. A component called `PlayerControlled` might contain no fields at all, serving only as a marker, a tag that says this entity is the one the player is controlling. Components are designed to be small, focused, and orthogonal to one another. `Position` knows nothing about `Health`, and `Health` knows nothing about `PlayerControlled`. They are independent atoms of data that can be freely combined. An entity that has both `Position` and `Velocity` is something that exists in space and moves. Add `Health` and it is something that can be damaged. Add `PlayerControlled` and it is the player's avatar. Remove `PlayerControlled` and add `AiControlled` and it becomes a non-player character, all without changing a single line of code in the systems that process position, velocity, or health.

The third pillar is the system. A system is a function, typically a plain function with no special inheritance or trait requirements, that operates on entities based on which components they have. A movement system might query for all entities that have both `Position` and `Velocity`, and for each one, add the velocity to the position. A damage system might query for all entities that have `Health` and `DamageEvent`, subtract the damage, and remove the event component. A rendering system might query for all entities that have `Position` and `Sprite`, and draw them to the screen. The critical insight is that systems do not know or care about the identity of the entities they process. The movement system does not know whether it is moving a player, an enemy, a projectile, or a gently drifting cloud. It knows only that it has been given a position and a velocity, and its job is to combine them. Behavior, in the Entity Component System, is not a property of objects. It is an emergent consequence of which data an entity carries and which systems happen to match that data.

This is the inversion that solves the problems of deep inheritance. The turret with health but no movement? Give it `Position`, `Health`, and `Turret` components, but no `Velocity`. The movement system will never touch it because it lacks `Velocity`, but the damage system will process it just fine because it has `Health`. The flying fish? Give it `Position`, `Velocity`, `Flying`, and `Swimming` components. A flight system queries for `Flying` and adjusts altitude; a swimming system queries for `Swimming` and applies buoyancy. There is no diamond, because there is no hierarchy. Components are flat, independent, and freely composable. You do not inherit behavior from a parent; you attract behavior by carrying the right data.

## The World and Archetypes

If entities are row numbers and components are column values, then the world is the entire spreadsheet. In Bevy, the `World` is the single, authoritative database that contains every entity and every component in your application. It is the ground truth, the state of the universe. Every system reads from and writes to the world, and the world is what persists between frames, between system executions, between every tick of your application's logic. The world is not a collection of objects with interleaved data; it is a structured store that knows exactly which components exist, which entities carry them, and where in memory they live.

This is where Bevy's implementation becomes especially interesting, because the way the world organizes its data has profound consequences for performance. Bevy uses a concept called archetypes. An archetype is a unique combination of component types. Every entity that has exactly `Position` and `Velocity` and nothing else belongs to one archetype. Every entity that has `Position`, `Velocity`, and `Health` belongs to a different archetype. When you add or remove a component from an entity, that entity moves from one archetype to another. Within a single archetype, all the component data is stored contiguously in memory, packed together in arrays. All the `Position` components for entities in a given archetype are laid out next to each other in one array, and all the `Velocity` components are laid out next to each other in another array, side by side in memory.

This layout matters enormously because of how modern processors access memory. A central processing unit does not fetch individual bytes from main memory; it fetches entire cache lines, typically 64 bytes at a time. When the movement system iterates over entities with `Position` and `Velocity`, and those components are packed contiguously in memory, the processor loads one cache line and gets the positions of several entities at once. The next iteration hits data that is already in the cache. This is called cache-friendly or data-oriented design, and it can be the difference between processing a million entities in a few milliseconds versus tens of milliseconds. In a traditional object-oriented design, where each entity is a separately allocated object on the heap, iterating over a million entities means chasing a million pointers to a million different memory locations, and the cache misses accumulate into a wall of latency. The archetype-based storage in Bevy eliminates this entirely for the common case of iterating over entities that share the same component set.

Archetypes also enable a form of structural typing. When you write a system that queries for entities with `Position` and `Velocity`, Bevy does not scan every entity in the world to check whether it has those components. Instead, it identifies which archetypes contain both `Position` and `Velocity`, and iterates only over the dense arrays within those archetypes. If your world contains a hundred thousand entities but only five hundred of them have both `Position` and `Velocity`, distributed across three archetypes, your movement system iterates over three compact arrays of five hundred elements, not a hundred thousand entities with a conditional check on each one. The archetype is, in effect, a precomputed index that makes queries fast.

## Parallelism and the Shape of Systems

One of the quieter but most consequential benefits of the Entity Component System pattern is how naturally it lends itself to parallel execution. In an object-oriented architecture, determining which operations can safely run in parallel is extraordinarily difficult, because objects hold references to other objects, methods mutate shared state, and the dependency graph between operations is implicit and tangled. In an Entity Component System, the dependency graph is explicit. Each system declares, through its function signature, exactly which components it reads and which components it writes. A system that reads `Position` and writes `Velocity` can run in parallel with a system that reads `Health` and writes `DamageEvent`, because they touch completely disjoint sets of data. Bevy's scheduler analyzes these declarations at startup and automatically parallelizes systems that do not conflict, distributing work across all available processor cores without the programmer writing a single line of threading code.

This is not a theoretical benefit. In a complex game with dozens of systems, the automatic parallelization can mean the difference between a frame that takes sixteen milliseconds and one that takes eight, which is the difference between sixty frames per second and one hundred twenty. The programmer's job is simply to write small, focused systems with narrow component queries, and the engine takes care of running them as efficiently as the hardware allows. The narrower your queries, the more opportunities the scheduler has to run systems in parallel, which creates a virtuous incentive structure where good architectural practices and good performance are aligned rather than in tension.

It is worth pausing to contrast the Entity Component System with other patterns that might seem superficially similar. The actor model, popular in systems like Erlang and Akka, also distributes state across many independent units, but actors encapsulate both data and behavior, communicate through message passing, and are inherently about concurrency and isolation. An actor owns its state and responds to messages; an entity owns nothing and has no behavior. The Entity Component System inverts the actor model by separating data from behavior entirely and centralizing state in the world rather than distributing it. A relational database also stores data in tables with rows and columns, and the analogy to entities and components is a useful one, but a relational database is designed for ad hoc queries, transactional consistency, and persistent storage. The Entity Component System is optimized for a very different access pattern: iterating over every row that matches a particular column set, once per frame, at maximum throughput. You would not build a game loop on top of PostgreSQL, and you would not build a banking system on top of an Entity Component System, because they are tuned for fundamentally different workloads. Finally, the Entity Component System differs from traditional object-oriented programming not merely in syntax or convention but in its foundational assumption about where behavior lives. In object-oriented programming, behavior is a property of the object: the `Player` class knows how to move, how to take damage, how to render itself. In the Entity Component System, behavior is a property of the system: the movement system knows how to move anything, the damage system knows how to damage anything, and the rendering system knows how to render anything. The object is passive data; the system is active logic. This is a Copernican shift in perspective, and like the Copernican shift, it makes many previously intractable problems suddenly tractable.

There is an honest admission to make here, which is that the Entity Component System feels deeply strange when you first encounter it, especially if you have spent years thinking in objects and classes. Your instinct is to reach for a `Player` struct with methods. The idea that a player is not a thing but rather a coincidence of components, an entity that happens to carry `Position`, `Velocity`, `Health`, `PlayerControlled`, and `Sprite`, feels uncomfortably indirect. Where is the player? Everywhere and nowhere. The player is not a single coherent object in any one place in your code; the player is the intersection of a dozen systems that each see a different slice of the player's data. The movement system sees position and velocity. The damage system sees health. The input system sees the player-controlled marker. The rendering system sees position and sprite. No single system has a complete picture of what the player is, and that is by design, because no single system needs a complete picture. Each system needs only the data relevant to its job, and the Entity Component System enforces this minimal-knowledge principle structurally.

The moment this stops feeling strange and starts feeling powerful is the moment you realize that you can add entirely new behaviors to your game without modifying any existing code. Want entities to catch fire? Define a `Burning` component and a fire system that queries for `Burning` and `Health` and applies damage over time. Every entity in your game that has `Health` is now potentially flammable, from the player to the enemies to the destructible crates, and you did not touch a single existing system. In an inheritance-based architecture, adding a `Burning` behavior would require either modifying the base `Character` class, which risks breaking everything that inherits from it, or creating a new branch of the hierarchy, which risks duplicating logic. In the Entity Component System, new behavior is purely additive. You compose by attaching components, not by restructuring hierarchies.

## Bevy's Innovations

Bevy's implementation of the Entity Component System is not merely a textbook rendition of the pattern; it introduces several innovations that address real problems encountered in practice. One of the most significant, introduced in version 0.15, is the concept of Required Components. Before this feature existed, Bevy used a mechanism called bundles, which were predefined groups of components that you would insert together when spawning an entity. If you wanted a sprite, you would insert a `SpriteBundle` that contained a sprite component, a transform, a visibility flag, and several other pieces. The problem with bundles was that they were inflexible and verbose, and they pushed the responsibility of remembering which components were needed onto the programmer rather than the type system. Required Components inverts this by allowing a component to declare that it requires other components to be present. When you insert a `Sprite` component, Bevy automatically inserts `Transform` and `Visibility` as well, because `Sprite` declares them as requirements. If those required components are already present on the entity, they are left untouched; if they are missing, they are added with default values. This eliminates an entire category of bugs where entities were spawned with missing components and silently failed to appear on screen or respond to physics, and it replaces the bundle pattern entirely with something more composable and less error-prone.

Another innovation, arrived in more recent versions, is the support for fallible system parameters and queries that return `Result` types. In earlier versions of Bevy, if a system parameter could not be satisfied, perhaps because it queried for a single specific entity that did not exist, the system would either panic or silently skip execution, and diagnosing the problem required careful reading of log output. With fallible parameters, systems can explicitly handle the case where expected data is missing, returning an error that the engine can report clearly, including the system's name and the nature of the failure. This makes systems more robust and the development experience more forgiving, especially for newcomers who are still building intuition about when and why a query might return nothing.

Perhaps the most architecturally interesting addition is the relationship system, introduced in Bevy 0.16. The classical Entity Component System pattern treats entities as independent, isolated points of data, but real games are full of relationships between entities. A sword belongs to a character. A child entity is attached to a parent entity. A projectile was fired by a particular turret. Before the relationship system, expressing these connections required storing raw entity identifiers inside components and manually managing the bookkeeping of what points to what. The relationship system formalizes this by making entity-to-entity connections a first-class concept. You can express that entity A is a `ChildOf` entity B, and the engine maintains this relationship, provides efficient queries for traversing it, and automatically cleans up when either entity is removed. Relationships turn the Entity Component System from a flat collection of independent entities into something closer to a graph, where the nodes are entities and the edges are typed relationships, without sacrificing the performance characteristics or the compositional flexibility that make the Entity Component System valuable in the first place.

Taken together, these features represent the maturation of the Entity Component System from a clever architectural pattern into a fully realized data model for interactive applications. The world is the database. Entities are the primary keys. Components are the columns. Systems are the queries, except that they execute sixty or more times per second and process results at speeds that would make a database administrator weep with envy. Archetypes are the indices. Required Components are the foreign key constraints. Relationships are the join tables. The analogy is not perfect, as no analogy ever is, but it captures the essential character of the thing: the Entity Component System is a way of organizing data and computation that prioritizes composition over inheritance, data locality over pointer indirection, and explicit dependencies over hidden coupling. It asks you to give up the comfort of the class hierarchy, where every object has a name and a place and a lineage, and to embrace instead a world of anonymous identifiers, flat data, and emergent behavior. The trade is not always easy, but it is almost always worth it, and once you have built a complex system on top of an Entity Component System and felt the freedom of adding new behaviors without touching old code, of parallelizing without thinking about threads, of iterating over a million entities in a handful of milliseconds, it becomes very difficult to go back.
