# Systems and Scheduling

Every game engine faces a fundamental question: how does the game actually do things? In a traditional engine, you might write a massive loop that checks input, updates physics, moves characters, and draws pixels to the screen, all in a single monolithic function that grows unwieldy as your game grows complex. Other engines ask you to attach scripts to objects, where each object carries its own update method that gets called every frame. Bevy takes a different path entirely. In Bevy, the things your game does are expressed as systems, and a system is nothing more exotic than a plain Rust function. There are no special base classes to inherit from, no traits you must implement on some object, no macros you must wrap around your logic. You write a function, you tell Bevy about it, and Bevy calls it at the right time with the right data. This simplicity is deceptive, though, because behind that plain function signature lies a scheduling engine of remarkable sophistication, one that can automatically parallelize your game logic across multiple processor cores, enforce ordering constraints where they matter, and give you fine-grained control over when and whether each piece of logic runs.

The simplest possible system in Bevy is a function that takes no parameters at all. Imagine writing a function called `greet` that simply prints "hello" to the console. In Rust, that looks like `fn greet() { println!("hello"); }`, and you can register it with your Bevy application by calling `.add_systems(Update, greet)` on your app builder. When Bevy runs, it will call `greet` once per frame during the Update phase. That is a complete, working system. It does not query any game data, it does not modify any state, and it serves no practical purpose beyond demonstrating that a system really is just a function. But from this humble starting point, you can build toward systems of extraordinary capability, because the magic of Bevy's system design lies not in the function body but in the function signature -- specifically, in its parameters.

When you add parameters to a system function, you are telling Bevy what data that system needs to do its work. Each parameter type implements a trait called `SystemParam`, which is Bevy's way of knowing how to fetch and provide that data. If your system needs to iterate over every entity that has a `Transform` component and a `Velocity` component, you declare a parameter of type `Query<(&Transform, &Velocity)>`. If your system needs to read a shared resource like the current game time, you declare a parameter of type `Res<Time>`. If your system needs to spawn new entities or despawn existing ones, you declare a parameter of type `Commands`. Bevy inspects these parameter types at app startup, determines what each system reads and writes, and uses that information to build an execution plan. You never call your system function yourself. You never pass arguments to it manually. You declare what you need, and Bevy provides it.

Consider a system that moves entities based on their velocity. You might write `fn move_entities(mut query: Query<(&mut Transform, &Velocity)>)` and then, inside the function body, iterate over the query to update each entity's position. The `&mut Transform` tells Bevy that this system writes to Transform components, while the `&Velocity` tells Bevy that it only reads Velocity components. Now consider a second system that reads Transform components to determine what is visible on screen: `fn render_visible(query: Query<&Transform>)`. Because the first system writes Transform and the second system reads it, Bevy knows these two systems cannot run at the same time safely. It will ensure one finishes before the other begins. But if you had a third system that only reads Velocity components, Bevy recognizes that it has no conflict with the rendering system -- both are only reading, not writing -- and it can schedule them to run simultaneously on different processor cores. This automatic parallelism is one of Bevy's most powerful features, and it emerges entirely from the type information in your function signatures. You do not need to write any synchronization code, manage any mutexes, or reason about thread safety. The type system does the reasoning for you.

Resources deserve special attention because they represent a different kind of data than components. While components are attached to entities and accessed through queries, resources are singular, global values -- there is exactly one instance of each resource type in your application. The `Time` resource, for example, tracks how much time has elapsed since the last frame and since the application started. You access it with `Res<Time>` for read-only access or `ResMut<Time>` if you need to modify it. The distinction between `Res` and `ResMut` matters for the same reason the distinction between `&Transform` and `&mut Transform` matters in queries: it tells the scheduler whether this system is a reader or a writer of that particular piece of data, which in turn determines what can run in parallel.

Events provide yet another communication channel. Where queries let you pull data from entities and resources let you access global state, events let systems send messages to each other across frames. You read events with an `EventReader<MyEvent>` parameter and send them with an `EventWriter<MyEvent>` parameter. A common pattern is a collision detection system that writes `CollisionEvent` values, and a separate sound-effect system that reads those events and plays audio accordingly. Events decouple the producer from the consumer -- the collision system does not need to know that sounds should play, and the sound system does not need to know how collisions are detected. They communicate through a shared event type, and Bevy handles the buffering and delivery.

Commands round out the most commonly used system parameter types. A `Commands` parameter lets you make structural changes to the world: spawning new entities, despawning existing ones, inserting or removing components on entities, and inserting or removing resources. These operations are deferred rather than immediate, meaning they do not take effect inside your system function. Instead, they are queued up and applied at specific synchronization points in the schedule. This deferral is necessary because applying structural changes while other systems might be reading or writing the same data would violate Bevy's safety guarantees. In practice, you rarely notice the deferral, because Bevy applies commands at well-defined points between system groups, and the mental model of "I asked for this entity to be spawned, so it will exist by the time the next batch of systems runs" holds reliably.

## Schedules and When Systems Run

Not all systems should run at the same time or at the same frequency, and Bevy addresses this through the concept of schedules. A schedule is a named collection of systems that runs under specific circumstances. The most commonly used schedules are Update, Startup, FixedUpdate, PreUpdate, and PostUpdate, though Bevy defines several others and you can create your own.

The Update schedule runs once per frame, and it is where most of your game logic will live. Anything that should happen every time a new frame is rendered -- moving a camera based on player input, updating animation states, checking win conditions -- belongs in Update. The frame rate is not fixed; it varies depending on how much work the engine has to do and how fast the hardware is. This means the time between Update ticks fluctuates, which is why you typically multiply velocities and other rate-of-change values by the delta time from the `Time` resource to ensure smooth, frame-rate-independent behavior.

The Startup schedule runs exactly once, at the very beginning of your application, before the first Update. This is where you set up the initial state of your game world: spawning the player entity, loading assets, creating the user interface layout, inserting initial resource values. Because it runs only once, you do not need to worry about guarding against repeated execution, and it provides a clean conceptual boundary between "setting things up" and "running the game."

FixedUpdate is perhaps the most important schedule to understand correctly, because it behaves differently from the others in a way that matters enormously for certain kinds of game logic. Where Update runs once per frame with a variable timestep, FixedUpdate runs at a fixed interval -- by default, 64 times per second, regardless of the frame rate. If the game is rendering at 120 frames per second, FixedUpdate will run roughly once every two frames. If the game is rendering at 30 frames per second, FixedUpdate will run roughly twice per frame to catch up. This fixed cadence is essential for physics simulations, because physics calculations that depend on a consistent timestep produce wildly different results when the timestep varies. A ball falling under gravity will follow a subtly different trajectory at 30 frames per second than at 144 frames per second if you use a variable timestep, but it will follow the exact same trajectory in FixedUpdate regardless of the frame rate. Networking code also benefits from FixedUpdate, since synchronized multiplayer simulations need deterministic tick rates. If your system models something where consistency and determinism matter more than visual smoothness, put it in FixedUpdate. If it needs to respond to every rendered frame for visual fidelity, put it in Update.

PreUpdate and PostUpdate serve as bookkeeping phases that run before and after Update, respectively. Bevy's internal systems use these schedules to synchronize transforms, propagate hierarchy changes, and perform other engine-level housekeeping. You can add your own systems to these schedules when you need to guarantee that your logic runs before or after the main Update phase, but in practice, most application-level code lives in Update, Startup, or FixedUpdate.

## System Sets, Ordering, and Run Conditions

As your application grows, you will accumulate dozens or even hundreds of systems, and you will need ways to organize them. System sets provide this organization. A system set is a named group of systems to which you can apply shared configuration: ordering constraints, run conditions, or both. For instance, you might define a system set called `MovementSystems` that contains all systems responsible for moving entities, and then declare that the entire `MovementSystems` set must run before another set called `CollisionSystems`. This is far more maintainable than specifying pairwise ordering between individual systems, because as you add new movement or collision systems, they automatically inherit the set-level ordering. In Bevy 0.17, the convention for naming system sets shifted to a `*Systems` suffix, so you would see names like `PhysicsSystems`, `InputSystems`, and `RenderSystems` throughout the ecosystem, making it immediately clear at a glance that you are dealing with a set rather than an individual system.

Ordering within a schedule is opt-in. By default, Bevy makes no guarantees about the order in which systems within the same schedule will execute, beyond the constraint that conflicting data accesses will not overlap. If you need system A to run before system B regardless of their data access patterns -- perhaps because A produces a side effect that B depends on through some channel the type system cannot see -- you can declare that ordering explicitly with `.before()` and `.after()` annotations. You can also chain systems in a sequence using `.chain()`, which is syntactic sugar for a series of before and after constraints. The key insight is that ordering is a separate concern from parallelism. Two systems might not conflict in their data access (and thus could safely run in parallel) but still need to run in a specific order for logical correctness.

Run conditions let you control whether a system executes at all during a given tick. A run condition is a function that returns a boolean: if it returns true, the system runs; if false, the system is skipped entirely. The most common use case is game states. You might have a `GameState` enum with variants like `Menu`, `Playing`, and `Paused`, stored as a Bevy state resource. You can then attach a run condition to a system or a system set so that it only runs when the game is in a particular state. For example, your player movement systems should only run during `GameState::Playing`, not while the pause menu is open. Run conditions can also depend on resources, time, or any other data available through system parameters. In Bevy 0.17, what was previously called the `Condition` trait was renamed to `SystemCondition` to more precisely convey its purpose and avoid confusion with the broader concept of conditions in programming. This kind of intentional renaming reflects the Bevy project's commitment to clear, self-documenting APIs (Application Programming Interfaces), even when it means a migration cost for existing users.

The combination of schedules, system sets, ordering constraints, and run conditions gives you a layered system for controlling execution flow. At the broadest level, schedules determine the phase of the frame in which systems run. Within a schedule, system sets group related systems and apply shared constraints. Within or across sets, ordering annotations control sequence. And run conditions gate whether any of it happens at all. Each layer addresses a different concern, and together they replace the tangled web of priorities, execution orders, and manual if-checks that plague less structured approaches to game architecture.

## Hotpatching and the Evolution of the System Model

One of the more remarkable developments in recent Bevy versions is the introduction of hotpatching through a crate called subsecond. Traditionally, changing any game logic in a compiled language like Rust meant recompiling the entire application and restarting it, a process that could take anywhere from a few seconds to a few minutes depending on the size of the project. Hotpatching changes this by allowing you to modify a system function's code, recompile just that function, and have the running application pick up the new behavior without restarting. The game keeps running, the world state is preserved, and your change takes effect almost immediately. This dramatically tightens the iteration loop during development, bringing Rust-based game development closer to the instant-feedback experience of scripting languages or engines with built-in hot reloading. Hotpatching works at the system level because systems are already self-contained functions with well-defined interfaces -- Bevy knows what data they access and how to call them, so swapping one implementation for another is a natural operation. This would be far harder in an architecture where game logic was spread across intertwined methods on long-lived objects.

It is worth pausing to consider how Bevy's system and scheduling model compares to what developers encounter in other engines, particularly Unity, since many game developers come to Bevy with Unity experience. In Unity, game logic lives in MonoBehaviour scripts attached to GameObjects. Each MonoBehaviour can implement an `Update` method that runs every frame and a `FixedUpdate` method that runs at a fixed timestep, superficially similar to Bevy's Update and FixedUpdate schedules. But the similarities are shallow. In Unity, each MonoBehaviour's Update runs independently, with no automatic parallelism -- Unity's traditional scripting model is single-threaded, and while Unity has introduced the Data-Oriented Technology Stack (known as DOTS) and the job system for multithreaded workloads, these require a fundamentally different programming style from standard MonoBehaviours. In Bevy, parallelism is the default. You do not opt into it; you get it automatically, and you opt out (by declaring mutable access) only when you genuinely need exclusive access to some data. Furthermore, Unity's MonoBehaviour model ties logic to objects. If you want to change how all enemies move, you modify the script attached to the enemy prefab. In Bevy, a movement system operates on all entities with the relevant components, regardless of what kind of entity they are. An enemy and a friendly non-player character (NPC) with the same Velocity and Transform components will both be moved by the same system. This decoupling of logic from identity is a core philosophical difference, and it leads to code that is more composable and easier to refactor.

Compared to a hand-written game loop -- the kind of thing you might build if you were writing a game from scratch without an engine -- Bevy's scheduler provides two main advantages. First, it handles parallelism for you. Writing a correct multithreaded game loop is notoriously difficult, requiring careful synchronization to avoid data races, deadlocks, and heisenbugs that only manifest under specific timing conditions. Bevy's type-driven scheduling eliminates entire classes of these bugs by construction. Second, the scheduler provides a declarative model for execution flow. Instead of imperatively writing "do A, then do B, then do C," you declare individual systems and their constraints, and Bevy figures out a valid execution order. This means you can add, remove, or reorganize systems without restructuring a monolithic loop. Each system is a self-contained unit of logic that can be reasoned about independently, tested independently, and reused across different applications.

The mental model to carry away is this: systems are where your game logic lives, and each system is a small, focused function that declares what data it needs through its parameter types. The scheduler reads those declarations, builds a dependency graph, and executes systems with maximum parallelism while respecting data-access constraints and any ordering you have specified. Schedules partition systems into phases that run at different times or frequencies. System sets group systems for shared configuration. Run conditions gate execution based on runtime state. And the whole apparatus is designed so that the common case -- writing a function that does one thing well and letting Bevy figure out the rest -- remains simple, even as the framework supports arbitrarily complex scheduling scenarios for those who need them. When you sit down to write a new piece of game logic in Bevy, you should not be thinking about threads, or frame phases, or execution order, at least not at first. You should be thinking about what data you need and what you want to do with it. Write that function. Register it as a system. Let Bevy handle the rest. And if you later need to constrain when or in what order it runs, the tools are there, layered and composable, waiting for you to reach for them only when you need them.
