# Resources and State Management

When you first encounter Bevy's Entity Component System, it is natural to think of components as the only way to store data. After all, every tutorial begins by attaching a `Transform`, a `Sprite`, or a `Health` component to an entity, and you quickly develop an intuition that data in Bevy lives on entities. But there is an entire category of data that does not belong to any particular entity — data that exists exactly once in your entire application, data that describes the world itself rather than any individual thing inside it. Bevy calls this kind of data a Resource, and understanding the distinction between resources and components is one of the first conceptual leaps that separates someone who is merely following a tutorial from someone who truly understands how to architect a Bevy application.

Consider a simple example. Your game has a score counter. The score does not belong to the player entity, nor to any enemy, nor to any user interface widget. The score is a property of the game session as a whole. It would be awkward and misleading to attach a `Score` component to some arbitrary entity just so you have a place to store it. The same reasoning applies to window configuration — the width and height and title of your application window are not attributes of any entity in your game world. Physics settings like gravity strength or simulation step size describe the rules of your universe, not any object within it. Audio volume, network connection status, the path to a save file, the current difficulty level — all of these are singleton data, meaning there is exactly one instance of each, and they apply globally. This is what resources are for.

A component, by contrast, is data that can exist in multiple instances, one per entity, and that gains its meaning from the entity it is attached to. A position is a component because every entity in your scene might have its own position. A health bar is a component because each character has its own health. A sprite is a component because dozens or hundreds of entities might each display their own image on screen. The distinguishing question is straightforward: does this data describe a specific entity, or does it describe something about the world at large? If the former, make it a component. If the latter, make it a resource.

Defining a resource in Bevy is syntactically simple. You create a struct, derive the `Resource` trait, and insert it into the world, typically during your application's setup phase using `app.insert_resource(...)` or by initializing it with `app.init_resource::<T>()` if your type implements `Default`. Once a resource exists in the world, any system can request access to it through special system parameters. Bevy provides two flavors of access: `Res<T>` for immutable, read-only access, and `ResMut<T>` for mutable access that allows you to modify the resource's contents. This distinction is not merely stylistic — it is load-bearing for Bevy's scheduling system, which uses the declared access patterns of every system to determine which systems can safely run in parallel. If two systems both request `Res<Score>`, they can run simultaneously because neither will modify the score. But if one system requests `ResMut<Score>`, the scheduler knows it must not overlap that system with any other system that also accesses the score, whether mutably or immutably. This is how Bevy achieves fearless concurrency without requiring you to write a single line of synchronization code — the type system and the scheduler collaborate to make data races impossible at compile time and schedule time respectively.

It is worth pausing to appreciate how different this is from the way global mutable state works in traditional programming. In a conventional game engine, or really in any large application written in an imperative style, singleton data tends to live in global variables, static fields, or some variant of the Singleton design pattern. The fundamental problem with all of these approaches is that any piece of code, anywhere in the program, can read or write the data at any time, and there is no mechanism to prevent two threads from stomping on each other's changes. Programmers learn to manage this with mutexes, locks, atomic operations, and careful discipline — but discipline is fragile, and the bugs that arise from its failure are among the most difficult to diagnose in all of software engineering. Bevy's resource model eliminates this entire class of problems. You cannot access a resource without declaring your intent in the system's function signature, and the scheduler uses those declarations to build a dependency graph that guarantees correctness. You get the ergonomics of global data — any system can ask for the score and get it — without the hazards. The data is global in scope but controlled in access.

## States and Application Modes

Resources handle singleton data beautifully, but there is a higher-level pattern that many applications need: the concept of modes. A game might be on its main menu, actively playing, paused, or displaying a game-over screen. A productivity application might be in an editing mode, a preview mode, or a settings mode. These are not just pieces of data — they represent fundamentally different configurations of which systems should be running, which entities should be visible, and how input should be interpreted. Bevy addresses this need with its State system.

A State in Bevy is an enum that you define, where each variant represents one mode of your application. You derive the `States` trait on the enum, and Bevy gives you a suite of tools for managing transitions between those modes and for running code in response to those transitions. At its simplest, a state might look like an enum with variants `MainMenu`, `Playing`, `Paused`, and `GameOver`. You register this state with your app and specify the initial variant. From that point forward, Bevy maintains the current state as a special kind of resource, and you can query it, match against it, and trigger transitions.

The real power of states lies in what you can do at transition boundaries. Bevy provides `OnEnter` and `OnExit` schedules for each state variant, which are one-shot schedules that run exactly once when the application enters or leaves a particular state. When the player presses "Start Game" and you transition from `MainMenu` to `Playing`, the `OnExit(MainMenu)` schedule runs, giving you an opportunity to despawn menu entities, stop menu music, and tear down any menu-specific resources. Then the `OnEnter(Playing)` schedule runs, where you can spawn the player character, load the level, start the gameplay music, and initialize any gameplay-specific resources. This pattern keeps your state management code organized and predictable — you do not need to scatter conditional checks throughout every system to determine whether you are on the main menu or in gameplay. Instead, you configure which systems run in which states, and Bevy handles the rest.

One of the most common and tedious tasks during a state transition is cleaning up entities that belong to the state you are leaving. If you spawn a hundred menu buttons when entering the main menu state, you need to despawn all of them when leaving. Doing this manually is error-prone — it is easy to forget an entity, especially as your application grows and new developers join the project. Bevy 0.17 introduced a component called `DespawnOnExit` (previously known as `StateScoped` in earlier versions) that automates this cleanup entirely. When you spawn an entity and attach `DespawnOnExit(MainMenu)` to it, Bevy will automatically despawn that entity whenever the application transitions out of the `MainMenu` state. This is a small feature with enormous practical impact, because it turns entity lifecycle management from a manual bookkeeping chore into a declarative annotation. You simply mark each entity with the state it belongs to, and the engine handles the rest. The rename from `StateScoped` to `DespawnOnExit` in version 0.17 reflects a broader effort to make Bevy's application programming interface (API) names more self-documenting — the new name tells you exactly what the component does without requiring you to look up its documentation.

States can also be combined with run conditions to control which systems execute during which states. Rather than putting an `if` statement at the top of every gameplay system to check whether the game is paused, you can attach a run condition like `in_state(Playing)` to those systems, and Bevy will simply skip them when the state is anything other than `Playing`. This approach is both more efficient — the scheduler can eliminate systems from the execution graph entirely rather than invoking them only to have them immediately return — and more maintainable, because the relationship between a system and the states it applies to is declared up front rather than buried in the system's body.

## Events and Messages as Cross-System Communication

Resources and states cover a great deal of ground, but there is another dimension to managing shared information in a Bevy application: communication between systems. When one system detects that the player has taken damage, how does it inform the system that updates the health bar, the system that plays a sound effect, and the system that shakes the camera? One approach would be to use a resource as a shared mailbox, but this quickly becomes unwieldy, especially when multiple events of the same type can occur in a single frame or when you need to ensure that every interested system has a chance to respond to each event.

Bevy provides two dedicated mechanisms for this kind of cross-system communication, and the distinction between them, clarified in Bevy 0.17, is important to understand. The two mechanisms are Events and Messages, and while they serve related purposes, they differ in a fundamental way: Events are observable, meaning they trigger Observers, while Messages are buffered, meaning they are collected into a queue and read by a `MessageReader`.

An Event in Bevy is a signal that something has happened — a collision occurred, an item was picked up, a level was completed. When you send an event, any Observer that has registered interest in that event type will be triggered to run. Observers are a reactive pattern: you do not poll for events, and you do not iterate over a buffer. Instead, you register a function that says "when this kind of event happens, run this code," and Bevy calls that function for you. This makes events ideal for situations where the response should be immediate and where you want a clean, decoupled relationship between the system that detects something and the systems that respond to it. The observer pattern also means that if no system is interested in a particular event, sending it has essentially zero cost — there is no buffer to fill and no reader to drain.

Messages, by contrast, follow a producer-consumer model. When a system sends a message, it is placed into a buffer. Other systems can then read from that buffer using a `MessageReader`, processing each message in the order it was sent. This is the right tool when you need to process a batch of communications in a specific order, when you want to handle them at a particular point in the schedule rather than reactively, or when you need to ensure that every message is seen by every interested reader. The buffered nature of messages means they persist for a brief window — typically the current frame and the next — giving systems time to read them even if the reader runs after the writer in the schedule.

The distinction between events and messages might seem subtle at first, but it maps onto a real architectural choice. Use events when you want to say "something happened, and anyone who cares should react." Use messages when you want to say "here is a piece of information that specific systems need to process in order." Events are announcements; messages are letters. Events are fire-and-forget signals that trigger behaviors; messages are data packets that flow through a pipeline. In practice, many applications use both, and developing an intuition for which is appropriate in a given situation is part of becoming fluent in Bevy's data flow model.

## Putting It All Together

The interplay between resources, states, events, and messages forms the backbone of data flow and control flow in a Bevy application, and it is worth stepping back to see how they complement each other. Resources provide a home for singleton data that any system can access in a controlled, scheduler-mediated way. States provide a declarative mechanism for organizing your application into modes, with clean transitions and automatic entity lifecycle management through `DespawnOnExit`. Events provide a reactive communication channel where observers respond immediately to signals. Messages provide a buffered communication channel where readers process information in order at their own pace.

Together, these four tools replace the tangled web of global variables, callback registrations, message queues, and state machines that you might build by hand in a traditional application. And they do so in a way that the compiler and the scheduler can reason about, which means that entire categories of bugs — data races, forgotten cleanup, missed notifications, out-of-order processing — are either impossible or dramatically less likely. This is not just convenience; it is a fundamentally different programming model, one where you describe what data exists and how it flows, and the engine figures out how to execute your code safely and efficiently.

When you are deciding how to store and communicate data in your Bevy application, the decision tree is fairly intuitive once you have internalized the concepts. If the data is per-entity — a position, a velocity, a name, a health value — it is a component. If the data is per-world — a score, a setting, a configuration — it is a resource. If you need to signal that something happened and let interested parties react — it is an event. If you need to send structured data from one system to another for ordered processing — it is a message. And if your application has distinct modes with different behaviors and different visible entities — that is a state.

Each of these mechanisms is simple in isolation, but their composition is where Bevy's architecture truly shines. A state transition might trigger an `OnEnter` schedule that inserts a resource, spawns entities with `DespawnOnExit` markers, and sends an event announcing that the new state has begun. An observer reacting to that event might initialize some gameplay system by sending a series of messages. A system reading those messages might update a resource that another system queries every frame. The data flows cleanly from one mechanism to the next, each doing what it does best, and the scheduler ensures that none of these operations conflict with each other.

This layered approach to state management is one of the reasons Bevy scales well from tiny prototypes to large, complex applications. In a small project, you might use a single resource and no states at all. As your project grows, you add states to organize your code into modes, events to decouple your systems, and messages to manage ordered data flow. At no point do you need to refactor your fundamental architecture — you simply layer in the mechanisms you need, when you need them. The resource system, the state system, the event system, and the message system were all designed to work together, and understanding how they fit together is one of the most valuable things you can learn as a Bevy developer. It transforms Bevy from a collection of interesting features into a coherent, principled framework for building interactive applications of any scale.
