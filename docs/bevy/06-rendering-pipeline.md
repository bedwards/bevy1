# The Rendering Pipeline: From Data to Pixels

Every time you look at a game running on your screen, you are witnessing a small miracle of coordination. Sixty or more times per second, the game must answer a deceptively simple question for every single pixel on the display: what color should this one be? The answer depends on where objects are in the virtual world, what shapes they have, what materials cover their surfaces, where lights are shining, and where the camera is positioned. A modern monitor might have over two million pixels, and each one needs a fresh answer every sixteenth of a second. This is rendering, and understanding how Bevy accomplishes it requires starting from the very beginning: what it actually means to turn a description of a world into an image.

A frame is a single still image. When a game runs at sixty frames per second, it produces sixty of these images every second, each one subtly different from the last, creating the illusion of motion much like the individual frames of a film strip. Producing a single frame involves traversing every visible object in the scene, figuring out which pixels on screen it covers, and calculating the color of each of those pixels based on the object's surface properties and the light falling on it. The sheer volume of arithmetic involved in this process is staggering. Consider a simple scene with a few hundred objects, each made of thousands of triangles, lit by several light sources, some casting shadows. For every triangle, you must determine which pixels it covers. For every covered pixel, you must compute how light bounces off the surface at that precise point. Multiply this by two million pixels and sixty frames per second, and you begin to see why rendering is not a job for a general-purpose Central Processing Unit, or CPU.

This is where the Graphics Processing Unit, or GPU, enters the picture. A CPU is designed to be very fast at doing one thing at a time, working through complex branching logic and irregular tasks with great flexibility. A GPU, by contrast, is designed to do many simple things simultaneously. Where a CPU might have eight or sixteen cores, a GPU has thousands of smaller cores, each capable of performing the same mathematical operation on different data at the same time. Rendering is almost perfectly suited to this architecture, because calculating the color of one pixel is independent of calculating the color of the pixel next to it. You can compute them all in parallel. The GPU is, in a very real sense, a machine built specifically for the kind of work that rendering demands.

But a GPU cannot simply reach into your game's data structures and start drawing. There is an elaborate pipeline that must transform entity data, the components and positions and materials that your game logic manipulates, into the precise format the GPU expects. Understanding this pipeline is the key to understanding how Bevy renders anything at all.

## The Two Worlds

One of the most distinctive architectural decisions in Bevy's rendering system is what is sometimes called the dual-world architecture. Bevy maintains two separate Entity Component System worlds that run in parallel: the Main World and the Render World. The Main World is the one you interact with when you write game logic. It contains your entities, your components, your systems that move characters and respond to input and advance game state. The Render World is a separate world, with its own entities and components, dedicated entirely to the task of producing frames. It lives alongside the Main World, but its data is structured not around what makes sense for gameplay, but around what makes sense for feeding the GPU efficiently.

Why bother with two worlds? The answer is parallelism and performance. If the renderer operated directly on the Main World's data, it would need to lock that data while it worked, preventing game logic from running. Alternatively, game logic would need to wait for rendering to finish before it could modify anything. Either way, one of the two would be sitting idle at any given moment, wasting precious time. By splitting into two worlds, Bevy allows game logic in the Main World and rendering work in the Render World to happen simultaneously. While the GPU is busy drawing the current frame using data in the Render World, the CPU can run game logic in the Main World, preparing state for the next frame. This overlap is a significant source of performance, and it is the reason Bevy can keep frame times low even as scenes grow complex.

The separation also brings clarity to the rendering code itself. The Render World's components are designed specifically for GPU consumption. They are often packed into contiguous arrays, formatted to match the layout that shaders expect, stripped of any information that is irrelevant to drawing. This is very different from the way you might structure components in the Main World, where clarity and convenience for the programmer take priority over memory layout. The dual-world design lets each world optimize for its own concerns without compromise.

Of course, the two worlds must communicate. Data flows from the Main World to the Render World through a carefully orchestrated sequence of phases, and understanding those phases is understanding the rendering pipeline itself.

## The Phased Pipeline: Extract, Prepare, Queue, Render

Bevy's rendering pipeline is divided into distinct phases, each with a clearly defined responsibility. Data flows through these phases in order, like an assembly line, transforming from high-level game state into low-level GPU commands.

The first phase is Extract. This is where data crosses the boundary from the Main World into the Render World. During extraction, systems read components from the Main World, things like the positions of entities, their mesh references, their material properties, their visibility flags, and copy the relevant information into the Render World. This copy is deliberate: it creates a snapshot of the game state at a particular moment, which the renderer can then work with independently while the Main World moves on to compute the next frame's game logic. Extraction is designed to be as fast as possible, because it is the one moment where the two worlds must synchronize. The less time spent here, the more time both worlds have to do their real work in parallel.

The second phase is Prepare. Once the Render World has its snapshot of the relevant data, it must get that data ready for the GPU. This means uploading mesh vertex data into GPU buffers, writing material properties into uniform buffers, computing transformation matrices, and generally translating the engine's internal representations into the binary formats that the graphics hardware understands. The Prepare phase is where Bevy does the heavy lifting of marshaling data across the CPU-to-GPU boundary. If you have ever wondered where the cost of changing a material or moving a mesh actually lands in the frame, much of it lives here.

The third phase is Queue. With data uploaded and ready, Bevy must now decide the order in which to draw things. This is not as simple as it might sound. Opaque objects can be drawn in any order and still produce a correct image, because the depth buffer, a GPU-side data structure that tracks the distance of the nearest surface at each pixel, ensures that closer objects occlude farther ones regardless of draw order. But transparent objects must be drawn from back to front, because their colors blend with whatever is already on screen behind them, and the result depends on the order of blending. The Queue phase sorts draw calls, groups objects that share the same material or mesh to reduce the number of state changes the GPU must perform (a process called batching), and generally organizes the work into the most efficient sequence possible. Batching is particularly important because the GPU is fast at drawing many triangles with the same settings, but relatively slow at switching between different settings. Drawing a thousand objects that all use the same material is dramatically cheaper than drawing a thousand objects that each use a different one, even if the total triangle count is identical.

The fourth and final phase is Render. This is where Bevy submits the organized draw calls to the GPU and the GPU actually executes them, rasterizing triangles into pixels, running fragment shaders to compute colors, writing results into framebuffers, and ultimately producing the image that appears on your screen. By the time this phase begins, all the hard decisions have been made. The GPU has its data, the draw order is set, and the work is simply execution.

These four phases, Extract, Prepare, Queue, and Render, repeat every single frame. They form the heartbeat of Bevy's rendering system, a steady rhythm that transforms the living, changing state of your game world into a stream of images fast enough that the human eye perceives fluid motion.

There are important actors that participate in this pipeline, and it is worth understanding each of them. A camera defines the viewpoint from which the scene is observed. In Bevy, you use Camera2d for two-dimensional games and Camera3d for three-dimensional ones. The camera determines the projection, whether perspective (where distant objects appear smaller, mimicking how human eyes work) or orthographic (where objects retain their size regardless of distance, commonly used in two-dimensional games and technical visualizations). As of Bevy 0.18, the RenderTarget, which specifies where the camera's output goes, such as a window, a texture, or an offscreen buffer, is a separate component that you attach to the camera entity rather than a field buried inside the camera's configuration struct. This change makes it easier to swap render targets dynamically and keeps the camera component itself focused on defining the view.

Lights determine how surfaces are illuminated. Bevy provides several kinds. A DirectionalLight simulates a light source infinitely far away, like the sun, casting parallel rays across the entire scene. A PointLight emanates from a single point in space, like a light bulb, radiating equally in all directions with intensity falling off over distance. A SpotLight is similar to a point light but confined to a cone-shaped region, like a flashlight or a stage spotlight. Each of these light types contributes differently to the final appearance of surfaces, and a scene often uses a combination of all three to achieve realistic or stylistically interesting illumination.

Meshes define the shape of objects. A Mesh3d component associates an entity with a particular piece of geometry, a collection of triangles, really, that describes the surface of an object. Meshes can be simple primitives like cubes and spheres, or complex models loaded from files, with thousands or millions of triangles forming detailed characters, environments, and props. The mesh tells the renderer what shape to draw; the material tells it what the surface looks like.

## Physically Based Rendering and the Nature of Materials

A material describes how a surface interacts with light, and this is where Bevy's rendering becomes genuinely beautiful. Bevy uses a technique called Physically Based Rendering, abbreviated PBR, for its default material system. The core idea behind PBR is that instead of using arbitrary artistic parameters to describe how a surface looks, you describe the physical properties of the surface, and the renderer uses those properties to compute how light would actually behave when striking that surface, at least to a very good approximation.

Bevy's StandardMaterial, the default PBR material, exposes several key properties. The base color is simply the inherent color of the surface, what you might think of as its "paint." The metallic property describes, on a scale from zero to one, how much the surface behaves like a metal versus a non-metal (sometimes called a dielectric). Metals reflect light differently from non-metals: a metal's reflections are tinted by its base color, while a non-metal's reflections tend to be the color of the light source itself. The roughness property describes how smooth or rough the surface is at a microscopic level. A perfectly smooth surface (roughness of zero) produces sharp, mirror-like reflections, while a rough surface (roughness of one) scatters reflected light in many directions, producing a soft, diffuse appearance. There are additional properties too, such as emissive color for surfaces that glow, normal maps that fake fine surface detail without adding geometry, and occlusion maps that approximate the subtle shadows that form in crevices, but the core triad of base color, metallic, and roughness captures the essence of PBR.

To appreciate why PBR matters, it helps to understand what came before it. Older rendering approaches used simpler lighting models. Flat shading, the simplest of all, computes a single color for an entire triangle based on the angle between the triangle's surface and the light direction. Every pixel within that triangle gets the same color, which produces a faceted, angular appearance, like looking at a gemstone or a low-polygon model from the early days of three-dimensional games. Phong shading, developed by Bui Tuong Phong in 1975, was a major improvement. It interpolates the surface normal, the direction the surface is "facing," across each triangle, producing smooth gradients of light and dark that make curved surfaces look genuinely curved rather than faceted. Phong shading also introduced the concept of a specular highlight, a bright spot on a surface where light reflects most directly toward the camera, controlled by a "shininess" exponent that the artist would adjust until it looked right. The problem with Phong shading, and with many models that followed it, is that the parameters are essentially arbitrary. The shininess exponent does not correspond to any real physical property. An artist might set one value for a surface that is supposed to be plastic and another for one that is supposed to be metal, but there is no principled reason why those particular values produce convincing results, and they often do not, especially when the lighting conditions change. A material tuned to look good under one light may look wrong under another.

PBR solves this by grounding the calculation in physics. The metallic and roughness parameters correspond to real, measurable properties of real materials. A roughness of 0.05 looks like polished marble under any lighting because the math models what polished marble actually does to light. A metallic value of 1.0 with a gold base color looks like gold whether it is in direct sunlight or dim indoor lighting, because the Fresnel equations and microfacet distribution functions that underpin PBR correctly model how light interacts with a metallic surface at all angles and intensities. This physical grounding means that PBR materials are more portable across scenes and more consistent across lighting conditions, which is an enormous practical benefit for game developers who would otherwise spend countless hours tweaking materials every time they adjusted a light.

## From Rasterization to Ray Tracing, and Where Bevy Stands

Everything described so far, the phased pipeline, the draw calls, the fragment shaders computing pixel colors, falls under the umbrella of rasterization. Rasterization is the technique of determining which pixels a triangle covers and then computing a color for each of those pixels. It is an approximation, sometimes a very clever one, of how light actually works. Shadows in a rasterized renderer are typically computed using shadow maps, which render the scene from the light's point of view and use the resulting depth information to determine which surfaces are illuminated and which are in shadow. Reflections might use environment maps, pre-baked or dynamically rendered cube-shaped textures that approximate what a reflective surface would see if it looked around. Global illumination, the way light bounces between surfaces and fills a room with indirect light, is often faked with lightmaps or screen-space techniques. These approximations are individually imperfect, but collectively they produce images that look remarkably convincing, and they are fast. Rasterization is the reason real-time rendering exists at all. It trades physical accuracy for speed in a way that has proven enormously effective for decades.

At the opposite end of the spectrum is ray tracing. Where rasterization asks "which pixels does this triangle cover?", ray tracing asks "what does the world look like from this pixel?" For each pixel, a ray is cast from the camera into the scene. When it hits a surface, the renderer calculates the lighting at that point. If the surface is reflective, a new ray is cast in the reflection direction. If the surface is transparent, a ray is cast through it with appropriate refraction. To determine shadows, a ray is cast toward each light source; if it hits something on the way, the point is in shadow. This approach naturally handles reflections, refractions, shadows, and global illumination with a unified algorithm, producing images of remarkable physical accuracy. The catch is cost: tracing rays through a scene for every pixel, potentially bouncing each ray multiple times, is vastly more expensive than rasterization. For decades, ray tracing was the domain of offline rendering, used to produce the beautiful images in animated films where a single frame might take minutes or hours to compute, not games where a frame must be produced in under seventeen milliseconds.

Modern GPU hardware has begun to change this equation. Dedicated ray tracing cores on recent GPUs can accelerate the intersection tests that are the bottleneck of ray tracing, making real-time ray tracing feasible for at least some effects, if not yet for full scenes at high resolutions and high frame rates. Bevy has been moving along this spectrum. Starting with version 0.16, Bevy introduced GPU-driven rendering, a technique that shifts much of the work of organizing and dispatching draw calls from the CPU to the GPU itself. In a traditional renderer, the CPU decides what to draw and issues individual draw calls for each object or batch of objects. In GPU-driven rendering, the CPU uploads the entire scene's data to the GPU, and the GPU itself culls invisible objects, sorts draw calls, and dispatches work. This is a better match for modern GPU architectures, which are designed to process enormous amounts of data in parallel but are poorly served by the latency of receiving many small commands from the CPU one at a time. Bevy's adoption of GPU-driven rendering resulted in roughly a three-times speedup for many scenes, a dramatic improvement that makes larger and more detailed worlds feasible.

With version 0.17, Bevy introduced the experimental Solari renderer, a raytraced rendering path. Solari represents a step toward full ray tracing, using hardware-accelerated ray tracing to compute effects like global illumination, reflections, and shadows with greater physical accuracy than rasterization-based techniques can achieve. It is experimental, meaning it is not yet the default and may have limitations in terms of platform support and feature completeness, but its presence signals the direction Bevy is heading. The long-term trajectory of real-time rendering is almost certainly toward more ray tracing and less rasterization, as GPU hardware continues to improve and the quality advantages of ray tracing become achievable at interactive frame rates. Solari gives Bevy developers a way to explore that future today, and it gives the engine a foundation to build upon as the hardware landscape evolves.

It is worth emphasizing that rasterization and ray tracing are not opposites in the sense that one is right and the other wrong. They are points on a continuum of trade-offs between speed and accuracy. Most modern games, including those built with Bevy, use a hybrid approach: rasterization for the bulk of the image, with ray tracing selectively applied to effects where its quality advantage is most visible, such as reflections on smooth surfaces, soft shadows from area lights, or the subtle ambient light that fills a room. Bevy's architecture, with its phased pipeline and its separation of concerns between the Main World and the Render World, is well-suited to supporting this kind of hybrid approach, because the rendering backend can be evolved independently of the game logic that feeds it.

The rendering pipeline, from the extraction of entity data to the final pixel on screen, is among the most intricate subsystems in any game engine. It bridges two very different worlds: the world of high-level game logic, where you think about characters and items and health bars, and the world of GPU computation, where everything is triangles and texture samples and parallel arithmetic. Bevy's contribution is making this bridge as transparent and as performant as possible. The dual-world architecture keeps game logic and rendering from stepping on each other's toes. The phased pipeline provides a clear mental model for how data flows from your components to the GPU. Physically Based Rendering ensures that the images produced are grounded in how light actually works. And the ongoing evolution from traditional rasterization through GPU-driven rendering to experimental ray tracing ensures that Bevy stays at the edge of what is possible in real-time graphics, not as an abstract technical exercise, but in the service of making games that look as good as they play.
