# Building User Interfaces in a Game Engine

Every game, no matter how immersive or visually stunning, eventually needs to talk to its player through flat, two-dimensional overlays. A health bar hugging the bottom of the screen tells you how close you are to death. A settings menu lets you adjust volume and resolution. A dialogue box delivers a quest-giver's monologue in neatly wrapped text. These elements feel mundane compared to the particle effects and physics simulations happening behind them, but they represent one of the most genuinely difficult design problems in game development: how do you build a responsive, styled, interactive layer of interface elements that floats on top of — or weaves into — a real-time scene that is doing something entirely different underneath?

The difficulty is easy to underestimate if you come from web development, where the entire application is the interface. In a browser, you have one layout system, one rendering pipeline, and one input model, all dedicated to the task of presenting interactive documents. The Document Object Model (DOM) is the scene. Cascading Style Sheets (CSS) is the styling language. The browser handles text shaping, accessibility, focus management, scrolling, and a thousand other details that took decades to refine. When you build a game's user interface (UI), you want all of those same capabilities — layout, styling, text rendering, input handling, scrolling, focus traversal — but you need them to coexist with a rendering pipeline that is busy drawing textured meshes, computing lighting, and running animation systems at sixty or more frames per second. The UI is not the main event; it is a guest in someone else's house, and it needs to behave accordingly.

This tension has produced three broad families of solutions across the history of game engines and graphical applications, and understanding all three is essential to understanding why Bevy made the choices it did.

## Immediate Mode, Retained Mode, and the Entity Component System Approach

The first family is immediate-mode graphical user interface (GUI) libraries, the most famous of which are Dear ImGui (originally written for C++) and egui (its spiritual successor in Rust). The core idea behind immediate mode is radical simplicity: you do not build a persistent tree of widgets that the library remembers between frames. Instead, every single frame, you call functions that both describe and draw the interface. You might write something like `if ui.button("Click me").clicked() { do_something(); }`, and that single line creates the button, renders it, checks whether the user clicked it, and returns the result, all in one pass. The next frame, the same code runs again. If you stop calling the function, the button ceases to exist. There is no widget object to clean up, no state to synchronize, no tree to diff. The entire interface is a side effect of your code running each frame.

This approach is intoxicating for developer tools, debug overlays, and editor interfaces. You can throw together a panel showing entity counts, a slider controlling a debug variable, or a window listing every asset in the scene, and you can do it in a handful of lines with zero boilerplate. The reason immediate-mode GUI works so well for these use cases is that developer tools tend to be simple, functionally driven, and tolerant of visual imperfection. You do not care that your debug panel looks generic. You care that you can add a new field to it in thirty seconds.

But immediate-mode GUI has real limitations when you try to use it for a game's player-facing interface. Styling is constrained: you can theme the entire library, but achieving the bespoke, art-directed look that a shipped game demands is difficult. Layout is limited to what the library provides, which is usually a simple top-to-bottom or left-to-right flow. Animation and transitions require you to manage state yourself, defeating the supposed simplicity. And performance can become an issue if you have hundreds of complex widgets, because the library must process all of them every frame whether anything changed or not. Perhaps most importantly, immediate-mode GUI exists in its own rendering world — it draws its own quads and text on top of the scene — which makes it hard to integrate UI elements that need to interact deeply with the game world, like a name tag floating above a character's head or a health bar embedded in the 3D scene.

The second family is retained-mode GUI, which is what web browsers implement and what desktop toolkits like Qt, GTK, and SwiftUI use. In retained mode, you build a persistent tree of objects — nodes, views, widgets — and the framework remembers that tree between frames. When something changes, you mutate the tree (or signal a change through a reactive system), and the framework figures out what needs to be re-laid-out and re-rendered. This is more complex to set up initially, but it maps naturally to the kinds of interfaces players expect: nested containers with flexible layouts, styled text that wraps and scrolls, buttons with hover and pressed states, animated transitions between screens. Retained-mode systems can also be far more efficient when most of the interface is static, because they only recompute layout for the parts that changed.

The cost of retained mode is that you now have a persistent tree that must be kept in sync with the rest of your application's state. In web development, this synchronization problem spawned an entire ecosystem of frameworks — React, Vue, Svelte — each offering its own philosophy for how to bridge the gap between application data and the DOM. In a game engine, the synchronization problem is even thornier, because the authoritative state lives in the game world (entity positions, health values, inventory contents), and the UI tree is a derived view of that state.

Bevy's approach to UI is a third path that leverages the engine's defining architectural feature: the Entity Component System (ECS). In Bevy, UI elements are not objects in a separate widget tree. They are entities, the same kind of entities that represent your player character, your enemies, your terrain, and your particle effects. A button is an entity. A health bar is an entity. A settings menu is a collection of entities arranged in a parent-child hierarchy. What makes these entities behave as UI rather than as game objects is the components attached to them, most fundamentally the `Node` component, which marks an entity as a participant in Bevy's UI layout system and carries layout properties like width, height, padding, margin, alignment, and display mode.

This might sound like retained mode with extra steps, and in a sense it is. Bevy's UI is retained — you spawn entities once, and they persist until you despawn them. You mutate their components to change their appearance or behavior, and the engine's layout and rendering systems pick up those changes automatically. But because the UI lives in the ECS, it inherits all of the ECS's strengths. Queries let you efficiently find and modify UI elements. Systems let you define behavior that runs every frame or in response to events. Change detection lets the engine skip work when nothing has changed. And perhaps most valuably, there is no impedance mismatch between the UI and the game world. A system that updates a health bar can query for both the player's `Health` component and the health bar's `Node` and `Text` components in the same query, because they are all just entities with components in the same world.

## Layout, Styling, and the Shape of a Bevy UI Tree

The layout engine that powers Bevy's `Node` component is called Taffy, a Rust implementation of the Flexbox and CSS Grid layout algorithms. If you have written CSS for the web, the layout model will feel familiar, though the syntax is different because you are setting struct fields on components rather than writing stylesheet rules. You can arrange children in a row or column using Flexbox, control how they grow and shrink to fill available space, align them along the main or cross axis, add gaps between them, and nest containers to create complex layouts. You can also use CSS Grid for two-dimensional layouts with explicit rows and columns. The goal is not to replicate the entirety of CSS — there are no selectors, no cascade, no specificity wars — but to provide the layout primitives that web developers have proven are sufficient for virtually any two-dimensional interface.

Each `Node` entity can have a `BackgroundColor`, a `BorderColor`, a `BorderRadius` for rounded corners, and an optional `ImageNode` for background images. Text is rendered through dedicated text components that support font selection, font size, color, line height, and text wrapping. Since Bevy 0.17, UI positioning uses `UiTransform` rather than the standard `Transform` component that the rest of the engine uses for spatial positioning. The reason for this separation is that UI elements live in a fundamentally different coordinate space than game-world entities. A button's position is determined by the layout algorithm relative to its parent container, not by a position in world space. `UiTransform` allows you to apply additional offsets, rotations, and scaling on top of the computed layout position — useful for animations, tooltips that need manual nudging, or creative effects — without conflicting with the layout engine's output.

Speaking of layout output, Bevy separates the input (what you want) from the result (what you get) through a component called `ComputedNode`. After the layout engine runs, every `Node` entity receives a `ComputedNode` that contains the final resolved position, size, and other layout metrics as read-only values. This separation is clean and intentional. You set your desired width to "fifty percent of the parent" on the `Node`, and the layout engine writes the actual pixel width to `ComputedNode`. Your systems can read `ComputedNode` to know exactly where and how large a widget ended up, which is essential for things like positioning particle effects relative to a UI element or implementing custom hit-testing logic.

Scrolling is handled through overflow properties on `Node`, which can be set to clip content that extends beyond the node's bounds and to enable scroll behavior. Children that overflow a scrollable container can be navigated through scroll events from the mouse wheel or touch input, and the system tracks scroll offsets per-entity, allowing multiple independent scrollable regions.

Text rendering in Bevy's UI deserves special mention because text is one of the hardest problems in computer graphics, and getting it right is essential for any interface that a human being is expected to read. Bevy uses a text pipeline that handles font loading, glyph rasterization, text shaping (the process of converting a sequence of Unicode characters into correctly positioned glyphs, which is far more complex than it sounds for languages with ligatures, combining marks, or right-to-left scripts), and layout including line wrapping. Text nodes participate in the same Flexbox and Grid layout as every other node, meaning text can grow, shrink, and wrap naturally within its container.

## Widgets, Interaction, and the Road From 0.17 to 0.18

For a long time, Bevy's UI provided only the raw building blocks: nodes, images, text, and a basic interaction system that could tell you whether the cursor was hovering over or clicking on an entity. If you wanted a button, you spawned a `Node` entity, gave it a background color, added text as a child entity, and wrote a system that listened for interaction events and changed the background color on hover or press. If you wanted a checkbox, you did the same thing but with additional state tracking. This was flexible but tedious, and it meant every game (and every Bevy learning tutorial) was reinventing the same basic widget logic.

Bevy 0.17 brought a significant change with the introduction of headless widgets through the `bevy_ui_widgets` module. These are components and systems that implement the logic and accessibility semantics of common controls — buttons, sliders, checkboxes, toggle switches — without dictating their visual appearance. A headless button widget knows how to handle click events, keyboard activation, focus state, and screen-reader announcements, but it does not know what color it should be or what font its label should use. This separation of logic from presentation mirrors a philosophy that has gained traction in web development through libraries like Radix and Headless UI: provide the behavior, let the developer provide the look.

Alongside the headless widgets, the Bevy Feathers project arrived as an opinionated styling layer that provides ready-made, visually cohesive widget implementations built on top of the headless primitives. If headless widgets are the unstyled skeleton, Bevy Feathers is a complete outfit — buttons with hover and active states, input fields with focus rings, sliders with track and thumb styling, all following a consistent design language. Developers who want a quick, professional-looking UI can use Bevy Feathers directly. Developers who want a completely custom look can use the headless widgets and style them from scratch. Developers who want maximum control can ignore both and build everything from raw `Node` entities, as before. This layered approach — raw nodes, headless widgets, styled widgets — gives every project the level of abstraction that fits its needs.

Bevy 0.18 continued the expansion of the widget ecosystem. Popovers — floating panels that appear in response to a trigger and position themselves relative to their anchor, handling edge cases like screen-edge collision — joined the toolkit. Radio buttons provided mutually exclusive selection groups, a fundamental interaction pattern that had previously required manual state management. Color planes offered two-dimensional color pickers for applications and tools that need color selection, such as material editors or creative applications. And automatic directional navigation arrived, which is quietly one of the most important features for any game that supports gamepads or keyboard-only input. Directional navigation means that when a player presses up, down, left, or right on a gamepad's directional pad (d-pad) or analog stick, the focus system automatically determines which widget should receive focus next based on the spatial layout of the UI. This is something that web browsers handle natively for tab-key navigation but that games must implement themselves for spatial (directional) navigation, and getting it wrong leads to the frustrating experience of pressing "down" and having the cursor jump to an unexpected widget on the other side of the screen.

## When to Use Bevy's UI, When to Use egui, and How They Coexist

Given that Bevy has its own UI system, you might wonder why the `bevy_egui` crate — which integrates the egui immediate-mode GUI library into Bevy — remains so widely used in the Bevy ecosystem. The answer is that the two systems serve different audiences within the same application, and trying to use one where the other excels leads to unnecessary pain.

Bevy's built-in UI is the right choice for anything the player sees. Main menus, in-game heads-up displays (HUDs), inventory screens, dialogue systems, settings menus, tutorials, and any interface that needs to match the game's art direction — all of these should be built with Bevy's native UI. The reasons are straightforward: native UI entities participate in the ECS, so they integrate naturally with the rest of the game's logic. They can be styled with complete freedom, using custom fonts, colors, images, and layouts that match the game's visual identity. They benefit from the widget system's accessibility and input-handling features, including gamepad navigation. They can be animated through Bevy's animation systems. And they render through Bevy's own rendering pipeline, which means they can potentially interact with post-processing effects, render layers, and other rendering features in the future.

Egui, accessed through `bevy_egui`, is the right choice for developer-facing tools: debug panels, entity inspectors, performance profilers, asset browsers, level editors, and the kind of ad-hoc diagnostic interfaces that you throw together during development and may never ship to players. Egui's immediate-mode approach shines here because the iteration speed is unmatched. You can add a new debug window in a few lines of code, display any value you can format as a string, and tweak it as your debugging needs evolve, all without defining new components, spawning entities, or writing systems. The visual style does not matter because these tools are for you, not for the player. Egui also provides a rich set of built-in widgets — text inputs, color pickers, plots, tables, tree views — that would take significant effort to replicate in Bevy's native UI.

The two systems coexist peacefully. Egui renders its own output as an overlay, and `bevy_egui` handles the integration so that input events are correctly routed — when the cursor is over an egui window, egui consumes the input; otherwise, Bevy's systems receive it. Many Bevy projects use both simultaneously: the native UI for the game's interface and egui for the development tools, with the egui panels hidden in release builds through a simple conditional compilation flag or a runtime toggle.

There is a deeper philosophical point here about the nature of game UI that is worth sitting with. A game's user interface is not a web page. It is not a desktop application. It is a hybrid thing that must be both functional and atmospheric, both informative and immersive, both responsive and beautiful. The health bar is not just a data display; it is part of the game's aesthetic. The settings menu is not just a form; it is the player's first impression of the game's level of polish. The dialogue box is not just a text container; it is the vehicle through which the story reaches the player. Building these things with the same ECS that runs the rest of the game is not just an architectural convenience — it is a statement that the UI is part of the game, not an afterthought bolted on top.

Bevy's UI system is still younger than the browser's DOM, and it does not yet match the web's decades of refinement in areas like accessibility, internationalization, and the sheer breadth of available components. But it is growing rapidly, with each release closing gaps and adding capabilities that make it more viable for production game interfaces. The trajectory from raw nodes to headless widgets to styled widget libraries to automatic gamepad navigation tells a story of a system that is maturing with purpose, building each layer on a foundation that was designed from the start to be part of a game engine rather than adapted from a different domain. For developers building games in Bevy, the practical advice is clear: learn the `Node` component, learn Flexbox (it is the same mental model as the web, just with different syntax), use headless widgets when they fit, reach for Bevy Feathers or a similar styled layer when you want speed, and keep egui in your back pocket for the debug tools that make development bearable. The result is a UI that belongs to the game — not one that merely sits on top of it.
