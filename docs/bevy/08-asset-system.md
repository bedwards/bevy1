# The Asset System: Loading the World Into Memory

A game without assets is nothing more than logic running in a void. Every texture painted onto a surface, every three-dimensional model placed in a scene, every sound effect triggered by a collision, every font used to render a score counter, every shader that determines how light interacts with a material — all of these begin their lives as files sitting on a disk somewhere. The fundamental challenge of any asset system is deceptively simple to state and remarkably difficult to solve well: how do you get those files off the disk and into memory in a form the game can use, without making the player stare at a frozen screen while it happens?

Consider what actually occurs when a game needs to display a texture. The image data might be stored as a PNG or a KTX2 file, compressed and encoded in a format optimized for storage rather than for rendering. The game engine must locate that file, read its bytes into memory, decode the compressed format, potentially convert it into a GPU-friendly layout, upload it to the graphics card, and finally associate it with whatever material or sprite needs it. For a single small texture this process might take a handful of milliseconds — barely noticeable. But a typical game scene might reference hundreds of textures, dozens of meshes, a collection of audio files, and various configuration data. If you tried to load all of that synchronously, blocking the main thread while each file was read and processed in sequence, the game would freeze for seconds or even minutes. Players would see nothing, hear nothing, and likely assume the application had crashed.

The problem compounds when you consider that assets do not only come from local storage. In a web browser, assets must be fetched over HTTP, where latency is measured in hundreds of milliseconds at best and where network interruptions are a fact of life. Even on local storage, solid-state drives have latency, mechanical drives have seek times, and operating systems have file system caches that may or may not contain the data you need. The only honest answer to "how long will this file take to load?" is "we do not know, and we cannot afford to wait around to find out."

This is the problem that Bevy's asset system exists to solve. It provides a principled, asynchronous bridge between the world of files — slow, unpredictable, potentially remote — and the world of live game data, where everything must be available at the speed of a frame update. The design reflects a core conviction: the game loop must never block on input/output. Instead of waiting for an asset to be ready, the system gives you a lightweight token — a promise, in a sense — that you can hold onto while the actual loading happens in the background. When the data finally arrives, it slots into place, and anything referencing that token can begin using it.

## Handles, the AssetServer, and Assets as Storage

At the center of Bevy's asset system sits the AssetServer, a resource that coordinates all loading operations. When you ask the AssetServer to load a file — say, a PNG image that will become a texture — it does not hand you back the image data. It cannot, because the image has not been loaded yet. Instead, it immediately returns a Handle of type T, written in Rust's generic syntax as `Handle<T>`, where T is the type of asset you requested, such as `Image` or `AudioSource` or `Mesh`. This handle is your receipt. It is small, cheap to clone, and safe to store anywhere in your game's data structures. You can pass it to a sprite component, tuck it into a material, or hold it in a resource for later use, all before a single byte of the actual file has been read.

Behind the scenes, the AssetServer dispatches the loading work to background threads (or, on web platforms, to asynchronous fetch operations). The file is located, its bytes are read, and an appropriate asset loader — a trait implementation that knows how to turn raw bytes into a specific Rust type — processes the data. When loading completes, the finished asset is inserted into the `Assets<T>` collection, which is a specialized resource that acts as a typed storage map. The Assets collection is keyed by handles: given a `Handle<Image>`, you can look up the corresponding Image data in `Assets<Image>`. If the asset has not finished loading yet, the lookup simply returns nothing, and your system can gracefully skip that work for now and try again next frame.

This pattern — return a handle immediately, load in the background, store the result in a typed collection — is what makes Bevy's asset system non-blocking. Your game loop keeps running. Your systems keep executing. The player sees the application remain responsive. And when the asset arrives, it simply appears in the collection, ready for use. Many engines use a variation of this pattern, but Bevy's implementation leans heavily on Rust's type system to make the whole process feel natural and to catch errors at compile time rather than at runtime.

Handles in Bevy are reference-counted, which means the system tracks how many active handles point to a given asset. This is where the distinction between strong handles and weak handles becomes important, and it is a distinction worth understanding clearly because it governs the lifecycle of every asset in your game.

A strong handle is what you get back from the AssetServer when you call load. It represents active ownership: as long as at least one strong handle to an asset exists somewhere in your program, that asset will remain in memory. The reference count increments when you clone a strong handle and decrements when a strong handle is dropped. When the last strong handle disappears — perhaps because the entity holding it was despawned, or the component storing it was removed — the reference count reaches zero, and the asset system knows it is safe to free that asset's memory. This is automatic and deterministic. You do not need to manually call an unload function or manage a cache eviction policy. The ownership semantics of Rust, combined with reference counting, handle it for you.

A weak handle, by contrast, is just a reference. It points to the same asset, identified by a Universally Unique Identifier (UUID), but it does not contribute to the reference count. Holding a weak handle does not keep the asset alive. If all strong handles are dropped, the asset will be freed even if weak handles still exist. Weak handles are useful when you want to refer to an asset without claiming ownership of it — for example, in a cache or lookup table where you want to check whether an asset is still loaded but do not want your data structure to be the reason it stays in memory. The UUID-based identification also means weak handles can refer to assets that have not been loaded yet, or that were loaded in a previous run of the application. They are identifiers, not ownership claims.

## The Asset Lifecycle and Hot Reloading

Every asset in Bevy moves through a lifecycle that begins when a load is requested and, if all goes well, ends either when the asset is no longer needed or when the application shuts down. The stages are straightforward: first, the load is requested, and a handle is returned. The asset enters a loading state, during which the file is being read and processed asynchronously. Once processing completes successfully, the asset transitions to a loaded state, and the data becomes available in the `Assets<T>` collection. If loading fails — because the file was not found, or the data was corrupt, or the network request timed out — the system can report an error, and the handle remains valid but will never resolve to usable data.

What makes this lifecycle more interesting than a simple load-once-and-forget model is hot reloading. During development, you frequently want to change an asset — tweak a texture, adjust a model, modify a shader — and see the result in the running game without restarting it. Bevy supports this by watching the file system for changes. When a loaded asset's source file is modified, the system detects the change, re-reads the file, re-processes it through the appropriate loader, and replaces the data in the `Assets<T>` collection. Because everything in the game references the asset through handles, and because handles point into the Assets collection, the updated data automatically propagates everywhere the asset is used. The next frame that reads the asset will see the new version. This is enormously valuable during iteration, turning what would be a stop-restart-navigate-back-to-the-spot workflow into an instant visual update.

Since version 0.16, Bevy introduced a query filter called AssetChanged, which lets systems react specifically when the underlying data behind a handle has been modified. This is distinct from the handle itself changing — the handle stays the same, but the data it points to has been replaced, perhaps by hot reloading or by programmatic modification. AssetChanged lets you write systems that efficiently respond to these updates, re-computing derived data or updating visual state only when the source asset actually changes, rather than polling every frame. It is a fine-grained reactivity mechanism that fits naturally into Bevy's existing query and system model.

An important design decision that arrived in version 0.15 deserves attention here, because it affects how you structure your code. In earlier versions of Bevy, `Handle<T>` was itself a component — you could attach it directly to an entity, and the Entity Component System (ECS) would treat it like any other piece of data on an entity. Starting with 0.15, this is no longer the case. Handles are no longer components. Instead, you are expected to wrap a handle inside your own custom component type. This might seem like an inconvenience at first, but it is a deliberate and well-reasoned separation of concerns. The asset system and the entity model are different subsystems with different responsibilities. An asset handle is a reference to shared, potentially reusable data — many entities might reference the same texture or mesh. A component, on the other hand, is data that belongs to a specific entity. By requiring you to wrap handles in your own component types, Bevy makes the relationship between entities and assets explicit, gives you a natural place to add related metadata, and avoids coupling the asset system's internal types to the ECS's component storage. It is the kind of small structural decision that pays dividends in a large codebase.

Since version 0.17, the asset system gained the ability to load assets directly from HTTP and HTTPS URLs, often referred to as web assets. This means you can point the AssetServer at a remote URL — perhaps a content delivery network hosting your game's assets, or a development server serving work-in-progress art — and the loading pipeline handles the network fetch transparently. The handle you get back behaves identically to one from a local file load. This capability is particularly important for web-targeted games, where all assets necessarily come from the network, but it is also useful for desktop and mobile applications that want to download content on demand rather than shipping everything in the initial install package.

## Three-Dimensional Assets and the glTF Format

While the asset system is agnostic about what types of data it loads — it works equally well with images, audio, fonts, custom binary formats, and anything else you can write a loader for — three-dimensional model data deserves special discussion because of its complexity and because of the specific format Bevy has chosen to prioritize.

GL Transmission Format, abbreviated as glTF and pronounced "gee-ell-tee-eff" by most of the industry, is an open standard maintained by the Khronos Group for representing three-dimensional scenes and models. It has become something of a lingua franca for 3D content interchange, sometimes called "the JPEG of 3D" for its role as a widely supported, reasonably compact, and well-specified format. A glTF file can contain meshes (the geometric shapes that make up a model), materials (descriptions of how surfaces interact with light), textures (images applied to surfaces), skeletal animation data, and even entire scene hierarchies with cameras and lights. It comes in two variants: a JSON-based .gltf file that references external binary buffers and images, and a self-contained binary .glb file that packs everything into a single blob.

Bevy uses glTF as its primary 3D model format, and the integration is deep. When you load a glTF file, Bevy's glTF loader parses the entire structure, creates meshes, materials, textures, and animation clips as separate assets, and assembles them into a scene that can be spawned as a hierarchy of entities. This is a non-trivial operation — a single glTF file might produce dozens of individual assets and entities — and the asset system's asynchronous, handle-based design is what makes it manageable. Each sub-asset gets its own handle, and the overall scene is itself an asset with handles referencing its constituent parts.

One of the practical challenges with glTF is that it is an extensible format. The base specification covers common use cases, but real-world content often includes extensions — additional data blocks that carry information the base spec does not define, such as proprietary material models, physics collision shapes, or custom metadata from a specific digital content creation tool. Bevy 0.18 addressed this with the introduction of GltfExtensionHandler, a mechanism that lets you write custom code to process glTF extensions during loading. Before this, encountering an unknown extension meant the data was simply ignored. With extension handlers, you can intercept that data, interpret it according to whatever specification the extension follows, and incorporate it into your game's data model. This is important for studios that have established content pipelines with custom tooling — they can embed game-specific data in their glTF files and have Bevy pick it up automatically during loading.

## Comparisons with Other Engines

Understanding Bevy's asset system in isolation is valuable, but understanding it in contrast with how other major engines approach the same problem illuminates the design choices more clearly.

Unity, the most widely used commercial game engine, employs what it calls an import pipeline backed by an asset database. When you add a file to a Unity project, the editor does not simply note its existence — it imports the file, processing it through a series of configurable importers that convert it from its source format into Unity's internal representation. This imported version is cached in a database (the Library folder, for those familiar with Unity's project structure), and it is this cached, processed version that the game actually uses at runtime. The advantage of this approach is that expensive processing — texture compression, mesh optimization, shader compilation — happens once during import rather than at load time, which can significantly reduce runtime loading costs. The disadvantage is that the import process itself can be slow, especially for large projects, and the asset database can become a source of friction: it needs to be kept in sync, it can become corrupted, and it imposes Unity's internal formats as an intermediary layer between your source files and your game. Unity's approach also means that the editor is deeply involved in the asset workflow; you cannot easily load assets without the editor having processed them first. At runtime, Unity uses an asynchronous loading system with reference counting, conceptually similar to Bevy's, but layered on top of this import-and-cache foundation.

Godot takes a different path with its resource system. In Godot, almost everything is a resource — textures, meshes, scripts, scenes, audio clips. Resources are loaded by path, cached automatically, and shared by reference. When two nodes reference the same texture path, they get the same resource object in memory. Godot's approach is simpler and more transparent than Unity's: there is less of an import pipeline (though Godot does have import settings for certain asset types), and the relationship between files on disk and resources in memory is more direct. Godot also supports its own scene format (.tscn and .scn files) as a primary way of organizing assets and hierarchies, blurring the line between "asset" and "scene" in a way that differs from both Bevy and Unity. The trade-off is that Godot's resource system, while easy to use, offers less fine-grained control over loading strategies, memory management, and the kind of typed, compile-time-checked handles that Bevy provides through Rust's generics.

Bevy's asset system sits in a distinctive position among these three. It shares Unity's commitment to asynchronous loading and reference-counted lifetime management, but it skips the heavy import pipeline in favor of loading from source formats at runtime (with the option to use pre-processed formats for performance-critical scenarios). It shares Godot's directness — files map fairly straightforwardly to in-memory assets — but it adds the rigor of Rust's type system, which means that a `Handle<Image>` and a `Handle<Mesh>` are fundamentally different types that cannot be confused at compile time, whereas in a dynamically typed engine, a resource reference is just a reference until you try to use it and discover it points to the wrong kind of data. Bevy's system also benefits from being designed after the others, with the advantage of hindsight: the separation of handles from components, the AssetChanged query filter, the extension handler system for glTF, and the web asset support all reflect lessons learned from the broader game development ecosystem.

There is one more dimension worth noting: the philosophical alignment between Bevy's asset system and the rest of the engine. Bevy is built on the idea that everything is data, and that systems transform data. Assets fit this model perfectly. An asset is data. A handle is a lightweight reference to data. The Assets collection is a typed store of data. Systems read from Assets the same way they read from any other resource. Loading is an asynchronous process that produces data and stores it where systems can find it. There is no special "asset magic" — just data flowing through well-defined channels, driven by the same scheduler that drives everything else in the engine. This coherence is not merely aesthetic. It means that asset-related logic composes naturally with the rest of your game. A system that responds to asset changes can also query for entities, read from resources, send events, and do anything else a normal Bevy system can do. The asset system is not a walled garden or a separate subsystem with its own rules; it is a participant in the same data-driven architecture that governs the entire engine.

For the game developer, the practical upshot is this: you call load, you get a handle, you store that handle somewhere your systems can reach it, and you go about your business. The file will load when it loads. The data will appear when it appears. Your game keeps running the whole time. And when you no longer need the asset — when the level changes, when the entity is despawned, when the menu closes — you let the handle drop, and the memory is reclaimed. It is a system designed to stay out of your way while solving one of the hardest practical problems in game engine design: getting the world off the disk and into the frame, one asynchronous load at a time.
