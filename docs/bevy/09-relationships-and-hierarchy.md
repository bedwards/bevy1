# Relationships and Hierarchy: Connecting Entities to Each Other

In any sufficiently complex game, entities do not exist in isolation. A sword belongs to a knight. A wheel belongs to a car. A tooltip hovers beside a button, a particle trail streams behind a rocket, a camera follows a player character through a dense forest. These connections — ownership, attachment, following, containment — are as fundamental to the structure of a game world as the entities themselves. Without a way to express that one entity is related to another, you are left with a flat sea of disconnected things, and the burden of tracking which thing goes with which other thing falls entirely on your own bookkeeping code, scattered across systems that must redundantly maintain parallel data structures just to answer the simple question: "What belongs to what?"

For most of Bevy's life, the engine addressed this need through a hardcoded parent-child hierarchy. Two built-in components, `Parent` and `Children`, allowed you to arrange entities into trees. A `Parent` component on an entity stored a reference to its parent entity, and a `Children` component on the parent stored a list of its child entities. Bevy's transform propagation system used this hierarchy to ensure that when a parent moved, its children moved with it. Visibility cascaded downward as well: hiding a parent hid its children. This was useful and, for many games, sufficient. But it was also limiting in a way that became increasingly apparent as the ecosystem grew and developers tried to model richer, more varied connections between entities.

Consider a role-playing game where a character can equip weapons, wear armor, carry items in an inventory, and have a pet companion following them. All of these are relationships between the character entity and other entities, but they are not the same relationship. Equipping a weapon is not the same as parenting a child entity in a transform hierarchy — you probably want the weapon to inherit the character's position, yes, but you also want to query for "all weapons currently equipped by any character" or "the character who has this specific sword equipped," and you want the semantics of equipping to be distinct from the semantics of, say, an item sitting in an inventory slot. With only a single, hardcoded parent-child relationship available, developers were forced to overload that one relationship to mean many different things, or to build their own ad hoc relationship tracking using plain components that stored `Entity` values and required manual synchronization. Neither approach was satisfying. Overloading the parent-child hierarchy muddied its semantics and made it harder to reason about transform propagation and visibility, while hand-rolled relationships duplicated work that the engine could handle more efficiently and more correctly.

This tension motivated one of the most significant architectural changes in Bevy's recent history: the introduction of generalized relationships in version 0.16.

## The Relationship and RelationshipTarget Pattern

The core idea behind generalized relationships is deceptively simple. A relationship in Bevy is a component that stores a reference to another entity, and it is marked with the `#[relationship]` attribute so that Bevy knows to treat it specially. On the other side of that connection, the referenced entity automatically receives a corresponding component called a relationship target, which contains a collection of all the entities that point to it. The relationship component says "I am connected to that entity over there," and the relationship target component says "here are all the entities that claim to be connected to me." Bevy maintains both sides of this connection automatically, so you never have to worry about them falling out of sync.

In concrete Rust syntax, defining a custom relationship looks something like this:

```rust
#[derive(Component)]
#[relationship(relationship_target = EquippedBy)]
struct EquippedTo(Entity);

#[derive(Component)]
#[relationship_target(relationship = EquippedTo)]
struct EquippedBy(Vec<Entity>);
```

The `EquippedTo` component is the relationship: it lives on the weapon entity and stores a reference to the character entity that has it equipped. The `EquippedBy` component is the relationship target: it lives on the character entity and contains a list of all weapon entities that are equipped to that character. When you add an `EquippedTo(character)` component to a weapon entity, Bevy automatically updates the `EquippedBy` collection on the character entity to include that weapon. When you remove the `EquippedTo` component or despawn the weapon, Bevy automatically removes the weapon from the character's `EquippedBy` list. This bidirectional bookkeeping, which was previously something you had to implement yourself and get right in every edge case — what happens if the weapon is despawned? what happens if the character is despawned? what happens if the weapon is re-equipped to a different character? — is now handled by the engine, consistently and correctly.

The old `Parent` and `Children` components have been reimplemented using this exact pattern. `Parent` became `ChildOf`, a relationship component that stores a reference to the parent entity, and `Children` became the relationship target that contains a list of all entities whose `ChildOf` points to a given parent. This is not merely a renaming; it is a conceptual unification. The parent-child hierarchy is no longer a special, privileged feature of the engine — it is just one instance of the general relationship pattern, differing from your custom `EquippedTo` relationship only in that Bevy's built-in systems (transform propagation, visibility cascading) happen to use it. You can define as many different relationship types as your game needs, and each one gets the same automatic synchronization, the same bidirectional tracking, and the same integration with Bevy's entity lifecycle.

This generalization also means that you can query for relationships with the same expressiveness you use to query for any other component. Want to find all entities that are children of a specific parent? Query for `ChildOf` and filter by the parent's entity identifier. Want to find all weapons equipped by any character? Query for `EquippedTo`. Want to find all characters who have at least one weapon equipped? Query for `EquippedBy` and check that the collection is non-empty. The relationship system does not introduce a new query language or a separate application programming interface (API) — it builds on the component and query primitives that Bevy developers already know, which keeps the learning curve gentle and the mental model unified.

## Spawning Hierarchies and Managing Lifecycles

One of the immediate practical benefits of the new relationship system is how natural it makes spawning hierarchies of entities. Bevy 0.16 introduced the `children!` macro, which allows you to declare a hierarchy of entities in a single, readable expression rather than having to spawn each entity separately and then wire up the parent-child connections manually. The syntax mirrors the structure of the hierarchy itself:

```rust
commands.spawn((
    Player,
    Transform::default(),
    children![
        (Weapon, Transform::from_xyz(1.0, 0.0, 0.0)),
        (Shield, Transform::from_xyz(-1.0, 0.0, 0.0)),
    ],
));
```

This spawns a player entity with two children — a weapon and a shield — each with their own transform offset from the parent. The macro handles the creation of the child entities and the insertion of the appropriate `ChildOf` components, so the resulting hierarchy is fully connected and ready for transform propagation from the first frame. Before this macro existed, the equivalent code required multiple `spawn` calls, manual `add_child` invocations or builder chains, and considerably more visual noise that obscured the hierarchical structure you were trying to express. The `children!` macro is a quality-of-life improvement, but it is one of those improvements that, once you have used it, makes the old way feel almost unbearably verbose.

Beyond spawning, the relationship system also provides fine-grained control over what happens when entities are despawned. One of the most important attributes you can place on a relationship is `linked_spawn`, which tells Bevy that the related entity's lifecycle is tied to the entity it references. If you define a relationship with the `linked_spawn` attribute and the target entity is despawned, all entities that relate to it through that relationship are automatically despawned as well. This is the behavior you would expect for parent-child hierarchies — despawning a parent should despawn its children — and indeed, `ChildOf` uses `linked_spawn` by default. But the power of generalized relationships is that you can choose this behavior independently for each relationship type. Perhaps despawning a character should despawn their equipped weapons (because those weapons are meaningless without the character), but despawning a character should not despawn the items in their inventory (because those items might be lootable by other characters). With generalized relationships, you make that decision at the relationship type level, not at the call site, which means the lifecycle semantics are consistent everywhere a given relationship is used.

This brings us to another important change in Bevy 0.16: the `despawn()` method is now recursive by default. In earlier versions of Bevy, calling `despawn()` on an entity would remove only that entity, leaving its children orphaned in the world — still alive, still consuming memory and processing time, but no longer connected to anything meaningful. This was a common source of bugs, because developers would despawn a parent entity and then wonder why the game was gradually slowing down as thousands of orphaned child entities accumulated. The solution was to call `despawn_recursive()` instead, but remembering to do so every time was error-prone, and forgetting even once could introduce a slow leak that was difficult to diagnose. Bevy 0.16 made the pragmatic choice to make recursion the default behavior of `despawn()`. When you despawn an entity, all entities related to it through relationships marked with `linked_spawn` are despawned as well, cascading through the entire relationship graph. The old `despawn_recursive()` method is no longer needed and has been removed. If you genuinely want to despawn a single entity without touching its related entities, you can opt out explicitly, but the default now matches what developers almost always intend.

Entity cloning received a similar upgrade in 0.16 with the introduction of `clone_and_spawn()`. This method creates a deep clone of an entity — duplicating all of its components — and spawns the clone as a new entity in the world. What makes this particularly useful in the context of relationships is that it can be configured to clone entire hierarchies, recursively duplicating not just the entity but all of its children and their children, preserving the internal structure of the hierarchy while producing an entirely independent copy. This is invaluable for scenarios like spawning multiple instances of a complex prefab — a tree with branches and leaves, a vehicle with wheels and a turret, a user-interface panel with buttons and labels — where you want each instance to be a self-contained hierarchy that can be modified independently of the template it was cloned from.

## The Disabled Component and the Art of Hiding Without Destroying

Not every entity that should stop participating in the game world needs to be destroyed. Imagine a pool of bullet entities that you create once at the start of a level and reuse throughout, activating them when the player fires and deactivating them when they hit something or fly off screen. Creating and destroying entities is not free — it involves memory allocation, archetype updates, and bookkeeping overhead — so reusing entities from a pre-allocated pool can be a meaningful performance optimization, especially in scenarios with high entity throughput like particle systems or bullet-hell games. The problem, historically, was that Bevy had no built-in way to "deactivate" an entity. You could add a custom `Inactive` component and filter it out of every query with `Without<Inactive>`, but this was tedious, error-prone (miss one query and you have a bug), and it did not interact with Bevy's built-in systems, which would continue to process the entity as if it were fully active.

Bevy 0.16 introduced the `Disabled` component to address this gap. Adding `Disabled` to an entity causes that entity to be hidden from all queries by default, as if it temporarily does not exist. Systems that iterate over entities with `Query<&Position>` will not see disabled entities. Transform propagation will skip them. Rendering will ignore them. The entity is still in the world — its components are still stored, its relationships are still intact, its entity identifier is still valid — but it is invisible to the game's logic until you remove the `Disabled` component, at which point it springs back to life exactly as it was, with all of its data preserved.

If you have used Unity, this will feel familiar: it is conceptually similar to calling `GameObject.SetActive(false)`, which deactivates a game object and all of its children, removing them from rendering, physics, and scripting without destroying them. The parallel is not exact — Unity's activation system is deeply integrated into every subsystem of the engine in ways that Bevy's component-based approach does not replicate one-to-one — but the intent is the same. You want a way to say "this thing exists but is not currently participating," and you want that statement to be respected globally without having to teach every individual system about it.

There is also an illuminating analogy to "soft deletion" in database design. When a row in a database represents something valuable — a user account, a financial transaction, an audit record — you often do not want to delete it permanently when it becomes inactive, because you might need to restore it later, or you need it for historical reporting, or regulatory requirements demand that you retain it. Instead, you add a column like `is_deleted` or `deleted_at` and set it to true or to the current timestamp, and then you modify all of your queries to filter out soft-deleted rows. The `Disabled` component in Bevy serves a similar purpose: it is a soft-deletion flag for entities, allowing you to remove them from active participation without losing their data or their place in the world's hierarchy. The key advantage of Bevy's approach over rolling your own is that `Disabled` is understood by the engine itself, so you do not have to remember to add `Without<Disabled>` to every query you write — the filtering happens automatically, and if a system genuinely needs to see disabled entities (perhaps a debug inspector or a pool manager), it can explicitly opt in.

This feature is particularly powerful in combination with relationships and hierarchy. Disabling a parent entity can cascade to its children through the relationship graph, effectively deactivating an entire subtree of the hierarchy with a single component insertion. This makes object pooling and level streaming much more practical than they were before, because you can hide and reveal entire hierarchies of entities without paying the cost of despawning and respawning them.

## Component Propagation Through Hierarchies

The relationship system and the parent-child hierarchy set the stage for another feature that arrived in Bevy 0.17: component propagation. The idea is straightforward in concept, though its implications are far-reaching. Component propagation allows a component's value to cascade from a parent entity down through the hierarchy to all of its descendants, so that children inherit properties from their parents unless they explicitly override them. This is not a new idea — it is how Cascading Style Sheets (CSS) work in web development, how transform inheritance works in virtually every game engine, and how file permissions propagate through directory trees in many operating systems — but having it as a general-purpose mechanism in Bevy, available for any component type, is remarkably powerful.

Before component propagation was generalized, each property that needed to cascade through the hierarchy required its own bespoke system. Transform propagation was one hand-written system. Visibility propagation was another. If you wanted physics layers, rendering layers, audio attenuation zones, or any other property to flow from parent to child, you had to write your own propagation system that walked the hierarchy and copied or merged values at each level. This was not only tedious but also inconsistent — each propagation system might handle edge cases differently, might run at different points in the schedule, and might interact with the `Disabled` component (or not) in its own idiosyncratic way.

With generalized component propagation, you can declare that a component type participates in hierarchy propagation, and Bevy handles the rest. The propagation system walks the hierarchy from roots to leaves, and for each entity, it computes the effective value of the propagated component by combining the entity's own value (if it has one) with the inherited value from its parent (if one exists). The combination logic is customizable — for visibility, it might be a logical AND (a child is visible only if both it and its parent are visible), while for a tint color, it might be a multiplication (the child's color is modulated by the parent's color). The key insight is that the pattern of walking the hierarchy and combining values is always the same; only the combination function varies, and that function can be defined per component type.

This has profound implications for how you structure a game's entity hierarchy. Consider a tower defense game where each tower has multiple turrets, and each turret has a barrel, a muzzle flash effect, and a projectile spawn point. You might want the entire tower to share a "team color" that tints all of its visual components, a "damage multiplier" that is applied by a power-up and should affect all turrets on the tower, and a "visibility" flag that hides the tower and all its parts when it is being constructed. Without component propagation, you would need to write three separate systems to push these values down through the hierarchy. With component propagation, you declare that `TeamColor`, `DamageMultiplier`, and `Visibility` are propagated components, define how each one combines with its parent's value, and the engine handles the traversal. When you change the tower's team color, every descendant that does not have its own override automatically picks up the new value on the next propagation pass.

The interaction between component propagation and the `Disabled` component is worth noting explicitly. When an entity is disabled, propagation treats it as if it does not exist in the hierarchy, so its children will either inherit from the disabled entity's parent (if the propagation system is configured to skip disabled entities) or receive no inherited value at all. This ensures that disabling an entity has predictable, consistent effects throughout the hierarchy, rather than leaving orphaned inherited values on descendants that no longer have an active ancestor to inherit from.

Taken together, generalized relationships, the `Disabled` component, and component propagation represent a maturation of Bevy's approach to entity connectivity and hierarchy. Earlier versions of the engine gave you the bare minimum — a single parent-child hierarchy and a handful of built-in propagation systems — and left you to improvise everything else. The new systems give you a vocabulary for expressing arbitrary connections between entities, for controlling entity visibility and lifecycle at the relationship level, and for cascading properties through those connections automatically. They do this without abandoning Bevy's core principles: relationships are components, so they participate in the query system; propagation is driven by the hierarchy, so it builds on the existing Entity Component System (ECS) architecture rather than introducing a parallel data model; and everything is explicit, composable, and discoverable through Rust's type system.

The result is an engine that can model not just the simple hierarchies of early game prototypes — a player with a child sprite, an enemy with a child health bar — but the rich, tangled webs of connections that real games demand. A character equips a weapon, which is attached to a bone, which is part of a skeleton, which is a child of the character entity, which is a member of a faction, which is allied with certain other factions and hostile to the rest. Each of these connections is a relationship, each has its own lifecycle semantics, and each can propagate properties in its own way. The engine does not need to anticipate every kind of connection a game might need, because the relationship system is general enough to express connections the engine's authors never imagined. That generality, grounded in the familiar primitives of components and queries, is what makes the system both powerful and learnable — you do not need to master a new paradigm to use it, you just need to understand components a little more deeply than you did before.
