# Observers, Events, and Messages: Communication Between Systems

In any sufficiently complex program, different parts need to talk to each other. A collision detection system discovers that two entities have overlapped and needs to tell the damage system so that health can be subtracted. A player clicks a button in the user interface and that click needs to ripple outward into the game world, triggering an animation, playing a sound, perhaps deducting currency from an inventory. A network packet arrives from a remote server carrying information about another player's movement, and that information needs to find its way to the right system so that the remote player's position can be updated on screen. These are all instances of the same fundamental problem: how do you get information from the part of the program that produces it to the part of the program that needs it, without tangling those two parts together so tightly that changing one requires rewriting the other?

This problem is as old as software engineering itself, and the history of programming is littered with solutions to it. Callback functions, where one piece of code hands another piece of code a function to call when something interesting happens, are perhaps the oldest and most direct approach. The observer pattern, formalized by the Gang of Four in their influential 1994 book on design patterns, introduced the idea of subjects that maintain a list of observers and notify them when state changes occur. Publish-subscribe systems, often abbreviated as pub/sub, generalized this further by introducing a broker or message bus between publishers and subscribers so that neither side needs to know about the other. Event-driven architectures built entire applications around the idea that things happen, those happenings are captured as discrete event objects, and the flow of the program is determined by the sequence of events rather than by a predetermined call stack. Each of these approaches represents a different set of tradeoffs around immediacy, decoupling, ordering, and performance, and understanding those tradeoffs is essential to understanding why Bevy settled on the particular design it did.

For much of its early history, Bevy had a single mechanism for inter-system communication: the event system. You would define a Rust type, derive the appropriate trait on it, and then systems could write instances of that type into a shared queue using an `EventWriter` and read them back using an `EventReader`. This was a buffered communication channel — events written in one frame would be available for reading in the next frame, or more precisely, they would persist for two frames before being automatically discarded. The system worked well for many use cases, but it had a fundamental limitation: there was no way to respond to something the moment it happened. If a component was added to an entity, there was no mechanism to run a piece of logic immediately in response. You had to wait until the next time the relevant system was scheduled to run, which might be later in the current frame or in the next frame entirely, depending on system ordering. For entity lifecycle events — an entity being spawned, a component being inserted or removed — this delay was more than just an inconvenience. It meant that the world could be in an inconsistent state between the moment a change occurred and the moment the response to that change executed, and working around that inconsistency required careful system ordering and defensive checks that added complexity without adding clarity.

Bevy 0.14 introduced observers to address exactly this gap, and in doing so it split the communication model into two fundamentally different modes. Over subsequent releases the design was refined significantly, and as of Bevy 0.17 the distinction has been made explicit in the naming: what was once a single overloaded concept called "events" has been separated into two clearly differentiated mechanisms, each with its own name, its own semantics, and its own ideal use cases. Understanding the difference between these two mechanisms — and developing an intuition for when to reach for one rather than the other — is one of the most important skills you can build as a Bevy developer, because the choice affects not just the correctness of your code but its performance, its readability, and its resilience to future changes.

## Events and Observers: The Phone Call

The first mode of communication is what Bevy 0.17 calls Events, and the mechanism that responds to them is called Observers. If you were familiar with the pre-0.17 terminology, note that the old `Event` type and `EventWriter`/`EventReader` pair have been renamed to something else entirely (we will get to that shortly), and the word "Event" now refers specifically to the observable kind — the kind that triggers observers. The analogy that best captures the semantics is a phone call. When you make a phone call, the other person's phone rings immediately. They are interrupted from whatever they were doing. The conversation happens right now, in real time, and when the call is over, both parties return to what they were doing before. There is no queue, no delay, no batch processing. The communication is synchronous and immediate.

An Event in Bevy works the same way. When code fires an event, any observers that have registered interest in that event type are invoked immediately, right there in the call stack, before execution returns to whatever triggered the event in the first place. This is not a suggestion or a request that will be fulfilled later; it is a direct, immediate execution of the observer's logic. The observer runs, it does its work, and then control returns to the caller. This immediacy is what makes observers so powerful for situations where you need to guarantee that a response happens before anything else proceeds. If an entity is spawned and you need to ensure that certain derived components are present on it before any other system sees it, an observer on the spawn event gives you exactly that guarantee.

The system parameter that an observer receives was originally called `Trigger` but was renamed to `On<T>` in Bevy 0.17, where `T` is the event type. This renaming makes the code read more naturally — a function signature like `fn on_collision(event: On<Collision>)` communicates its purpose with admirable clarity. The `On<T>` parameter gives the observer access to the event data itself, along with contextual information like the entity the event is targeted at (if any) and the ability to propagate or stop propagation of the event through entity hierarchies.

Observers can be either global or targeted. A global observer fires whenever the event occurs anywhere, regardless of which entity (if any) is involved. A targeted observer is attached to a specific entity and only fires when the event is directed at that entity. This distinction is surprisingly powerful. Consider a health component: you might have a global observer that listens for all `Damage` events and updates a statistics tracker, while individual entities have their own targeted observers that play a hurt animation or trigger a death sequence specific to that entity's type. The global observer handles the cross-cutting concern, and the targeted observers handle entity-specific responses, and they coexist without interference.

The entity lifecycle hooks are perhaps the most natural application of the observer system, and they illustrate why immediate response matters so much. Bevy provides built-in event types for component changes: `Add` fires when a component type is added to an entity for the first time, `Insert` fires whenever a component is inserted (whether or not it was already present), and `Remove` fires when a component is removed from an entity. These are not events you fire yourself — Bevy fires them automatically as part of the entity manipulation operations. When you call `commands.entity(id).insert(Health(100.0))`, Bevy will fire the appropriate `Add` or `Insert` event, and any observers listening for that event will execute immediately within the command processing. This means you can write initialization logic, validation logic, or bookkeeping logic as observers and trust that it will run at exactly the right moment, without worrying about system ordering or frame timing.

Bevy 0.17 also introduced `SpawnDetails` and the `Spawned` filter, which together provide a way to detect and respond to recently spawned entities with more granularity than was previously possible. The `Spawned` filter can be used in queries to match entities that were created during the current frame, which is useful for initialization systems that need to set up derived state for new entities. Meanwhile, observers on spawn events provide the immediate-response path for cases where you cannot afford to wait even until the next system runs within the same frame.

## Messages: The Letter

The second mode of communication is what Bevy 0.17 calls Messages, and this is what the old `EventWriter` and `EventReader` have been renamed to: `MessageWriter` and `MessageReader`. If events and observers are a phone call, messages are a letter dropped in a mailbox. You write the letter, you deposit it, and you walk away. The recipient will find it in their mailbox the next time they check, which might be later today or might be tomorrow. There is no interruption, no immediate response, no guarantee about exactly when the message will be processed. What there is, instead, is decoupling. The sender does not need to know who will read the message, or when, or how many readers there are. The sender does not even need to know if anyone is listening at all. It simply writes the message and moves on.

In Bevy's implementation, messages are accumulated in a double-buffered queue. Systems that write messages using `MessageWriter<T>` add items to the current frame's write buffer. Systems that read messages using `MessageReader<T>` consume items from the previous frame's read buffer. This double-buffering means that messages written in one frame become available for reading in the next scheduling opportunity, and they persist for two update cycles before being automatically cleaned up. The implication is that message-based communication has an inherent latency of at least one system execution boundary — the reader will not see a message until it runs after the writer, and if they happen to be scheduled in an order where the reader runs first, the message will not be seen until the next frame.

This latency is not a bug; it is a feature. By decoupling the timing of writing from the timing of reading, messages allow systems to communicate without creating ordering dependencies between them. The score system can write a `ScoreChanged` message whenever the score updates, and the user-interface system can read those messages and update the score display whenever it happens to run. Neither system needs to know about the other's schedule. Neither system needs to be ordered relative to the other in the system graph. This loose coupling makes messages ideal for communication where the exact timing of the response does not matter — where a one-frame delay is imperceptible to the player and irrelevant to the correctness of the simulation.

Messages are also the right choice when multiple producers and multiple consumers need to participate in the same communication channel. Because the message queue is a shared resource, any number of systems can write to it and any number of systems can read from it. Each reader maintains its own cursor into the queue, so readers are independent of one another — one reader consuming a message does not prevent another reader from seeing it. This makes messages a natural fit for fan-out communication patterns, where a single event (in the informal sense of the word) needs to be noticed by several different systems. A `PlayerDied` message might be read by the respawn system, the statistics system, the achievement system, and the audio system, each processing it according to their own logic and their own schedule.

## Choosing Between the Two

The distinction between events-with-observers and messages maps onto a broader tension in software architecture between synchronous and asynchronous communication, between immediacy and decoupling, between tight coordination and loose coordination. Neither mode is universally superior. They serve fundamentally different needs, and a well-architected Bevy application will use both.

Reach for observers when the response must happen immediately, when you need to guarantee that some logic runs before any other system sees the entity or the world in its post-change state. Entity lifecycle management is the canonical example: when a component is added, removed, or when an entity is spawned, you often need to perform bookkeeping — updating indices, initializing derived state, cleaning up references — that must happen atomically with the change itself. Observers give you that atomicity. Reach for observers also when the communication is inherently targeted at a specific entity. A collision event between entity A and entity B is meaningful in the context of those specific entities, and an observer targeted at entity A can respond to collisions involving A without needing to filter through a global stream of all collisions in the world.

Reach for messages when the communication is a broadcast that does not need an immediate response. Score changes, level completions, settings modifications, analytics tracking, UI notifications — these are all situations where a one-frame delay is perfectly acceptable and the decoupling benefits of the message queue outweigh the immediacy benefits of an observer. Reach for messages also when you specifically want to avoid the performance implications of synchronous dispatch. Because observers run immediately, they execute within the call stack of whatever triggered the event, which means they can extend the duration of the triggering operation in ways that might be difficult to predict or profile. Messages, by contrast, are processed during the normal system scheduling pass, which means their cost is visible in the same profiling tools and scheduling graphs that you use for every other system.

There is also a subtler consideration around control flow and reentrancy. Because observers fire immediately, an observer can itself trigger events that fire other observers, which can create chains of immediate execution that are powerful but potentially confusing to debug. If observer A fires event X, and observer B for event X fires event Y, and observer C for event Y modifies a component that fires observer D, you have a cascade of logic executing synchronously within a single system call. This is occasionally exactly what you want — it lets you model chain reactions and complex state transitions compactly — but it demands discipline and careful documentation, because the execution path through a cascade of observers is not visible from any single point in the code. Messages, by their nature, cannot cascade in this way. A message written by one system will not be read until the next scheduling pass, which provides a natural firebreak that limits the complexity of any single execution path.

It is instructive to compare Bevy's approach with the communication mechanisms offered by other engines and frameworks. Unity provides several overlapping systems: C-sharp events and delegates for synchronous callback-style communication, UnityEvent for serializable event connections configured in the editor's inspector, and a message-passing system built around `SendMessage` that uses string-based method names and runtime reflection. The C-sharp event system is powerful but exists entirely outside Unity's entity-component model — it is a pure language-level feature with no engine integration, which means Unity cannot schedule or optimize around it. UnityEvent provides editor integration but carries performance overhead from serialization and reflection, and it binds events to specific GameObjects in ways that can make refactoring difficult. `SendMessage` is widely considered an anti-pattern due to its reliance on string matching and its complete lack of type safety. Bevy's observer system, by contrast, is deeply integrated with the Entity Component System. The engine knows about observers, schedules them, and can reason about their data access patterns. Event types are ordinary Rust types, checked at compile time, with no strings or reflection involved.

Traditional object-oriented implementations of the observer pattern typically involve a subject object that maintains a list of observer interfaces, and concrete observer classes that implement an update method. This approach couples the observer to a specific interface, requires explicit registration and deregistration (with the attendant risk of memory leaks if an observer forgets to deregister), and provides no mechanism for the framework to reason about what data the observer will access. Pub/sub systems like those found in Redis, RabbitMQ, or cloud service buses solve the decoupling problem admirably but introduce infrastructure complexity — a separate broker process, network serialization, delivery guarantees, dead letter queues — that is wildly excessive for communication between systems within a single process running on a single machine. Bevy's message system captures the essential benefit of pub/sub — the sender and receiver do not know about each other — without any of the infrastructure overhead, because the message queue is simply a data structure in the same process, accessed through the same ECS resource system that governs everything else.

The evolution from Bevy's original single event system to the current dual system of events-with-observers and messages reflects a broader maturation in the engine's understanding of what game developers actually need. The original event system tried to be one thing that served two very different purposes, and the result was a mechanism that was adequate for neither. It was too slow for immediate response, because events were buffered, and it was too tightly scheduled for truly decoupled communication, because readers needed to be ordered after writers within the same frame to avoid missing events. By splitting the concept in two and giving each half a distinct name, a distinct implementation, and a distinct set of semantics, Bevy 0.17 made it possible to reach for exactly the right tool without compromise.

There is something deeply satisfying about a design that takes a muddled concept and cleaves it into two sharp, well-defined pieces. The phone call and the letter. The immediate and the deferred. The targeted and the broadcast. These are not just implementation details; they are conceptual tools that shape how you think about the flow of information through your game. When you sit down to design a new feature and you ask yourself "how should this system learn about that change?", having two clearly distinct answers — "it should observe it, right now, as it happens" versus "it should receive a message about it, in due course, when it next runs" — is immensely clarifying. It forces you to think about whether timing matters, whether ordering matters, whether the response is entity-specific or global, and whether you want tight coupling or loose coupling. These are the right questions to be asking, and Bevy's communication model ensures that you ask them every time, because the language of the application programming interface itself makes the distinction impossible to ignore.

In practice, many features will use both mechanisms in concert. A collision detection system might use observers to immediately apply damage and update physics state — things that must happen before the next simulation step — while also writing a `CollisionOccurred` message that the audio system, the particle effects system, and the analytics system can each pick up in their own time. The observer handles the urgent, entity-specific response. The message handles the leisurely, cross-cutting notification. Together they cover the full spectrum of communication needs, and neither is asked to do something it was not designed for. This is the hallmark of good architecture: not a single mechanism stretched to cover every case, but a small number of well-chosen mechanisms, each excellent at its particular job, composing gracefully to cover the full range of requirements. Bevy's observer and message systems, born from the hard-won experience of earlier versions that tried and failed to unify these concerns, represent exactly that kind of thoughtful, principled design.
