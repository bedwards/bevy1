# The Plugin Architecture and the Bevy Ecosystem

Every sufficiently complex piece of software eventually faces the same organizational question: how do you add new capabilities without making the existing system harder to understand, harder to maintain, and harder to change? This question has been answered differently by different traditions of software design. Libraries offer functionality you call into. Frameworks offer a skeleton you fill in. Middleware offers interception points in a pipeline. And plugins offer self-contained modules that register themselves with a host application, extending its behavior without modifying its core. Bevy chose the plugin model, and this choice has shaped everything about how Bevy applications are structured, how the ecosystem has grown around the engine, and how individual developers organize their own game code.

A Bevy plugin, at its most fundamental, is a Rust struct that implements the Plugin trait, which requires a single method called `build` that receives a mutable reference to the App. Inside that method, the plugin registers whatever it needs: systems, resources, events, states, assets, and other plugins. That is the entire contract. There is no configuration file, no registration ceremony, no dependency injection framework. You call `app.add_plugins(MyPlugin)` and the plugin does its work during application startup. This simplicity is deliberate and powerful. A plugin that adds a physics simulation registers its components, its systems, and its resources. A plugin that adds networking registers its connection management systems and its message types. A plugin that adds a debug overlay registers its UI systems and its diagnostic resources. Each plugin is self-contained in the sense that it declares everything it needs, and composable in the sense that multiple plugins can coexist without knowing about each other, as long as they do not conflict on resource types or system ordering.

The default Bevy application begins with `DefaultPlugins`, which is itself a plugin group — a collection of plugins bundled together for convenience. DefaultPlugins includes the window manager, the renderer, the asset server, the input system, the audio system, the UI system, and dozens of other foundational capabilities. You can customize individual plugins within the group using the `.set()` method, which replaces a specific plugin's configuration without affecting the others. You can also build your own plugin groups, which is useful for organizing a large game into feature modules: a `GameplayPlugins` group might include your player controller, your enemy AI, your inventory system, and your quest tracker, each as a separate plugin that can be developed and tested independently.

The composability of plugins has enabled a thriving ecosystem of community crates that extend Bevy's capabilities in directions the core team has not yet reached or has deliberately left to the community. Understanding the major ecosystem plugins is part of understanding Bevy itself, because in practice most Bevy games use several community crates alongside the core engine. The ecosystem divides roughly into categories: physics, UI extensions, asset management, networking, audio, and development tools, though the boundaries are not always clean and many crates span multiple categories.

Physics is perhaps the most critical ecosystem category, because nearly every game that involves movement, collision, or physical interaction needs a physics engine, and Bevy does not include one in its core. The two major options as of early 2026 are Avian (formerly known as bevy_xpbd, which stood for Extended Position Based Dynamics) and bevy_rapier, which wraps the Rapier physics library. The difference between them is architectural and significant. Avian is Entity Component System-native, meaning it represents physics bodies, colliders, joints, and forces as Bevy components and runs its simulation as Bevy systems. You add a `RigidBody` component and a `Collider` component to an entity, and the physics simulation operates on them directly through the same query system that everything else in Bevy uses. Rapier, by contrast, maintains its own internal physics world — a separate data structure that mirrors the Bevy world — and synchronizes between them. This means Rapier has more features and a longer track record (it exists as a standalone physics library outside Bevy), but the synchronization layer adds complexity and overhead. As of version 0.4, Avian has achieved roughly three times the performance of bevy_rapier for common scenarios and has become the community's preferred choice for new projects, though Rapier remains a reasonable choice for games that need its specific advanced features or that are porting existing Rapier-based code.

For user interface work beyond what Bevy's built-in system provides, bevy_egui integrates the egui library (which stands for "easy GUI," an immediate-mode graphical user interface library) into Bevy. Immediate-mode GUI is a different paradigm from Bevy's native retained-mode UI: instead of spawning entities that persist, you call functions every frame that describe what the UI should look like, and the library handles rendering and input. This approach is particularly well-suited to developer tools, debug panels, and editor interfaces where the UI changes frequently and does not need complex animation or styling. Many Bevy projects use bevy_egui for their development-time UI (level editors, diagnostics panels, entity inspectors) and Bevy's native UI system for their player-facing interface (health bars, menus, dialogue boxes), combining the strengths of both paradigms without committing entirely to either one.

The asset management category includes crates like bevy_asset_loader, which provides structured, declarative asset loading with progress tracking — you define a struct listing all the assets a particular game state needs, and the loader handles fetching them, tracking progress, and transitioning to the next state when everything is ready. This solves a common pain point where games need to show a loading screen while dozens or hundreds of assets are fetched from disk, and where forgetting to load a single asset causes a runtime panic when you try to use it. The declarative approach makes missing assets a compile-time error rather than a runtime surprise, which aligns with the broader Rust philosophy of catching errors as early as possible.

Networking in Bevy is still an area of active development without a single dominant solution, which reflects the genuine difficulty of the problem. Game networking involves state synchronization, latency compensation, prediction, and rollback — techniques that are deeply intertwined with game logic and do not lend themselves to a one-size-fits-all library. Several community crates exist in this space, each with different trade-offs, and the choice between them depends on the type of game you are building. A turn-based strategy game has very different networking requirements from a fast-paced first-person shooter, and a library optimized for one will be poorly suited to the other.

The development tools category includes Bevy's own built-in diagnostics (frame time, entity count, system timing), the bevy-inspector-egui crate that provides a runtime entity inspector (similar to Unity's Inspector panel), and various profiling integrations. The Tracy profiler, a real-time frame profiler popular in the game development community, has Bevy integration through the tracy feature flag, allowing you to see exactly how long each system takes, where frame time is being spent, and where parallelism is or is not being achieved. These tools are not luxuries — they are necessities for any game that aims to run at a consistent frame rate, because performance problems in games are often non-obvious and non-local, caused by interactions between systems that are individually fast but collectively create bottlenecks.

The plugin architecture also shapes how you organize your own game code, even if you never publish a crate. A well-structured Bevy game treats each major feature as a plugin: the player controller is a plugin, the camera system is a plugin, the enemy spawning system is a plugin, the score tracking is a plugin. This organization has several benefits beyond mere tidiness. Each plugin can be developed and tested in isolation — you can write tests for your player controller plugin without needing the enemy system, the UI, or the renderer. Plugins can be enabled or disabled for debugging — if something is wrong with enemy behavior, you can comment out the enemy plugin and verify that the rest of the game works correctly. And plugins provide natural boundaries for code review and team collaboration — one developer can work on the inventory plugin while another works on the combat plugin, with minimal merge conflicts because each plugin lives in its own module with its own systems and components.

The rate at which the Bevy ecosystem evolves is both a strength and a challenge. With major releases roughly every three months, community crates must update frequently to remain compatible, and there is often a lag between a new Bevy release and the availability of updated ecosystem crates. This is a real cost of using an engine that has not yet reached version 1.0 — your dependencies may not all be available for the latest Bevy version at the same time, and you may need to make trade-offs between staying current with Bevy and staying current with your ecosystem dependencies. The community has developed coping mechanisms for this: most major ecosystem crates maintain branches for the latest Bevy release, many publish beta versions aligned with Bevy's release candidates, and the Bevy community Discord is an active venue for tracking compatibility status across the ecosystem.

What makes Bevy's plugin model particularly interesting, compared to plugin systems in other engines, is that it does not create a distinction between "engine code" and "game code." In Unity, there is a clear line between what the engine provides and what the game developer builds on top of it, and crossing that line (modifying engine behavior) requires source access and deep internal knowledge. In Bevy, the engine itself is composed of plugins, and your game's plugins are structurally identical to the engine's plugins. You can replace a built-in plugin with your own version, extend an existing plugin's behavior by adding systems that operate on its components, or compose engine and game plugins into new combinations that the original authors never envisioned. This uniformity is a direct consequence of the Entity Component System architecture: because everything is expressed in terms of entities, components, systems, and resources, there is no special "engine layer" that operates by different rules than the "game layer." The whole thing is one flat, composable system, and plugins are how you carve it into manageable pieces.

This architectural choice has implications that extend beyond code organization into the philosophy of what a game engine should be. A traditional engine is a product: a finished tool that you use as-is, with an upgrade path controlled by the engine vendor. Bevy is more like a material: a set of composable building blocks that you assemble into the engine your game actually needs. This means more work up front — you are assembling the engine as well as building the game — but it also means that you never hit a wall where the engine cannot do what you need, because the engine is not a fixed thing but a composition you control. For some developers and some projects, the overhead of this assembly is not worth it, and they would be better served by a more opinionated, more complete engine like Unity or Godot. But for developers who value control, transparency, and the ability to understand every layer of their stack, Bevy's plugin architecture is not just a feature of the engine — it is the reason the engine exists.
