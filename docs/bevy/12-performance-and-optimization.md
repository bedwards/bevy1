# Performance and Optimization: Making Games Run Fast

Performance in games is not like performance in most other software. When a web server takes fifty milliseconds to respond instead of ten, the user probably does not notice. When a database query takes two seconds instead of one, the user sighs and waits. But when a game takes twenty milliseconds to render a frame instead of sixteen, the player feels it — not as a number, not as a benchmark, but as a visceral sense that something is wrong, that the world they are interacting with is sluggish and unresponsive. This is because games operate under a contract with human perception that no other category of software shares: they must update the world, process input, and render a complete image fast enough that the result appears continuous to the human eye and responsive to the human hand. Sixty frames per second means sixteen point six milliseconds per frame, total, for everything. At one hundred and twenty frames per second, that budget is eight point three milliseconds. There is no "try again later" in real-time rendering. Every frame that misses its deadline is a frame the player experiences as stutter.

This constraint shapes everything about how game engines are designed, and understanding Bevy's performance characteristics requires understanding the multiple levels at which optimization happens: the language level, the architecture level, the algorithm level, and the data level. These levels interact in complex ways, and a game that is optimal at one level but pessimal at another will still run poorly. True performance comes from getting all the levels right simultaneously, or at least getting none of them catastrophically wrong.

At the language level, Bevy benefits enormously from being written in Rust. Rust compiles to native machine code with no garbage collector, no runtime interpreter, and no virtual machine overhead. This might sound like a minor technical detail, but for games it is foundational. Languages with garbage collectors — Java, C#, Go, JavaScript — periodically pause execution to reclaim unused memory, and these pauses are unpredictable in both timing and duration. A garbage collection pause that happens during a frame render creates a stutter that the player sees and feels. Rust eliminates this category of problem entirely through its ownership system, which determines at compile time when each piece of memory should be freed, resulting in deterministic memory management with zero runtime cost. This does not mean Rust code is automatically fast — you can write slow Rust code just as easily as slow C code — but it means the language will never sabotage your performance with hidden runtime costs.

At the architecture level, Bevy's Entity Component System is designed for cache-friendly data access, which is one of the most important and least intuitive aspects of modern performance optimization. Modern CPUs are extraordinarily fast at computation but relatively slow at fetching data from main memory — a cache miss (where the CPU needs data that is not in its fast local cache) can cost hundreds of clock cycles, during which the CPU sits idle waiting. Traditional object-oriented game architectures, where each game object is a separately allocated object containing all its data, scatter related data across memory in a pattern that generates frequent cache misses. When you iterate over a thousand game objects to update their positions, each object's position might be in a completely different region of memory, and the CPU spends most of its time waiting for data rather than processing it.

Bevy's archetype-based ECS storage solves this problem by storing components of the same type contiguously in memory. All the Transform components for entities that share the same set of components (the same "archetype") are packed together in a single array, and all the Velocity components for those entities are packed in another array right alongside. When a system iterates over all entities with Transform and Velocity to perform movement, the CPU fetches data in sequential order from tightly packed arrays, maximizing cache utilization and minimizing idle time. This is not a small difference — for systems that iterate over thousands of entities, the cache-friendly layout can provide an order-of-magnitude speedup compared to a scattered object-oriented layout.

The scheduler adds another layer of architectural optimization through automatic parallelism. Bevy analyzes the data access patterns of all registered systems and determines which systems can safely run concurrently. Two systems that only read the same component type can run in parallel on different CPU cores. Two systems that access completely different component types can run in parallel. Only systems that conflict — one writing a component type that another reads or writes — need to be serialized. This analysis happens at startup and imposes no per-frame overhead, and it means that well-structured Bevy games naturally take advantage of multi-core CPUs without the developer manually managing threads, locks, or synchronization primitives. The key to maximizing this parallelism is to keep systems small and focused: a system that reads every component type in the game cannot run in parallel with anything, while a system that only touches Transform and Velocity leaves room for many other systems to run alongside it.

At the rendering level, Bevy 0.16 introduced GPU-driven rendering, which represents a fundamental shift in how draw calls are managed. In traditional rendering, the CPU decides what to draw and issues individual draw calls to the GPU for each visible object — or at least for each batch of similar objects. The overhead of these draw calls limits how many objects can be drawn per frame, and complex scenes with many small objects can become CPU-bound even when the GPU has ample capacity. GPU-driven rendering moves the decision of what to draw onto the GPU itself, using compute shaders to cull invisible objects, sort draw calls, and batch similar materials — all at GPU speed rather than CPU speed. The result is approximately a three-times improvement in rendering throughput for complex scenes, with the greatest benefits for scenes containing many individual objects, which is exactly the scenario where traditional rendering struggles most.

The experimental Solari raytraced renderer, introduced in Bevy 0.17, takes a completely different approach to the rendering problem. Traditional rasterization (the technique used by GPU-driven rendering and essentially all real-time rendering since the 1990s) determines the color of each pixel by projecting triangles onto the screen and filling them in. It is fast but requires various hacks and approximations for effects that depend on global illumination — shadows, reflections, ambient occlusion, color bleeding between surfaces. Ray tracing works in the opposite direction: for each pixel, it shoots a ray from the camera into the scene and follows the ray as it bounces between surfaces, accumulating light information along the way. This is physically more accurate and produces stunningly realistic images, but it is computationally far more expensive. Solari requires dedicated ray tracing hardware (like Nvidia's RTX series) and is not suitable for all games or all target hardware, but for projects that can afford it, the visual quality improvement is dramatic. Understanding the tradeoff between rasterization and ray tracing — between speed and accuracy, between broad hardware support and visual fidelity — is important for making informed technical decisions about your game's rendering strategy.

Beyond these engine-level optimizations, there are patterns that individual developers should know for writing performant Bevy code. Query access patterns matter: iterating over a query with `for (transform, velocity) in &query` is faster than calling `query.get(entity)` for individual entities, because the iteration path uses the archetype storage directly while individual lookups require an indirection through the entity-to-archetype mapping. Marker components — components with no data, used purely for filtering — are free from a performance perspective and should be used liberally to narrow queries. A system that queries `Query<&Transform, With<Enemy>>` will iterate only over enemies, not over every entity with a Transform, and this filtering happens at the archetype level with zero per-entity cost.

Resource access patterns also matter, though in subtler ways. A system that takes `ResMut<T>` (mutable resource access) conflicts with any other system that accesses the same resource, preventing parallelism. If a resource is read frequently and written rarely, consider whether the writing can be batched into a single system that runs at a specific point in the schedule, leaving the rest of the frame free for parallel readers. Commands — the deferred mutation system that lets you spawn and despawn entities, add and remove components — are another area where understanding the execution model matters for performance. Commands are buffered and applied at specific synchronization points, not immediately when they are issued. This means that spawning an entity inside a system does not make it visible to other systems until the next synchronization point, which can be surprising but is necessary for the scheduler's parallelism guarantees.

The Fixed Timestep schedule, called FixedUpdate in Bevy, deserves specific mention in a performance context. Physics simulations and other deterministic systems need to run at a constant rate regardless of frame rate — if the frame rate drops, the physics should run more steps per frame to keep up, and if the frame rate is high, the physics should run fewer steps. FixedUpdate handles this automatically, running at a configurable rate (typically sixty times per second) with interpolation to smooth the result between frames. Getting FixedUpdate right is crucial for both correctness and performance: running the physics too frequently wastes CPU time, while running it too infrequently causes objects to tunnel through walls and collisions to be missed. The default rate is a good starting point, but games with fast-moving objects or precise collision requirements may need to adjust it.

Profiling is the compass that guides all optimization work, and it deserves to be treated with the same seriousness as testing. Bevy integrates with the Tracy profiler through a feature flag, providing detailed per-system timing, render pipeline timing, and memory allocation tracking. The Chrome tracing format is also supported through Bevy's built-in tracing integration, allowing you to visualize system execution on a timeline and identify gaps, stalls, and bottlenecks. The most important rule of performance optimization is also the most frequently violated: never optimize without profiling first. Human intuition about where time is being spent is wrong more often than it is right, and an optimization applied to code that is not actually a bottleneck wastes developer time without improving player experience. Profile, identify the bottleneck, fix it, measure the improvement, and then profile again to find the next bottleneck. This cycle, tedious as it sounds, is the only reliable path to fast software, and any shortcut that skips the measurement step is not a shortcut at all but a detour into wasted effort.

The relationship between performance and architecture cuts both ways, and this is perhaps the most important insight for developers starting a new project. Good architecture — small systems, focused components, clear data flow — tends to produce good performance almost accidentally, because it aligns with the assumptions that Bevy's scheduler and storage system are optimized for. Bad architecture — monolithic systems, kitchen-sink components, tangled data dependencies — tends to produce bad performance even when each individual operation is fast, because it prevents parallelism, generates cache misses, and makes profiling difficult. The time you invest in clean architecture at the beginning of a project pays performance dividends throughout its lifetime, and the performance problems you encounter later are often symptoms of architectural problems that should be fixed at the design level rather than patched with micro-optimizations. In this sense, performance optimization in Bevy is not a separate activity from good software design — it is a natural consequence of it.
