# The Game Loop: The Heartbeat of Interactive Software

Every interactive program that has ever run on a computer shares a common structural pattern, a pattern so fundamental that it often goes unnamed and unremarked upon, like the beating of a heart that you notice only when it skips. This pattern is the loop — a continuous cycle of gathering input, updating state, and producing output — and in game development it is called the game loop. Understanding the game loop is understanding the most basic unit of how games work, and from this single concept flow nearly all of the architectural decisions that make game engines what they are.

The simplest possible game loop looks something like this, expressed in pseudocode: forever, read what the player is doing, update the game world based on that input and any rules the game enforces, draw the result to the screen, and repeat. In the earliest video games, running on hardware with a single processor and a fixed refresh rate, this loop was literal: the game processed one frame of logic and one frame of rendering in lockstep with the display's vertical blank interval, and the speed of the game was tied to the speed of the hardware. If you played a game designed for a sixty-hertz display on a one-hundred-twenty-hertz display, it would run at double speed, because each tick of the loop would happen twice as fast. This coupling between game speed and display speed was not a design choice but a consequence of the hardware constraints of the era, and one of the first problems game developers learned to solve was decoupling game logic from display refresh rate.

The solution is the concept of delta time — the amount of real time that has elapsed since the last iteration of the loop. Instead of moving a character by a fixed distance per frame (which makes the movement speed dependent on the frame rate), you move it by a distance proportional to delta time (which makes the movement speed constant regardless of how fast the loop runs). If sixty frames elapse in one second, each frame moves the character by one-sixtieth of its per-second speed. If thirty frames elapse, each frame moves it by one-thirtieth. The character covers the same total distance per second in both cases, even though the loop runs at different rates. This simple idea — multiply per-frame values by the time since the last frame — is the foundation of frame-rate-independent game logic, and it appears in virtually every modern game in some form.

But delta-time integration, as it is called, introduces a subtle problem that becomes significant for physics simulations. When you multiply by delta time, you are making a linear approximation of what is really a continuous process: an object moving under gravity does not actually teleport from one position to the next each frame; it follows a continuous trajectory that the game approximates with discrete steps. The accuracy of this approximation depends on how small the time steps are, and when the frame rate drops (because the CPU is overloaded, because the player opened another application, because a background process spiked), the time steps get larger and the approximation gets worse. An object that should have collided with a wall might tunnel through it because a single large time step moved it past the wall entirely. Two objects in the same simulation might behave differently if one is running at sixty frames per second and the other at thirty, because the numerical integration accumulates different errors at different step sizes.

The solution to this problem is the fixed timestep, and understanding it is essential for anyone building a game with physics or any other system that requires deterministic behavior. A fixed timestep loop runs the simulation at a constant rate — say, sixty times per second — regardless of the actual frame rate. If the rendering runs faster than the simulation, some rendered frames interpolate between simulation states without advancing the simulation. If the rendering runs slower, the simulation runs multiple steps per rendered frame to catch up. The result is that the simulation always sees the same time step, always produces the same results from the same inputs, and is completely decoupled from the vagaries of rendering performance. In Bevy, this is the FixedUpdate schedule, and it is the correct place for physics, collision detection, gameplay timing, and anything else where reproducibility matters.

The relationship between the fixed timestep and the variable rendering loop creates a subtle but important architectural boundary in a game. On one side of that boundary, everything is deterministic: given the same inputs in the same order, the same outputs are produced every time, on every machine, at every frame rate. On the other side, everything is responsive: the rendering adapts to whatever frame rate the hardware can sustain, input is sampled as frequently as possible, and visual effects that do not affect gameplay (particle systems, camera smoothing, animation blending) update every frame for maximum fluidity. This boundary is not just a technical convenience — it is a conceptual division that helps you think clearly about which parts of your game need to be reproducible (game logic, networking, replays) and which need to be responsive (visuals, audio, input feedback).

The game loop also implies something about the nature of game programming that distinguishes it from most other forms of software development. In a web application, code runs in response to requests: a user submits a form, the server processes it, and the server waits for the next request. Between requests, the server is idle. In a game, the loop runs continuously, processing every entity in the world sixty or more times per second, whether or not the player is doing anything. This means that the performance of every system matters every frame, not just during peak load but at all times. A web server that occasionally takes a hundred milliseconds to respond is fine. A game that occasionally takes a hundred milliseconds to render a frame has a visible stutter that players immediately notice and find unpleasant. This constant performance pressure shapes the architecture of game engines at every level, from the memory layout of entities (cache-friendly for fast iteration) to the scheduling of systems (parallel where possible, sequential only where necessary) to the asset loading strategy (asynchronous, to avoid blocking the loop).

Modern game engines abstract the game loop so that developers rarely write it directly. In Bevy, the loop is managed by the App and its runner, and developers interact with it by registering systems in schedules rather than by writing an explicit loop. The Startup schedule runs once at launch. The Update schedule runs every frame. The FixedUpdate schedule runs at the fixed timestep. PreUpdate and PostUpdate handle engine-level bookkeeping like input processing and transform propagation. This abstraction is valuable because it separates the concern of what happens each frame (the systems you write) from the concern of how frames are scheduled (the engine's loop implementation), and it enables the automatic parallelism that is one of Bevy's key features: because the engine controls when and how systems run, it can schedule them across multiple CPU cores without the developer managing threads explicitly.

Yet even in a modern engine, understanding the underlying loop illuminates why the engine's abstractions work the way they do. Why does Bevy have separate schedules for PreUpdate, Update, and PostUpdate? Because the loop has a natural order — input must be processed before game logic reads it, and game logic must run before rendering draws its results. Why does FixedUpdate sometimes run zero, one, or multiple times per frame? Because the fixed timestep and the variable rendering timestep are decoupled, and the fixed step catches up or waits as needed. Why can't you spawn an entity and query it in the same system? Because commands are deferred to maintain the integrity of the loop's data access patterns, and entities spawned during a system's execution become visible only after the system completes. Each of these design decisions traces back to the requirements of the game loop, and understanding the loop makes these decisions feel not arbitrary but inevitable.

The game loop is also the lens through which to understand why game development feels different from other programming. In most software, you write code that runs in response to some trigger, does its work, and finishes. In a game, you write code that runs perpetually, thousands of times per second, maintaining a living world that must always be consistent, always be responsive, and always be ready for the next frame. This is closer to the engineering of a physical machine than to the engineering of a typical software system — the game loop is the engine's crankshaft, turning continuously, and every system you attach to it must pull its weight without slowing the whole mechanism down. The beauty of a well-built game is that this continuous operation is invisible to the player, who sees only a world that responds to their actions as naturally as the physical world responds to their body. The game loop is what makes that illusion possible, and understanding it is the first step toward building it well.
