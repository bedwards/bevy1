# State Machines in Games: Organizing Behavior That Changes Over Time

Almost every interesting behavior in a game involves something changing from one mode to another. A character is idle, then walking, then jumping, then falling, then landing. An enemy is patrolling, then alerted, then chasing, then attacking, then retreating. A door is locked, then unlocked, then opening, then open, then closing. A menu screen transitions to gameplay, which transitions to a pause screen, which returns to gameplay. Each of these scenarios involves an entity or system that behaves differently depending on what state it is in, that transitions between states in response to specific conditions, and that may perform specific actions when entering or leaving a state. The finite state machine is the formal concept that captures this pattern, and it is one of the most widely used design tools in game development, appearing in character controllers, artificial intelligence (AI), user interface navigation, game flow management, and animation systems.

A finite state machine, in its simplest form, consists of three things: a set of states, a set of transitions between states, and a specification of which state the machine starts in. At any given moment, the machine is in exactly one state. When certain conditions are met (a key is pressed, a timer expires, a collision occurs), the machine transitions to a different state. The word "finite" in the name simply means that the number of states is fixed and enumerable — you can list all of them ahead of time, which distinguishes a state machine from more open-ended behavioral models where the space of possible behaviors is not bounded.

The appeal of state machines for game development is that they make complex behavior tractable by breaking it into discrete, manageable pieces. Instead of writing a single monolithic function that handles every possible situation a character might be in, with nested conditionals and flags to track what is happening, you write a separate piece of logic for each state and a separate specification for each transition. The idle state might play a breathing animation and check for movement input. The walking state might play a walk animation, move the character based on input direction, and check for jump input. The jumping state might apply upward velocity, play a jump animation, and check for collision with the ground. Each state is simple in isolation, and the complexity of the overall behavior emerges from the transitions between them rather than from any single piece of code.

In Rust and Bevy, state machines can be implemented at multiple levels of granularity. At the highest level, Bevy provides a built-in State system for managing application-wide states like MainMenu, Playing, Paused, and GameOver. These states control which systems run (using run conditions like `in_state(GameState::Playing)`), which entities exist (using the DespawnOnExit component, renamed from StateScoped in version 0.17, which automatically removes entities when a state is exited), and what happens during transitions (using the OnEnter and OnExit schedules, which run systems once when a state is entered or left). This application-level state machine manages the broadest behavioral changes in the game: the menu state runs menu UI systems, the playing state runs gameplay systems, and the paused state runs pause overlay systems while the gameplay world remains frozen.

At a finer granularity, individual entities often have their own state machines for behavior that does not affect the whole application. An enemy's AI state (patrolling, chasing, attacking) is local to that enemy and should not be implemented with Bevy's global State system, because the global system is designed for single-instance states that affect the whole application, not for per-entity states that may differ between entities. Instead, per-entity state machines are typically implemented as a component that holds an enum representing the current state, with systems that match on the enum to determine behavior and that modify the enum to trigger transitions.

The distinction between these two granularities — application-wide states that control which systems are active, and per-entity states that control individual behavior within an active system — is important because they have different performance characteristics and different design implications. Application-wide state changes are infrequent (happening at most a few times per minute, when the player opens a menu or completes a level) and have global effects (enabling and disabling entire categories of systems). Per-entity state changes are frequent (potentially happening every frame for every enemy in the game) and have local effects (changing the behavior of one entity without affecting any others). Using the wrong mechanism for either case leads to problems: implementing per-entity AI states as global Bevy States would be absurd (you cannot have a separate global state for each enemy), while implementing the menu-to-gameplay transition as a component on a single entity would be fragile and miss the point (the transition affects everything, not just one entity).

State machines have a well-known limitation that becomes apparent as behaviors grow more complex: they can become unwieldy when the number of states and transitions grows large. A character that can stand, walk, run, crouch, crawl, jump, double-jump, wall-jump, slide, climb, swim, and fly has twelve states, and if any state can potentially transition to several others, the total number of transitions can be enormous. Each new state added to the machine increases the number of transitions quadratically in the worst case, because each existing state might need new transition logic to or from the new state. This phenomenon, sometimes called "state explosion," is why state machines are excellent for behaviors with a handful of states (a door: locked, unlocked, open, closed) but become cumbersome for behaviors with many states and complex transition rules (a character controller with a dozen movement modes).

The most common solution to state explosion is the hierarchical state machine, where states can contain sub-states. Instead of a flat list of twelve character states, you might have a top-level division between Grounded (standing, walking, running, crouching, crawling) and Airborne (jumping, double-jumping, wall-jumping, falling), with each group handling its own internal transitions and the top level handling only the transition between grounded and airborne. This reduces the total number of transitions because states within one group do not need to define transitions to states in another group directly — they transition to the group, and the group decides which sub-state to enter. Hierarchical state machines are not built into Bevy, but they can be implemented straightforwardly using nested enums in Rust, which is one of the language's strengths: an enum variant can itself contain another enum, providing natural hierarchy.

Another approach to managing behavioral complexity is the behavior tree, which is popular in game AI and deserves mention as an alternative to state machines rather than an extension of them. Where a state machine asks "what state am I in, and should I transition to another state?", a behavior tree asks "what is the highest-priority action I should be performing right now?" A behavior tree is a tree of nodes — selectors that try children in priority order, sequences that execute children in order, and leaf nodes that perform actions or check conditions — and each frame the tree is traversed from the root to determine the current behavior. Behavior trees are more flexible than state machines for complex AI because they naturally handle prioritization (if an enemy is on fire, that should override whatever else it was doing) and they do not require explicit transitions between every pair of behaviors. They are, however, more complex to implement and reason about, and for simple behaviors a state machine is almost always clearer and easier to maintain.

The relationship between state machines and the Entity Component System architecture is worth exploring because it illustrates a broader tension in game design between object-oriented thinking and data-oriented thinking. In an object-oriented game, you might implement a state machine as an object with a `current_state` field and a `update()` method that dispatches to state-specific behavior using polymorphism (a base class `State` with subclasses `IdleState`, `WalkingState`, etc.). In a Bevy ECS game, the idiomatic approach is different: the current state is a component (an enum), and the behavior for each state is a system that runs queries filtered by the current state value. Instead of the state owning its behavior, the behavior (the system) selects the entities it applies to by their state component. This inversion might feel unnatural to developers coming from object-oriented backgrounds, but it aligns with the ECS philosophy of separating data (the state component) from behavior (the systems that operate on it), and it enables Bevy's automatic parallelism: multiple systems processing different states can run concurrently, because they are accessing different sets of entities.

State machines are one of those concepts that seem simple when described in the abstract but reveal surprising depth when applied to real game development problems. The choice of state granularity, the organization of transitions, the handling of state entry and exit effects, the interaction between multiple state machines operating simultaneously (the game state, the player's movement state, the UI navigation state) — each of these is a design decision that affects the clarity, maintainability, and correctness of the game's behavior. The state machine does not solve these problems by itself, but it provides a structured framework within which the problems can be clearly stated, methodically addressed, and, when things go wrong, systematically debugged, which is more than can be said for the alternative of ad hoc conditionals and boolean flags scattered throughout the codebase.
