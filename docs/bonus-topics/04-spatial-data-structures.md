# Spatial Data Structures: Organizing Objects in Virtual Space

One of the most common questions a game asks, thousands of times per frame, is "what is near this thing?" A bullet needs to know which enemies it might have hit. A light needs to know which objects it illuminates. A camera needs to know which parts of the world are visible. A character needs to know what is within interaction range. An artificial intelligence needs to know which threats are nearby. In each case, the naive approach — comparing every object against every other object — becomes prohibitively expensive as the number of objects grows. If you have a thousand objects and you check every pair, that is nearly half a million comparisons per query. If you have ten thousand objects, it is fifty million. The game loop cannot absorb this cost and still render at sixty frames per second, which is why spatial data structures exist: they organize objects by their positions so that proximity queries can be answered by examining only a fraction of the total objects, reducing the cost from quadratic (proportional to the square of the number of objects) to something much more manageable.

The simplest spatial data structure is the grid, and it is worth understanding thoroughly because many more sophisticated structures are variations on the same theme. Imagine dividing the game world into a regular grid of cells, like a chessboard but potentially much larger and potentially three-dimensional. Each cell maintains a list of the objects that are currently inside it. When you need to find everything near a particular position, you look up the cell that contains that position and its neighboring cells, and check only those objects. Instead of searching the entire world, you search a small, local neighborhood, and the cost of the search depends on the number of objects in those few cells rather than the total number of objects in the world. For worlds where objects are distributed roughly uniformly, grids are extremely effective, simple to implement, and cache-friendly because each cell's objects are stored together in memory.

The limitation of regular grids is that they work poorly when objects are clustered in some regions and sparse in others, or when the world is very large but most of it is empty. A grid that covers a million square meters at one-meter resolution requires a million cells, most of which will be empty, wasting memory. A grid with coarser resolution reduces memory use but puts more objects in each cell, reducing the benefit of the spatial indexing. This tradeoff between resolution and coverage motivates more adaptive structures that allocate fine resolution only where it is needed.

The quadtree (in two dimensions) and the octree (in three dimensions) address this by recursively subdividing space into smaller regions only where objects exist. A quadtree starts with a single node representing the entire world. When more than a threshold number of objects occupy a node, it subdivides into four children, each covering one quadrant of the parent's area. The process continues recursively, creating fine subdivisions in dense areas and leaving large, coarse nodes in empty areas. This adapts automatically to the distribution of objects: a region with a thousand clustered enemies will be subdivided deeply, while an empty desert remains a single large node. Queries work by traversing the tree from the root, descending only into nodes that overlap the query area, which prunes vast swaths of the world from consideration.

Bounding Volume Hierarchies, commonly known by the abbreviation BVH, take a different approach. Instead of subdividing the space itself, they group objects by proximity and wrap each group in a bounding volume — typically an axis-aligned bounding box (a rectangular box whose sides are parallel to the coordinate axes). The hierarchy is a tree where each leaf node is a single object and each internal node is a bounding box that tightly encloses all objects in its subtree. To find objects near a point or to test for intersection with a ray, you start at the root and test against the bounding box: if the query does not intersect the bounding box, none of the objects inside it can match, and you can skip the entire subtree. If it does intersect, you descend into the children and repeat. BVHs are the dominant structure in modern ray tracing (including Bevy's experimental Solari renderer) because they efficiently organize objects for ray intersection tests, and the hardware-accelerated ray tracing units on modern GPUs are specifically designed to traverse BVH structures.

The choice between these structures — and others like spatial hash maps, k-d trees, and R-trees — depends on the specific characteristics of your game. Grids are best for uniform distributions with a known world size. Quadtrees and octrees are best for non-uniform distributions where the world size may be large but sparsely populated. BVHs are best for ray queries and are the standard in rendering. Spatial hash maps (which hash positions to buckets instead of using a fixed grid) are excellent for large worlds with unbounded coordinates, because they allocate memory only for buckets that contain objects, without requiring a predefined world size. K-d trees (binary trees that alternate splitting along different axes) are excellent for nearest-neighbor queries, which are common in AI decision-making and pathfinding. Each structure makes different tradeoffs between build time (how long it takes to construct or update the structure), query time (how long it takes to answer a query), memory use, and support for dynamic objects (objects that move between frames, requiring the structure to be updated).

For Bevy game development, you will encounter spatial data structures in several contexts. Physics engines like Avian use broadphase collision detection (typically spatial hashing or BVH) to quickly identify pairs of objects that might be colliding, before performing expensive narrow-phase collision checks on only those pairs. The rendering pipeline uses frustum culling (testing each object against the camera's viewing volume) and potentially occlusion culling (testing whether objects are hidden behind other objects) to avoid rendering invisible geometry, and these tests are accelerated by spatial structures. If you implement your own game mechanics that involve proximity — aggression ranges for enemies, pickup ranges for items, area-of-effect abilities — you will likely want a spatial index to avoid the quadratic cost of checking every pair.

The deeper lesson of spatial data structures is a general principle that applies throughout computer science: the way you organize data determines the efficiency of the operations you can perform on it. A flat list of objects supports every operation but makes some operations (proximity queries) expensive. A spatial tree supports proximity queries efficiently but makes other operations (iterating over all objects in insertion order) less natural. There is no universal best structure — there is only the structure that makes the operations you perform most frequently as cheap as possible, and choosing that structure wisely is one of the skills that separates games that run smoothly from games that chug under load.

Understanding spatial data structures also illuminates why the Entity Component System architecture pairs so well with game development. In an ECS, components are stored by type in dense arrays, which is effectively a one-dimensional spatial structure optimized for the most common operation: iterating over all entities with a particular set of components. But for operations that depend on the spatial positions of entities rather than their types, the ECS storage does not help, and you need an additional spatial index. The two structures are complementary: the ECS organizes data by type for fast component access, and the spatial index organizes data by position for fast proximity queries. A well-designed Bevy game uses both, choosing the right index for each query, and the result is a system that handles both "give me all entities with Health components" and "give me all entities within ten meters of the player" with equal efficiency.
