# Networking for Games

There is a saying among game developers that networking is the hardest problem in game development, and while every specialty claims its own problems are the hardest, this particular claim has more truth to it than most. The difficulty is not primarily in the protocol-level details of sending bytes over a wire, though those details matter. The difficulty is that networking forces you to confront the fundamental nature of physical reality: that information cannot travel faster than light, that two distant computers cannot agree on what time it is, and that any communication channel can lose, corrupt, duplicate, or reorder messages. In a single-player game, you have one authoritative simulation running on one machine, and the player sees exactly what is happening. In a networked game, you have multiple simulations running on multiple machines connected by an unreliable, high-latency communication channel, and you need to create the illusion that all players are sharing a single, consistent, responsive experience. The gap between those two scenarios is enormous, and bridging it requires techniques drawn from distributed systems theory, signal processing, and a great deal of pragmatic engineering.

Let us begin with the two fundamental network architectures for games: client-server and peer-to-peer. In a client-server architecture, one machine (the server) runs the authoritative simulation of the game world, and all other machines (the clients) connect to the server, send their inputs, and receive the game state in return. The server is the single source of truth: when there is a disagreement about what happened, the server's version is correct, and the clients adjust accordingly. This architecture has several significant advantages. It simplifies anti-cheat enforcement because the server can validate all client inputs and reject impossible actions. It scales relatively well because clients only need to communicate with the server, not with each other. And it provides a natural point of control for game logic, matchmaking, and persistence. The primary disadvantage is that it requires a server, which costs money to operate and introduces a single point of failure. If the server goes down, the game is over for everyone.

In a peer-to-peer architecture, there is no central server. Each player's machine communicates directly with every other player's machine, and some mechanism is used to keep the game state consistent across all of them. The advantage is that there is no server to provision or pay for, and the game can continue as long as at least two players are connected. The disadvantages are substantial, though. Every player needs to communicate with every other player, so the number of connections grows quadratically with the number of players. Anti-cheat is much harder because there is no trusted authority to validate inputs. And achieving consistency is more complex because there is no single source of truth. Despite these challenges, peer-to-peer architectures have been used very successfully in certain genres, particularly fighting games, where the number of players is small (typically two) and the input requirements are well-suited to the lockstep synchronization model that peer-to-peer naturally supports. Many modern games use hybrid approaches: a dedicated server handles authoritative game logic, but voice chat and other non-critical data may flow peer-to-peer.

## Transport Protocols and the Nature of Latency

At the transport layer, the two protocols that matter for game networking are the Transmission Control Protocol (TCP) and the User Datagram Protocol (UDP). TCP is a reliable, ordered protocol: it guarantees that every byte you send arrives at the destination, in the order you sent it, without corruption or duplication. It achieves this through acknowledgment packets, retransmission of lost data, and flow control mechanisms that adjust the sending rate to match the network's capacity. These guarantees come at a cost in latency. When a TCP packet is lost, all subsequent data is held back until the lost packet is retransmitted and received, a phenomenon called head-of-line blocking. For many applications (web browsing, file transfer, email), this tradeoff is perfectly acceptable. For a fast-paced game where the state changes sixty times per second, it is often not.

UDP is a simpler, unreliable protocol that provides none of TCP's guarantees. When you send a UDP packet, it might arrive, or it might not. It might arrive out of order. It might arrive twice. UDP makes no promises and performs no retransmission. This sounds terrible, and for many applications it would be, but for games it provides something invaluable: low and consistent latency. Because UDP does not wait for acknowledgments or retransmit lost packets, data flows as fast as the network allows, and a single lost packet does not hold up everything that follows. The tradeoff is that you must build your own reliability layer on top of UDP, deciding on a case-by-case basis which data needs guaranteed delivery and which can tolerate loss. Position updates for a moving character, for example, are sent many times per second, and a lost update is quickly superseded by the next one, so reliable delivery is unnecessary and even counterproductive. A chat message or a score update, on the other hand, must arrive exactly once, so you implement acknowledgment and retransmission for those specific messages.

Most modern game networking libraries build on UDP and provide a "reliable UDP" abstraction that gives you the best of both worlds: low-latency unreliable channels for time-sensitive data and reliable ordered channels for critical data, all over a single UDP socket. Libraries like ENet, GameNetworkingSockets (from Valve), and the various Rust networking crates used in the Bevy ecosystem provide these abstractions, along with connection management, packet fragmentation, and encryption.

Latency, the time it takes for data to travel from one machine to another, is the central constraint of all game networking. Physical distance alone imposes a minimum latency: light in a fiber optic cable travels at roughly two-thirds the speed of light in a vacuum, so a signal traveling from New York to London (about 5,500 kilometers of cable) takes a minimum of about 27 milliseconds one way, and that is before accounting for routing hops, processing delays at each network node, and the overhead of the protocol stack on the sending and receiving machines. In practice, round-trip times of 50 to 150 milliseconds are common for intercontinental connections, and even connections within the same country often have round-trip times of 20 to 60 milliseconds. For a game running at 60 frames per second, a single frame is about 16.7 milliseconds, so a round trip of 100 milliseconds means that the server's response to a player's input arrives six frames after the input was sent. During those six frames, the player's local game needs to show something, and what it shows had better be close to what the server eventually confirms, or the player will perceive the game as laggy, glitchy, or unresponsive.

## State Synchronization, Prediction, and Rollback

The problem of what to show the player while waiting for the server's response is the central challenge of networked game programming, and different genres have developed different solutions based on their specific requirements. The most common approach in fast-paced games (first-person shooters, action games, battle royale games) is client-side prediction with server reconciliation. The idea is simple in concept but fiendish in implementation. When the player presses a movement key, the client does not wait for the server to confirm the movement. Instead, it immediately simulates the movement locally, using the same physics and game logic that the server uses, and shows the predicted result to the player. The input is also sent to the server, which simulates it authoritatively and sends back the resulting state. When the client receives the server's state, it compares the server's position with its own predicted position. If they match, everything is fine and no correction is needed. If they differ (because of latency, packet loss, or a disagreement in simulation), the client must correct its state to match the server's.

The correction is where things get complicated. A naive correction, simply snapping the player's position to the server's position, produces visible teleporting that feels terrible. A better approach is to "rewind" the client's simulation to the point in time that the server's message refers to, apply the server's state, and then "replay" all the inputs that the client has made since that point in time, producing a new predicted state that incorporates both the server's correction and the client's subsequent inputs. This replay happens in a single frame and is invisible to the player unless the prediction error was so large that the corrected position is noticeably different from the predicted one. This technique, called server reconciliation or state rewinding, is described in detail in Gabriel Gambetta's excellent series of articles on networked game programming, and it is the foundation of the networking model used by most modern online shooters.

For other players' characters (entities that the local player does not control), the client cannot predict their movement because it does not have their inputs. Instead, it must interpolate or extrapolate based on the state updates it receives from the server. Entity interpolation displays other players at a position slightly in the past (typically one or two server ticks behind the current time), using the most recent received states to interpolate smoothly between known positions. This introduces a small amount of visual delay but produces smooth, jitter-free motion. Entity extrapolation (also called dead reckoning) attempts to predict other players' future positions based on their current velocity and acceleration, which eliminates the delay but can produce visible corrections when the prediction is wrong and the entity snaps to its actual position.

Fighting games and other genres where precise frame-by-frame synchronization is critical often use a different approach called rollback netcode. In a rollback system, both players run the simulation locally and predict the opponent's inputs (typically by assuming the opponent will repeat their most recent input). When the actual input arrives and differs from the prediction, the simulation is rolled back to the point of the incorrect prediction, the correct input is applied, and the simulation is fast-forwarded back to the current frame. If the prediction was correct, nothing happens and the game feels perfectly responsive. If the prediction was wrong, there is a brief visual discontinuity as the game corrects itself, but because the prediction is usually correct (inputs tend to be consistent from frame to frame), the corrections are infrequent and small. This approach, popularized by the GGPO library (now called GGBack), has become the standard for competitive fighting games because it provides much better responsiveness than the older "delay-based" approach, which simply delayed all inputs by a fixed number of frames to allow time for the opponent's input to arrive.

The authoritative server model, where the server has final say over all game state, is the standard for any game where cheating is a concern, which is to say essentially any competitive multiplayer game. The server validates all client inputs, rejects impossible ones, and runs the definitive simulation. Clients may predict locally for responsiveness, but they must always be prepared to accept corrections from the server. This model does not eliminate cheating entirely (aimbots and wallhacks operate on the client's rendering and input, which the server cannot control), but it prevents the most egregious forms of cheating, like teleporting, speed hacking, and spawning items, because those all require modifying the game state in ways that the server would reject.

## Why Networking Is So Hard

The reason networking is considered the hardest problem in game development is not any single technical challenge but the way that all of these challenges interact with each other and with every other system in your game. Adding networking to a single-player game is not like adding a new feature; it is like rebuilding the foundation of your house while living in it. Every system that touches game state needs to be aware that state may be provisional, that corrections may arrive from the server at any time, and that the game must handle those corrections gracefully. Your physics system needs to support rollback. Your animation system needs to handle position corrections without visible popping. Your game logic needs to distinguish between authoritative state and predicted state. Your input system needs to buffer and timestamp inputs for network transmission. Your serialization system needs to efficiently encode and decode game state for network transmission, often under tight bandwidth constraints.

The non-determinism introduced by floating-point arithmetic across different hardware platforms is another headache. Even if two machines run the exact same code with the exact same inputs, they may produce slightly different results due to differences in how their processors handle floating-point rounding. Over time, these tiny differences accumulate and cause the simulations to diverge, which means that even in a perfectly reliable lockstep system, periodic state synchronization is necessary to correct for drift. Some games address this by using fixed-point arithmetic for all gameplay-critical calculations, which is deterministic across platforms but loses the convenience and performance of hardware floating-point.

Bandwidth is another constraint that is easy to underestimate. A naive approach of sending the entire game state to every client every tick can easily exceed available bandwidth, especially on connections with limited upload speed. Practical networking systems use techniques like delta compression (sending only what changed since the last acknowledged state), interest management (sending clients only the state of entities they can perceive), quantization (reducing the precision of position and rotation values to reduce their size), and priority systems (sending updates for nearby or important entities more frequently than distant or unimportant ones).

In the Bevy ecosystem, networking is typically handled through community plugins and crates rather than being built into the engine core. Libraries like `bevy_replicon`, `lightyear`, and others provide frameworks for state replication, client prediction, and server authority that integrate with Bevy's Entity Component System. The ECS architecture is actually well-suited to networking in some ways: components are natural units of state that can be individually serialized and replicated, and systems provide clear boundaries for the simulation logic that needs to be rolled back or replayed. But the challenges described in this essay are inherent to the problem of distributed simulation over unreliable networks, and no architecture or library can make them disappear entirely. What good tools and good architecture can do is manage the complexity, providing clear abstractions and well-defined interfaces that keep the networking concerns from infecting every line of your game code. That is a worthy goal, even if the underlying problem remains, as it always has been, genuinely hard.
