# Input Handling

Every interaction a player has with a game begins with a physical action: pressing a key, moving a mouse, tilting a thumbstick, touching a screen. This physical action generates an electrical signal that travels through hardware, drivers, operating system layers, and windowing systems before finally arriving in your game code as a structured event. The journey from finger to game logic is longer and more complex than most developers realize, and understanding it helps explain why input sometimes feels sluggish, why certain polling strategies work better than others, and why the seemingly simple question of "is the player pressing the jump button?" has a surprisingly nuanced answer.

Let us trace the path of a keypress from start to finish. When you press a key on a keyboard, a mechanical switch closes (or, in the case of membrane keyboards, two conductive layers make contact), and a small microcontroller inside the keyboard detects the change. The keyboard's firmware scans its key matrix at a rate that varies by hardware, typically between 125 and 1000 times per second, and when it detects a new keypress, it sends a scan code to the computer via USB (or Bluetooth, for wireless keyboards). The USB host controller on the computer receives this scan code and raises a hardware interrupt, which the operating system's input subsystem processes. The OS translates the scan code into a virtual key code (accounting for keyboard layout, modifier keys, and other configuration), and dispatches a key event to the currently focused window. In the case of a game built with Bevy, this event is received by the windowing library (winit, in Bevy's case), which translates it into a platform-independent event and places it in an event queue that Bevy's input systems read from.

That entire chain, from finger pressing plastic to your system receiving an event, takes time. USB polling at 125 hertz introduces up to 8 milliseconds of latency. Operating system input processing adds a few more milliseconds. The game's own event processing adds the time until the next frame begins. All together, the input latency from physical action to game response can easily be 30 to 100 milliseconds, and that is before accounting for display latency (the time for the monitor to actually show the updated frame). Players are remarkably sensitive to this total latency. Fighting game players can perceive and be affected by differences of a single frame (roughly 16 milliseconds at 60 frames per second), and even casual players will describe a game as feeling "sluggish" or "unresponsive" if the total input-to-display latency exceeds about 100 milliseconds. This is why competitive gaming peripherals advertise 1000 hertz polling rates and why game developers obsess over minimizing the number of frames between input receipt and visual response.

## From Raw Input to Game Actions

A game engine's input system must bridge the gap between the low-level reality of hardware events and the high-level concepts that game logic cares about. Your game logic does not want to know that scan code 0x39 was pressed; it wants to know that the player jumped. This mapping from raw inputs to game actions is called input mapping or input binding, and it is a critical piece of infrastructure for any non-trivial game. The simplest approach is to hardcode the mapping: "if the space key is pressed, jump." This works for prototypes but quickly becomes problematic. Players expect to be able to rebind their controls, and a game that supports keyboard, gamepad, and touch input needs multiple mappings for the same action. Hardcoding these mappings throughout the codebase makes them difficult to change and impossible for the player to customize.

A better approach is to define an abstraction layer where game actions are named identifiers ("jump," "attack," "move_left") and the mapping from raw inputs to these actions is defined in a configuration that can be modified at runtime. When the input system detects a raw input, it consults the mapping table and generates action events. Game logic listens for action events rather than raw input events, which means that rebinding controls is simply a matter of updating the mapping table, and supporting a new input device means adding a new set of mappings without touching any game logic. In Bevy, this kind of input mapping can be implemented through resources and systems, or through community crates like `leafwing-input-manager` that provide a complete input mapping framework integrated with Bevy's ECS.

The distinction between different input states is more subtle than it first appears, and getting it wrong leads to maddening bugs. Consider the seemingly simple act of pressing and releasing a key. At any given moment, a key is either physically down or physically up, which is a binary state. But game logic often cares about the transitions between these states, not just the state itself. "The key is currently held down" is different from "the key was just pressed this frame" and "the key was just released this frame." In Bevy's input system, these three states are represented by the `pressed`, `just_pressed`, and `just_released` methods on the `ButtonInput` resource. The `pressed` method returns true for every frame that the key is held down. The `just_pressed` method returns true only on the first frame after the key transitions from up to down. The `just_released` method returns true only on the first frame after the key transitions from down to up.

The importance of this distinction becomes clear when you consider what happens if you use `pressed` when you should use `just_pressed`. If your jump logic checks `pressed`, the character will try to jump on every frame that the space bar is held down, not just when it is first pressed. If your game allows rapid-fire jumping, this might produce a machine-gun stream of jumps. If it does not, the jump will fire on the first frame and then be suppressed by whatever cooldown mechanism you have, but the intent was wrong. Conversely, if you use `just_pressed` for a continuous action like sprinting, the player would have to tap the sprint key every frame to maintain speed, which is obviously not what you want. Matching the right input check to the right game action is a small detail that makes a big difference in how the game feels.

Analog inputs add another dimension of complexity. A keyboard key is binary, either pressed or not, but a gamepad thumbstick produces a continuous value from negative one to positive one on each axis. Mouse movement produces a delta (change in position) on each frame. Trigger buttons on gamepads produce a continuous value from zero (not pressed) to one (fully pressed). These analog inputs need different handling than digital ones. A thumbstick value of 0.1 probably represents the stick's physical dead zone (the small amount of deflection that occurs when the stick is supposedly centered) rather than an intentional player input, so you need a dead zone threshold below which the input is treated as zero. The mapping from stick deflection to game movement is also important: a linear mapping, where the character's speed is directly proportional to stick deflection, often does not feel right because the low end of the stick's range is imprecise and the high end is saturated. Many games use a nonlinear response curve (similar to the easing functions discussed in the animation essay) that provides finer control at low deflections and more aggressive acceleration at high deflections.

## Input Buffering and Responsiveness

Input buffering is a technique drawn from fighting games that has gradually spread to other genres because of how much it improves the feel of a game. The problem it solves is this: many game actions have animations with committed durations. When a player performs an attack, the attack animation plays for a certain number of frames, and during some or all of those frames, the character cannot perform other actions. If the player presses the button for their next attack slightly before the current animation finishes, a naive input system would simply discard that input because the character is not yet in a state where it can attack. The player, who timed their input to chain smoothly from one action to the next, would perceive the game as dropping their input, which feels terrible.

Input buffering addresses this by storing recent inputs in a short buffer (typically 3 to 10 frames long). When the character enters a state where it can accept new actions, the system checks the buffer for any pending inputs and executes the most appropriate one. This means the player can press the next action's button a few frames early, and the action will execute as soon as it becomes possible, creating the sensation of smooth, responsive chaining. The buffer duration is a tuning parameter: too short and the player still feels like inputs are being dropped, too long and actions execute noticeably after the button press, feeling delayed. Different games tune this differently based on their pace and genre. Fast action games like Devil May Cry or Bayonetta use generous input buffers to enable rapid combo chaining. Souls-like games use smaller buffers because the deliberate, committed nature of each action is part of the intended difficulty.

A related concept is input queuing, where the game maintains a queue of pending actions and executes them in sequence. This is common in strategy games and role-playing games where the player issues commands faster than they can be executed. In an action context, input queuing allows the player to "stack" inputs during an animation and have them execute in order when the character becomes available.

Coyote time is another input-handling technique that improves game feel by being slightly generous with the player's timing. In a platformer, coyote time refers to a brief window (typically 50 to 150 milliseconds) after a character walks off the edge of a platform during which the player can still press jump and the game will honor it as if the character were still on the ground. Without coyote time, a player who presses jump one frame after leaving the platform edge will fall helplessly, which feels unfair because from the player's perspective they pressed jump while they were still on the platform. The visual representation on screen lags behind the game state by enough that the player's perception and the game's physics do not quite agree on when the character left the ground, and coyote time bridges this perceptual gap. Similarly, jump buffering allows a player to press jump a few frames before landing, and have the jump execute immediately upon landing, so the player does not need pixel-perfect timing to chain movements together.

## Multiple Input Devices and Accessibility

Supporting multiple input devices is essential for modern games and introduces its own set of challenges. A player might switch between keyboard and gamepad mid-session, or use a keyboard for some actions and a mouse for others simultaneously. The input mapping system needs to handle this gracefully, detecting which device is active and adjusting the UI prompts accordingly (showing keyboard key icons when the player last used a keyboard, gamepad button icons when they last used a gamepad). Touch input on mobile devices adds further complexity: touches have positions and areas (not just binary pressed/released states), multi-touch gestures need to be interpreted (pinch to zoom, swipe to scroll, tap to select), and the lack of physical buttons means that on-screen virtual controls must be rendered and hit-tested, consuming screen real estate and adding a layer of indirection between the player and the game.

Accessibility in input handling goes far beyond supporting multiple device types. Players with motor disabilities may not be able to press buttons rapidly, hold buttons for extended periods, use analog sticks precisely, or coordinate inputs across both hands simultaneously. Good accessibility options include the ability to remap any action to any input (including mouse buttons, gamepad buttons, and keyboard keys interchangeably), toggle options for actions that otherwise require holding a button (hold-to-sprint versus toggle-sprint), adjustable input sensitivity and dead zones, the option to reduce or eliminate quick-time events and rapid-button-press mechanics, and one-handed control schemes that map all essential actions to a single hand's worth of inputs.

The Xbox Adaptive Controller, released by Microsoft in 2018, demonstrated the gaming industry's growing awareness of input accessibility. It is a large, flat controller with two big programmable buttons and nineteen 3.5mm jacks that can accept a wide range of external switches, buttons, and joysticks, allowing players with severe motor disabilities to construct a custom control setup that works for their specific abilities. Supporting this kind of device in your game does not require special hardware-level code; it presents itself to the operating system as a standard gamepad. What it does require is a flexible input mapping system that does not assume the player has two thumbs on two thumbsticks and ten fingers distributed across triggers, bumpers, and face buttons.

In Bevy, input handling is built around a set of resources and events that provide access to the state of all connected input devices. The `ButtonInput<KeyCode>` resource tracks keyboard state. The `ButtonInput<MouseButton>` resource tracks mouse button state. The `Axis<GamepadAxis>` and `ButtonInput<GamepadButton>` resources track gamepad state. Mouse motion is available through the `MouseMotion` event, and touch input through the `TouchInput` event. These resources are updated by Bevy's built-in input systems at the beginning of each frame, before your game systems run, so they always reflect the current state of the hardware. Building an input mapping layer on top of these primitives is a matter of defining a set of game actions, creating a mapping from inputs to actions, and writing a system that reads the raw input resources and produces action events or updates an action state resource that your game logic can query.

The key takeaway from all of this is that input handling is not just plumbing. It is one of the most direct contributors to how your game feels to play. A game with perfectly implemented mechanics and beautiful art can still feel terrible if the input handling is slightly off, if there is an extra frame of latency, if the dead zones are wrong, if the input buffer is too short, if the analog response curve is not tuned to match the game's speed and precision requirements. Conversely, a game with modest production values but impeccable input handling will feel responsive, crisp, and satisfying in a way that players notice even if they cannot articulate why. Taking the time to understand the full input pipeline, from hardware to event to action to response, and to tune each stage carefully, is one of the highest-return investments you can make in your game's quality.
