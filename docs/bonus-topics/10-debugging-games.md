# Debugging Games

Debugging a game is a different animal from debugging most other software. When something goes wrong in a web application, you can usually reproduce the problem by sending the same HTTP request, inspect the database state, read the logs, and step through the code in a debugger at your leisure. When something goes wrong in a game, the bug might only manifest when three enemies are on screen while the player is mid-jump during a frame rate spike, and by the time you notice the bug, the game state has advanced by hundreds of frames and the conditions that caused it are gone. Games are real-time systems that produce sixty or more frames per second, each frame a complex interplay of physics, animation, input, rendering, audio, and game logic. They are visual, meaning that many bugs manifest as something looking wrong rather than a value being wrong in a way a test could catch. They are often non-deterministic, depending on frame timing, floating-point precision, and the exact sequence of player inputs. And they involve state spaces so vast that exhaustive testing is impossible. All of this makes debugging games uniquely challenging and uniquely rewarding when you finally track down the source of a confusing behavior.

The first and most important debugging tool is not any particular piece of software; it is a systematic approach to narrowing down the problem. When you observe a bug, the temptation is to immediately start forming hypotheses and testing them: "maybe the collision detection is wrong," "maybe the animation is overriding the position," "maybe the entity is being despawned prematurely." These hypotheses are fine as long as you are disciplined about testing them one at a time and not jumping to a new hypothesis before you have confirmed or eliminated the current one. The scientific method works here just as it does in any other empirical domain: observe the phenomenon carefully, form a hypothesis about the cause, design an experiment that would distinguish between the hypothesis being true and false, run the experiment, and let the result guide your next step. Experienced debuggers are not people who magically guess the right answer on the first try; they are people who eliminate wrong answers quickly and systematically.

## Visual Debugging: Overlays, Gizmos, and the Inspector

One of the most powerful categories of debugging tools for games is visual debugging, the practice of rendering additional information on top of the game to make invisible state visible. The game world is governed by data that is not ordinarily shown to the player: collision shapes, physics velocities, navigation meshes, AI sight lines, trigger volumes, entity bounding boxes, coordinate axes. When something goes wrong, being able to see this data overlaid on the game world is invaluable because it transforms abstract numerical debugging into spatial, intuitive reasoning. You can see that the collision box is not aligned with the sprite, or that the navigation mesh has a gap where it should not, or that the AI's line-of-sight cone is pointing in the wrong direction.

Bevy provides a gizmo system that makes it straightforward to draw debug shapes in the game world. Gizmos are immediate-mode drawing primitives, lines, circles, rectangles, spheres, arrows, and other shapes, that you draw from your systems and that are rendered for a single frame without persisting. This makes them perfect for debugging because you can add a gizmo draw call to any system, see the result, and remove it when you are done without worrying about cleanup or persistent state. A common pattern is to create debug-only systems that are gated behind a feature flag or a runtime toggle, so they can be enabled during development and disabled in release builds. You might have a system that draws the bounding boxes of all entities with collision components, another that draws velocity vectors as arrows, and another that draws the wireframes of navigation meshes.

Bevy's entity inspector capabilities, whether built-in or provided by community tools like `bevy-inspector-egui`, offer another crucial layer of debugging. An entity inspector provides a runtime UI that lets you browse all entities in the world, see their components, and in many cases modify component values live. This is extraordinarily useful for diagnosing issues because it lets you examine the actual runtime state of any entity without adding print statements or breakpoints. If a character is not moving, you can inspect its velocity component and see that it is zero, then inspect its input component and see that no input is being registered, which immediately narrows the problem to the input system rather than the movement system. The ability to modify values at runtime also lets you test hypotheses quickly: "what if I set this entity's speed to a large value? Does it move then?" If it does, the problem is in how speed is being calculated, not in how speed is being applied.

## Logging Strategies and Diagnostic Plugins

Logging is the workhorse of debugging in any software, and games are no exception, but effective logging in a game requires some discipline because the volume of data can be overwhelming. A game running at 60 frames per second that logs a message per entity per frame can produce millions of log lines per minute, which is worse than useless because the signal is completely drowned in noise. The key is to be strategic about what you log and when. Log state changes rather than continuous state: instead of logging "entity position is (3.5, 2.1)" every frame, log "entity entered region B" when the transition happens. Use log levels appropriately: trace-level logging for per-frame details that you only enable when actively debugging a specific system, debug-level for significant state changes during development, info-level for major game events, warn-level for recoverable anomalies, and error-level for things that should never happen. Most logging frameworks, including the `tracing` crate that Bevy uses, support structured logging with spans and fields, which lets you attach context to log messages (which entity, which system, which frame) and filter the output to show only the messages relevant to your current investigation.

Bevy includes several diagnostic plugins that provide real-time insight into the engine's behavior. The `FrameTimeDiagnosticsPlugin` tracks frame times and frames per second, which is essential for identifying performance problems. The `EntityCountDiagnosticsPlugin` tracks the number of entities in the world, which helps detect entity leaks (entities being spawned but never despawned). The `SystemInformationDiagnosticsPlugin` provides information about the system hardware. These plugins write their data to Bevy's diagnostics system, which can be consumed by various display plugins that render the information as an overlay on the game window. You can also write custom diagnostic plugins that track game-specific metrics, like the number of active projectiles, the size of a spatial index, or the time spent in a specific phase of your game logic.

For serious performance debugging, the Tracy profiler is an invaluable tool. Tracy is an open-source profiler designed for real-time applications, particularly games, and it provides a detailed timeline view showing exactly how long each system takes to execute, how systems are scheduled across threads, where the CPU is spending its time, and where frames are exceeding their time budget. Bevy has built-in support for Tracy through the `trace_tracy` feature flag, which instruments the engine's scheduler and systems with Tracy markers. When you run your game with Tracy connected, you get a frame-by-frame timeline that shows the execution of every Bevy system, the scheduling of work across thread pools, and any custom spans you have added in your own code. This level of visibility is essential for diagnosing performance issues that are caused by unexpected system ordering, excessive parallelism overhead, or individual systems that take too long.

## Common Bevy Debugging Patterns and Reproducing Intermittent Bugs

Certain categories of bugs appear frequently in Bevy projects, and recognizing these patterns can save you significant debugging time. One of the most common is systems that depend on each other's output but are not explicitly ordered, causing them to sometimes run in the wrong order. Because Bevy's scheduler runs systems in parallel when it can, two systems that both access the same components may run in either order on any given frame. If system A writes a value that system B reads, and they are not ordered relative to each other, system B might sometimes read the value from the current frame (if A ran first) and sometimes read the stale value from the previous frame (if B ran first). This produces intermittent bugs that depend on scheduling order, which itself depends on thread timing and system load, making them difficult to reproduce. The fix is to add explicit ordering constraints using Bevy's system ordering APIs, but the difficult part is recognizing that the bug is an ordering issue in the first place. If a behavior is inconsistent from frame to frame, or between runs, system ordering should be one of the first things you investigate.

Another common pattern is query conflicts, where two systems attempt to mutably access the same component type simultaneously. Bevy's scheduler will detect these conflicts at runtime and either serialize the systems (running them sequentially instead of in parallel) or, in some configurations, panic. If you notice that adding a new system causes an unexpected performance drop, it may be because the new system's queries conflict with existing systems, forcing them to run sequentially when they were previously running in parallel. The `ambiguity_detection` setting can help identify these conflicts during development.

Entity lifecycle bugs are another frequent source of confusion. If you spawn an entity in one system and try to access it in another system within the same frame, the entity might not exist yet because Bevy's command buffers are not applied until a synchronization point. This means that `commands.spawn(...)` does not immediately create the entity; it queues a command that will be processed later. If another system tries to query for the newly spawned entity before the command buffer is flushed, it will not find it. Similarly, despawning an entity does not immediately remove it; it queues a despawn command. Understanding the timing of command buffer application is essential for avoiding bugs where entities seem to appear or disappear a frame late.

Reproducing intermittent bugs is perhaps the most frustrating aspect of game debugging, and it requires a combination of patience, instrumentation, and sometimes luck. The first strategy is to make the game more deterministic. If you can seed your random number generator with a fixed value, record the exact sequence of player inputs, and fix the frame rate to a constant value, you can often make an intermittent bug reproducible by replaying the same sequence of inputs with the same random seed. This is called a replay system, and while building one is a significant investment, it pays for itself many times over in debugging productivity. Even a partial replay system that records and replays inputs for a few seconds is enormously useful for bugs that happen during specific gameplay sequences.

The second strategy is to add assertions and invariant checks that detect the conditions leading to the bug, even if the bug itself is not immediately visible. If a bug manifests as a character teleporting to an incorrect position, add an assertion that fires whenever a character's position changes by more than a threshold in a single frame. This assertion will trigger even in cases where the teleport is brief enough that you would not notice it visually, giving you a reliable reproduction and a stack trace to work from. Liberal use of `debug_assert!` in Rust, which is compiled away in release builds, lets you add these checks without worrying about performance impact.

The third strategy is logging with enough context to reconstruct the sequence of events after the fact. When the bug finally does occur, you want your logs to tell you exactly what happened in the frames leading up to it. This means logging system entry and exit times, entity state changes, input events, and any other data that might be relevant. Ring buffers that store the last N seconds of log data are useful here because they give you detailed context around the bug without the cost of writing millions of log lines to disk.

Debugging games is hard, but it is also a deeply satisfying intellectual challenge. Every bug is a puzzle, and solving it requires understanding the system deeply, thinking about timing and state and causality, and sometimes recognizing that the bug is not where you think it is but somewhere else entirely, in a system you were not even looking at, caused by a condition you did not think was possible. The tools and techniques described in this essay, gizmos, inspectors, logging, profiling, deterministic replay, assertions, will not eliminate the difficulty, but they will give you a systematic framework for approaching it, and that makes all the difference between flailing and making progress.
