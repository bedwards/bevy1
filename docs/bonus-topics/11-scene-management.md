# Scene Management

A game world is, at its heart, a collection of entities arranged in some meaningful configuration. A level in a platformer is a collection of platforms, enemies, collectibles, and spawn points. A room in a role-playing game is a collection of furniture, non-player characters, light sources, and interactive objects. A saved game is a snapshot of all the entities in the world and the values of all their components at a particular moment in time. The systems that create, load, save, and transition between these collections of entities fall under the broad umbrella of scene management, and while the concept is straightforward, the practical details involve some of the most architecturally consequential decisions you will make in your game project.

The word "scene" is used in different ways by different engines, and it helps to be precise about what we mean. In its most general sense, a scene is a serialized collection of entities and their components, a description of a set of game objects that can be saved to a file and later instantiated into a running game world. This is the sense in which Bevy uses the term, and it is a powerful and flexible concept because it makes no assumptions about what a scene represents. A scene could be an entire game level, or it could be a single prefab (a reusable template for a complex entity, like a treasure chest with its sprite, collision box, interaction trigger, and loot table). A scene could represent the player character with all their equipment, or a particle effect with dozens of sub-entities. The scene format is just a container for entity data; the semantics of what that data means are determined by the components it contains and the systems that operate on them.

Bevy's scene system serializes entities and their components into a format that can be written to and read from files. The engine has evolved its scene representation over time, and the current approach supports both a human-readable format suitable for hand-editing and a more compact binary format suitable for asset distribution. When you load a scene, Bevy creates the entities described in the scene file and inserts them into the world with the components specified. When you save a scene, Bevy serializes the entities you specify (or all entities, depending on your approach) and writes them to a file. This round-trip, from live entities to serialized data and back, is the foundation of both level loading and save/load systems.

## The Relationship Between Scenes and States

One of the most important architectural decisions in a game is how to manage the transition between different major modes of the game: the main menu, the gameplay, the pause screen, the cutscene, the loading screen. These modes are typically modeled as states in a state machine, and the transition between them involves tearing down one set of entities and systems and setting up another. Bevy's state system (`States` trait and associated scheduling constructs) provides the mechanism for defining these states and running different sets of systems depending on which state is active, while the scene system provides the mechanism for defining which entities should exist in each state.

The relationship between states and scenes is complementary but not identical. A state defines the behavioral mode of the game: which systems run, which inputs are accepted, which rules apply. A scene defines the entity content of the game: which entities exist and what their components are. In a simple game, there might be a one-to-one correspondence between states and scenes: the "main menu" state loads the main menu scene, and the "gameplay" state loads the level scene. In a more complex game, the relationship is many-to-many: the "gameplay" state might load multiple scenes (a base level scene, plus additional scenes for dynamic content like enemies and items), and a single scene (like the player character) might persist across multiple states.

Managing this relationship cleanly requires a strategy for entity lifecycle across state transitions. When the game transitions from the main menu to gameplay, the main menu entities need to be despawned and the level entities need to be spawned. When the game transitions from gameplay to a pause screen, the level entities might need to remain but be frozen (systems that update them stop running), and the pause menu entities need to be spawned on top. When the player returns from the pause screen to gameplay, the pause menu entities need to be despawned and the level entities need to resume updating. Bevy provides the `StateScoped` component (and related mechanisms) to help manage this, allowing you to tag entities with the state they belong to so they can be automatically despawned when that state exits. This is enormously helpful for preventing entity leaks, where entities from a previous state linger in the world because nobody remembered to despawn them, which is one of the most common bugs in games with multiple states.

A typical pattern in Bevy projects is to use the `OnEnter` and `OnExit` schedules for a given state to handle scene loading and unloading. When entering the gameplay state, an `OnEnter(GameState::Playing)` system loads the level scene and spawns the player character. When exiting the gameplay state, entities marked with `StateScoped(GameState::Playing)` are automatically cleaned up. This keeps the entity lifecycle tied to the state lifecycle in a predictable and maintainable way.

## Level Loading Strategies

How you load levels is one of those decisions that seems simple at first and gradually reveals its complexity as your game grows. The simplest approach is to define your levels entirely in code: a system that spawns entities with hardcoded positions, sizes, and properties. This works for prototypes and very simple games, but it quickly becomes unwieldy for levels of any complexity because you have to recompile the game every time you change a level, and the level design is buried in Rust code rather than being accessible to designers or level editors.

The next step up is to define levels in data files that are loaded at runtime. This could be a Bevy scene file, a custom level format parsed by a loading system, or an export from a third-party level editor. Using a level editor dramatically improves the iteration speed of level design because designers can place objects, test the layout, and adjust it without touching any code. Tools like the Tiled map editor (for 2D tile-based games), LDtk (a modern 2D level editor), and Blender (for 3D environments) can export level data in formats that Bevy can parse, either natively or through community plugins. The exported data typically includes the positions and types of all objects in the level, and a loading system in your game translates this data into spawned entities with the appropriate components.

For games with large or numerous levels, the loading time becomes a significant concern. A naive approach of loading the entire level synchronously at the start of the gameplay state will produce a noticeable freeze, which is unacceptable for any game that values polish. The standard solutions are loading screens and asynchronous loading. A loading screen is a separate state that displays a progress indicator while the level data is being loaded and processed in the background. Bevy's asset loading system supports asynchronous loading natively: when you request an asset (a scene file, a texture, a mesh), the load happens on a background thread, and you receive a handle that transitions from "loading" to "loaded" when the data is ready. Your loading screen system can poll the status of these handles and transition to the gameplay state once all required assets are loaded.

For very large levels, even asynchronous loading of the entire level at startup may be insufficient because the level is too large to fit in memory all at once. This is where level streaming comes in: the technique of loading and unloading sections of the level dynamically as the player moves through the world. The level is divided into chunks (regions, rooms, sectors, or whatever spatial subdivision makes sense for the game), and only the chunks near the player are loaded at any given time. As the player moves, chunks that are now distant are unloaded to free memory, and chunks that are now nearby are loaded in anticipation of the player reaching them. This requires careful design to ensure that loading happens early enough that the player never encounters an unloaded area, and that unloading happens gracefully without visible popping or hitching.

Open-world games take level streaming to its extreme, managing worlds that may contain millions of objects spread over vast areas. These games typically use a hierarchical spatial system where the world is divided into coarse regions, each of which contains finer chunks, each of which contains individual objects. The level-of-detail system works in concert with the streaming system: distant regions are represented by simplified stand-in geometry (imposters or low-detail meshes) that are swapped for full-detail content as the player approaches. The total amount of data managed by the streaming system far exceeds what could fit in memory, so the system must make intelligent decisions about what to prioritize based on the player's position, movement direction, and viewing angle.

## Saving and Loading Game State

Saving and loading a game is, in principle, simple: serialize the entire game state to a file, and later deserialize it to restore the game to that state. In practice, it is one of the most architecturally challenging features a game can support, because "the entire game state" is a much more complex and subtle concept than it first appears.

The first challenge is deciding what to save. Not all components need to be serialized. Some components are derived from other data and can be recomputed on load (like render-related components generated from sprite and mesh data). Some components are transient and meaningless in a saved context (like animation playback state). Some components reference runtime resources like asset handles that will be different in a future session. You need a strategy for marking which components should be saved and which should be reconstructed, and this strategy needs to be maintained as you add new components to the game.

The second challenge is versioning. If you ship a save file format and then change your component layouts in a patch (adding a field, renaming a component, changing a type), old save files will not deserialize correctly with the new code. You need either a migration system that can upgrade old save data to the new format, or a serialization format that is tolerant of missing or extra fields, or both. This is a problem that most games solve imperfectly, and many games simply declare old saves incompatible with new versions, which is acceptable during development but not after release.

The third challenge is entity references. If entity A has a component that references entity B (for example, a "following" component on a minion that references the player entity), the entity identifiers in the save file need to be resolved correctly on load. Entity identifiers in Bevy are runtime-assigned and will not be the same across sessions, so you need a mapping layer that translates save-file entity identifiers to runtime entity identifiers during deserialization. Bevy's scene system handles this by maintaining an entity map that remaps identifiers during scene instantiation, but if you have custom reference types or relationships between entities, you may need to extend this mapping to cover them.

For many games, a pragmatic approach to saving is to avoid serializing the full entity world and instead save a higher-level representation of the game state: which level the player is on, what items they have collected, which enemies have been defeated, what quest flags have been set. On load, the game loads the level scene normally and then applies the saved state on top: removing collected items, despawning defeated enemies, setting quest flags. This approach is simpler to implement and more robust to code changes because the saved data is a small, well-defined set of game-specific state rather than a serialization of the entire entity hierarchy. The tradeoff is that it cannot perfectly reproduce every detail of the game state (the exact position of every physics object, the exact frame of every animation), which may or may not matter depending on the game.

Regardless of the approach you choose, the design of your save system should be considered early in the project, not bolted on at the end, because it influences how you structure your components and relationships. If you know from the start that certain components need to be serializable, you can ensure they derive the right traits and avoid containing non-serializable types. If you know that entity references need to survive serialization, you can use a stable identifier scheme from the beginning rather than retrofitting one later. Scene management, in all its forms, from loading screens to level streaming to save files, is fundamentally about the lifecycle of your entity world, and thinking clearly about that lifecycle from the start of your project will save you from painful refactoring later.
