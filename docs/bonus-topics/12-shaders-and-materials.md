# Shaders and Materials: Programming the GPU

Every pixel you see on screen when playing a game was computed by a small program running on the Graphics Processing Unit (GPU), and that program is called a shader. The name comes from the original purpose — computing the shading (color and brightness) of surfaces based on lighting — but modern shaders do far more than that. They determine the shape of geometry on screen, the color of every pixel, the post-processing effects applied to the final image, and increasingly the simulation of particles, physics, and other computational tasks that benefit from the GPU's massively parallel architecture. Understanding shaders, even at a conceptual level, transforms your understanding of game rendering from "the engine draws things" to an appreciation of the pipeline through which data becomes images.

A GPU is a fundamentally different kind of processor from the Central Processing Unit (CPU) that runs your game logic. Where a CPU is optimized for running complex, branching logic on a few threads very quickly, a GPU is optimized for running simple, uniform operations on thousands of threads simultaneously. A modern gaming GPU might have several thousand shader cores, each capable of executing shader instructions in parallel with all the others. This architecture is a perfect match for rendering, where the same operations (compute the color of a pixel based on lighting and material properties) need to be applied to millions of pixels every frame, with each pixel's computation independent of every other pixel. The GPU processes all those pixels in parallel, which is why real-time rendering is possible at all — computing millions of pixel colors sequentially on a CPU would be far too slow for sixty frames per second.

The rendering pipeline processes geometry through a series of stages, each handled by a different type of shader. The vertex shader runs once for every vertex (corner point) of every triangle in the scene, transforming the vertex from its position in model space (relative to the object) through world space (relative to the scene) to screen space (relative to the display). This transformation uses the matrix math described in the linear algebra essay — the model matrix positions the object in the world, the view matrix positions the camera, and the projection matrix converts 3D coordinates to 2D screen coordinates. The vertex shader can also modify vertex positions for effects like waves on water, wind-blown vegetation, or skeletal animation.

After the vertex shader, the GPU rasterizes the triangles — determining which pixels on screen are covered by each triangle — and then runs the fragment shader (also called the pixel shader) once for every pixel of every triangle. The fragment shader is where most of the visual magic happens: it computes the final color of each pixel based on the surface material properties (color, roughness, metallic-ness), the lighting environment (direction and color of lights, ambient light, shadows), and any textures mapped onto the surface. The fragment shader might sample a color texture to get the surface's base color, a normal map to add fine surface detail that affects lighting, a roughness map to vary how shiny different parts of the surface are, and an emissive map to make certain areas glow. The combination of all these inputs, processed through a lighting model like Physically Based Rendering (PBR), produces the final pixel color that appears on screen.

In Bevy, the connection between shaders and game code is mediated by materials. A material, in rendering terms, is a description of how a surface should look: its color, its texture, its shininess, its transparency, and any other visual properties. Bevy's StandardMaterial provides a PBR material with properties like base_color, metallic, perceptual_roughness, emissive, and various texture map slots. When you attach a MeshMaterial3d component with a StandardMaterial to an entity, Bevy uses the StandardMaterial's built-in shader to render that entity, automatically handling lighting, shadows, and all the complexity of PBR shading. For many games, StandardMaterial is sufficient, and you never need to think about shaders directly.

When you need a visual effect that StandardMaterial does not support — a custom toon shader, a dissolve effect, a holographic display, a water surface with reflections — you write a custom material by implementing the Material trait and providing a shader written in WGSL (WebGPU Shading Language). WGSL is the shader language used by Bevy's rendering backend, wgpu, which implements the WebGPU standard. It is a modern shader language, less cryptic than its predecessors GLSL (OpenGL Shading Language) and HLSL (High-Level Shading Language), with syntax influenced by Rust and designed for safety and portability across GPU vendors and platforms.

Since Bevy 0.17, data-driven materials allow material properties to be modified at runtime through plain data rather than requiring different material types for different configurations. This means you can change a material's behavior (switching between opaque and transparent rendering, enabling or disabling features) by modifying component data rather than swapping to a different material type, which simplifies the common case where materials need to respond to game state.

Bevy 0.18 introduced fullscreen materials, which simplify the creation of post-processing effects — shaders that operate on the entire rendered image rather than on individual objects. Common post-processing effects include bloom (making bright areas glow), color grading (adjusting the overall color palette), vignette (darkening the edges of the screen), and depth of field (blurring objects that are not at the focal distance). Before fullscreen materials, implementing these effects required understanding Bevy's render graph and creating custom render passes, which was complex and poorly documented. Fullscreen materials reduce this to implementing a trait and providing a shader, which is dramatically more accessible.

The performance of shaders is a critical concern because they run for every vertex and every pixel every frame. A fragment shader that takes one microsecond per pixel does not sound slow, but at a resolution of 1920 by 1080, that is over two million pixels per frame, and if some pixels are covered by multiple overlapping triangles (overdraw), the total shader invocations can be much higher. GPU optimization is a specialized skill, but the most important principles are straightforward: minimize the number of texture samples (each sample is a memory access, and memory is slow), avoid branches (conditional logic causes different GPU threads to diverge, reducing parallelism), and reduce overdraw (sort opaque objects front-to-back so that the depth test prevents shading pixels that are behind already-rendered surfaces).

The relationship between artistic intent and technical implementation in shader development is one of the most creative intersections in game development. A technical artist who can write shaders bridges the gap between what the art director envisions and what the GPU can produce, translating visual concepts like "the surface should look like beaten copper" or "the magic effect should shimmer like heat haze" into mathematical operations on vectors and textures. This is a skill that combines programming, mathematics, and visual artistry in equal measure, and it is one of the reasons why game development is often described as a uniquely interdisciplinary field. The shader is where art meets mathematics meets hardware, and the results, when done well, are among the most visually stunning artifacts that software can produce.
