# Shaders and Materials

There is a moment in every game developer's education when the rendering pipeline stops being a black box and starts being a sequence of programmable stages, each of which you can replace with your own code. That code, running not on the CPU but on the massively parallel graphics processing unit, is called a shader, and understanding what shaders are and how they work opens up a vast creative space that is otherwise inaccessible. Everything you see on screen in a modern game, every surface, every shadow, every particle, every post-processing effect, is the product of shaders executing millions of times per frame, once for every vertex and once for every pixel. The material system is the bridge between your game logic and these shaders, the interface through which you configure how each surface in your game should be rendered. In Bevy, this system has evolved significantly, and since version 0.17, the engine has moved toward a data-driven material pipeline that makes custom materials more accessible while maintaining the performance and flexibility that serious graphics programming demands.

To understand shaders, you need to understand the rendering pipeline they participate in, at least at a high level. When the game asks the GPU to draw a 3D object, the process follows a series of stages. The first programmable stage is the vertex shader. The GPU takes the mesh data, a list of vertices, each with a position, a normal, texture coordinates, and possibly other attributes, and runs the vertex shader once for every vertex. The vertex shader's primary job is to transform the vertex position from the object's local coordinate space into the final screen-space position where it will appear on the display. This transformation typically involves three matrix multiplications: the model matrix (which places the object in the world), the view matrix (which accounts for the camera's position and orientation), and the projection matrix (which maps 3D world coordinates to 2D screen coordinates, applying perspective foreshortening). The vertex shader can also pass along any per-vertex data that the fragment shader will need, such as the world-space position and normal (for lighting calculations), texture coordinates (for sampling textures), or custom data for special effects.

After the vertex shader has processed all vertices, the GPU assembles them into triangles (or other primitives) and performs rasterization, the process of determining which pixels on the screen are covered by each triangle. For each pixel covered by a triangle, the GPU generates a fragment, which is essentially a candidate pixel along with interpolated per-vertex data. The values that the vertex shader output for each vertex of the triangle are interpolated across the triangle's surface based on the fragment's position, so a fragment near vertex A gets values close to A's outputs, while a fragment near the midpoint of edge BC gets an average of B and C's outputs. This interpolation is hardware-accelerated and essentially free from a programming perspective; it just happens between the vertex and fragment stages.

The fragment shader runs once for every fragment (candidate pixel) and its job is to determine the final color of that pixel. This is where the bulk of visual complexity lives. The fragment shader might sample a texture using the interpolated texture coordinates, apply a lighting model based on the interpolated normal and the positions of lights in the scene, calculate shadows, apply fog, combine multiple texture layers, or implement any other visual effect you can imagine. The output of the fragment shader is a color (and optionally a depth value), which is written to the frame buffer and eventually displayed on screen.

## Writing Shaders in WebGPU Shading Language

Bevy uses WebGPU Shading Language, abbreviated WGSL, as its shader language. WGSL is a relatively new language designed specifically for the WebGPU graphics API, which Bevy targets through the wgpu library. If you have written shaders in GLSL (OpenGL Shading Language) or HLSL (High-Level Shading Language, used by DirectX), WGSL will feel familiar in its broad strokes but different in its syntax and some of its semantics. It uses Rust-like syntax for types and functions, which makes it feel natural in a Bevy project. Variables are declared with `var` or `let`, functions are declared with `fn`, and types include `f32`, `i32`, `u32`, `vec2<f32>`, `vec3<f32>`, `vec4<f32>`, `mat4x4<f32>`, and so on.

A minimal vertex shader in WGSL receives vertex attributes as input, performs the model-view-projection transformation, and outputs the transformed position along with any data the fragment shader will need. A minimal fragment shader receives the interpolated outputs from the vertex shader and returns a color. The connection between the two stages is defined by a struct that acts as the vertex shader's output and the fragment shader's input, with each field annotated with a `@location` attribute that tells the GPU how to match outputs to inputs. The built-in position output is annotated with `@builtin(position)` and has special meaning: it tells the rasterizer where the vertex is on screen.

The real power of shaders comes from the data they receive from the CPU side: textures, uniform buffers, and storage buffers that contain the parameters and data the shader needs to do its work. In WGSL, these are declared as global variables with `@group` and `@binding` attributes that specify which bind group and binding slot they correspond to. A bind group is a collection of resources (textures, samplers, uniform buffers) that are bound together and made available to the shader. Bevy's rendering pipeline sets up several bind groups automatically (for camera data, mesh transforms, and material properties), and when you create a custom material, you are essentially defining a new set of data that will be passed to your shader through a material-specific bind group.

Understanding how data flows from your Rust code to your shader code is essential for writing custom materials. The pipeline looks like this: you define a Rust struct that represents your material's properties (colors, float parameters, texture handles). You implement a trait that tells Bevy how to create a bind group from this struct, which textures it uses, and which shader it should use. Bevy's rendering pipeline then automatically creates the GPU resources, uploads the data, binds the resources before drawing each mesh that uses the material, and invokes your shader with access to all of that data. The shader reads the material data from its bind group and uses it to determine how to color each pixel.

## Materials: The Bridge Between Art and Code

A material, in the rendering sense, is the complete description of how a surface should be rendered. It encompasses the shader that will be used, the parameters that configure that shader (colors, roughness, metallic values, emission), and the textures that provide spatial variation in those parameters (a color texture, a normal map, a roughness map). When an artist says a surface is "shiny red metal," they are describing a material: the color is red, the metallic value is high, the roughness is low (which produces sharp reflections), and the shader is a physically based rendering shader that knows how to simulate light interaction with metals.

Bevy's built-in `StandardMaterial` provides a physically based rendering material with a wide range of properties: base color (and base color texture), metallic and roughness (with optional textures), normal maps for surface detail, emissive properties for surfaces that glow, occlusion maps for ambient shadow detail, and alpha blending modes for transparent or semi-transparent surfaces. For many games, `StandardMaterial` is sufficient for the majority of surfaces, and the art pipeline consists of creating textures that drive its parameters. A single surface might have four or five textures: an albedo (base color) map, a normal map, a metallic-roughness map (often packed into a single texture with metallic in one channel and roughness in another), an ambient occlusion map, and an emissive map. These textures are authored in tools like Substance Painter, Quixel Mixer, or Blender and exported alongside the mesh data in formats like glTF that Bevy can load directly.

But many games need visual effects that go beyond what a standard physically based material can express. Stylized rendering (cell shading, painterly effects, pixel art aesthetics), special effects (force fields, dissolve effects, holographic surfaces), and gameplay-driven visual changes (damage indicators, stealth cloaking, magical auras) all require custom materials with custom shaders. This is where Bevy's material system truly shines, because it provides a framework for creating custom materials that integrate fully with the engine's rendering pipeline without requiring you to rebuild that pipeline from scratch.

## Custom Materials and the Data-Driven Pipeline

Creating a custom material in Bevy involves defining a Rust type that holds the material's properties, implementing the `Material` trait (or `Material2d` for 2D rendering), and writing the WGSL shader that will use those properties to render the surface. The `Material` trait has several associated functions and types that tell Bevy how to set up the rendering pipeline for your material: which shader to use, how to create the bind group layout, what vertex and fragment entry points to call, and what rendering state to apply (blending mode, face culling, depth testing).

Since Bevy 0.17, the engine has moved toward a more data-driven approach to materials and rendering. The data-driven pipeline allows material properties to be defined and modified more flexibly, reducing the amount of boilerplate required for common patterns and enabling more dynamic material workflows. This evolution reflects a broader trend in game engine design toward separating the description of what should be rendered (the data) from the mechanism of how it is rendered (the pipeline code), making it easier to create and iterate on visual effects without deep changes to the rendering infrastructure.

A practical example helps illustrate the workflow. Suppose you want to create a material that makes an object pulse with a glow effect based on a game timer. You would define a struct with a `color` field and a `time` field, implement the `Material` trait to set up the bind group and reference your custom shader, and write a WGSL fragment shader that uses the time value to modulate the emission. On the Rust side, a system updates the material's time field every frame. On the GPU side, the shader reads the time value and calculates a pulsing intensity using a sine function, applying it to the emissive output. The result is a surface that smoothly pulses with light, driven by game logic and rendered by custom shader code, all integrated into Bevy's standard rendering pipeline so it interacts correctly with the rest of the scene's lighting and post-processing.

The rendering pipeline from the shader's perspective is a data flow. Vertex data flows in from the mesh, is transformed by the vertex shader, is interpolated by the rasterizer, and arrives at the fragment shader. Uniform data flows in from bind groups, providing the camera transform, the mesh transform, the material properties, and the lighting environment. Texture data flows in from texture bindings and is sampled by the fragment shader using texture coordinates or screen-space positions. The fragment shader combines all of this input to produce a final pixel color, which flows into the frame buffer and ultimately onto the screen. Understanding this data flow is the key to understanding how to write shaders that do what you want, because every shader is fundamentally a function that transforms inputs into a pixel color, and the art of shader programming is choosing the right mathematical transformations to produce the visual result you envision.

The gap between imagining a visual effect and implementing it as a shader can be daunting, but it narrows with practice. Start with simple modifications to existing shaders: change a color, add a texture scroll, modulate a value with time. Then progress to more complex effects: combine multiple textures, implement custom lighting models, add procedural patterns using noise functions. The shader development workflow benefits enormously from hot reloading, the ability to modify a shader file and see the results immediately without restarting the game. Bevy supports hot reloading of assets including shaders, which means you can edit your WGSL file in a text editor, save it, and see the updated rendering in the running game within seconds. This rapid feedback loop is essential for the iterative process of shader development, where you are constantly adjusting parameters and mathematical expressions to achieve the visual result you want.

Shaders and materials sit at the intersection of art and engineering, of mathematics and aesthetics. They are the programmable layer that determines what your game looks like, and mastering them gives you creative freedom that no amount of pre-built materials can match. The barrier to entry is real, there is a learning curve in understanding GPU execution, shader language syntax, and the rendering pipeline, but the reward is the ability to create any visual effect you can imagine, limited only by the computational budget of a single frame and the ingenuity of your approach.
