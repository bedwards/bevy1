# Testing Games: What Can Be Verified and What Must Be Experienced

The question of how to test a game is more subtle than it first appears, because games straddle an uncomfortable boundary between engineering and art, between systems that can be verified through automated checks and experiences that can only be evaluated by human judgment. A function that calculates damage based on attack power and defense can be unit tested with the same rigor as any business logic: given these inputs, expect this output. But the question of whether the damage feels right — whether the player perceives the numbers as fair, whether the visual and audio feedback of taking damage creates the intended emotional response — cannot be answered by any automated test, no matter how sophisticated. Effective game testing requires understanding which aspects of the game belong in each category and applying the right verification technique to each.

The foundation of automated game testing, as with all software testing, is the unit test. In Rust and Bevy, unit tests are conventionally placed in the same file as the code they test, inside a module annotated with `#[cfg(test)]`. For game logic that is expressed as pure functions — damage calculation, pathfinding algorithms, inventory management, quest state transitions — unit tests are straightforward and extremely valuable. You call the function with known inputs, assert that the output matches your expectations, and the test either passes or fails deterministically. These tests run in milliseconds, can be executed on every change as part of the development loop, and provide immediate confidence that the core logic works correctly.

Testing Bevy systems is slightly more involved because systems operate on the World — the complete database of entities and components — rather than on explicit inputs and outputs. To test a system, you create a minimal World, populate it with the entities and components that the system expects, run the system, and then assert that the world's state has changed in the expected way. Bevy provides the tools for this: you can create a World, spawn entities with specific components, construct a schedule containing only the system under test, run the schedule, and then query the world to verify the results. This pattern tests the system in isolation from the rest of the game, which is the defining characteristic of a unit test.

Integration tests verify that multiple systems work together correctly, and they are where the most valuable and most difficult game testing happens. A movement system that reads input and updates position is easy to test in isolation. A collision system that detects overlapping entities is easy to test in isolation. But do they work together? Does the movement system move an entity into a wall that the collision system then pushes it back from? Does the ordering of systems matter? Are there frame-ordering issues where one system's output from last frame is used by another system this frame, creating a one-frame delay that affects behavior? These questions can only be answered by integration tests that run multiple systems together on a shared world and verify the combined result.

Property-based testing is a technique that is particularly well-suited to game logic, though it is underused in the game development community. Instead of testing specific input-output pairs (given attack 10 and defense 5, expect damage 5), property-based testing specifies invariants that should hold for all inputs (damage should never be negative, damage should increase when attack increases, damage should be zero when defense exceeds attack by some threshold) and then generates thousands of random inputs to check those invariants. The proptest crate in Rust makes this straightforward: you define a strategy for generating random inputs and a property that should hold, and proptest automatically generates test cases, including edge cases and boundary values that you might not have thought to test manually. For game logic with many parameters and complex interactions (a crafting system, a skill tree, an economy), property-based testing can uncover bugs that specific-case unit tests miss because they happen to exercise only the typical paths.

Visual testing — verifying that the game looks correct — is the most challenging category to automate. The naive approach is screenshot comparison: render a frame under known conditions and compare the resulting image pixel-by-pixel against a reference image. This catches regressions (changes that unintentionally alter the visual output) but is brittle: any change to rendering parameters, window size, or GPU driver version can cause pixel differences that are visually imperceptible but fail the test. More robust approaches use perceptual comparison metrics that ignore tiny differences while flagging visually significant changes, or they test at a higher level of abstraction (verify that entity X is visible on screen, rather than verifying the exact color of every pixel). Bevy does not include built-in visual testing tools, but the community has developed approaches using headless rendering and image comparison libraries.

Playtesting, as discussed in the dedicated essay on that topic, is the form of testing that evaluates the experiential qualities of the game: fun, engagement, clarity, fairness, emotional impact. It is fundamentally manual and human, and it serves a different purpose than automated testing. Automated tests verify that the game works correctly. Playtesting verifies that the game works well. Both are necessary, and neither substitutes for the other.

Performance testing occupies a middle ground: it can be automated, but interpreting its results requires judgment. You can write a benchmark that spawns ten thousand entities, runs the game for one hundred frames, and measures the average and percentile frame times. If those times exceed a threshold, the benchmark fails. But what threshold? The answer depends on the target hardware, the expected scene complexity, and the game's performance goals, which are design decisions rather than engineering facts. Performance tests are most valuable as regression detectors: they do not tell you whether the game is fast enough (that requires profiling on target hardware under realistic conditions), but they tell you when the game has gotten slower, which is often the first sign of a performance bug.

The testing strategy for a Bevy game project should be layered, with each layer catching a different category of problem. At the base, unit tests for pure game logic functions catch mathematical and logical errors. Above that, system tests for individual Bevy systems catch component access and data transformation errors. Above that, integration tests for system combinations catch interaction and ordering errors. Above that, performance benchmarks catch regression. And at the top, playtesting catches design and experience errors. Each layer runs at a different frequency: unit tests on every change, system tests on every commit, integration tests on every pull request, performance benchmarks weekly or on release branches, and playtesting at major milestones. The entire automated portion of this pyramid runs in CI, providing a safety net that catches the mechanically verifiable errors and frees human attention for the judgment-dependent errors that only humans can evaluate.

The most important insight about testing games is that the untestable parts are not unimportant — they are the most important parts of all. Whether the game is fun, whether it is beautiful, whether it creates the emotions the designers intended: these are the properties that determine whether players love the game or forget it, and they cannot be verified by any automated system. The role of automated testing is not to evaluate these properties but to ensure that the mechanistic foundation upon which they rest is solid and correct, so that when the game fails to be fun, the problem is a design problem rather than a bug, and it can be addressed through design iteration rather than through debugging. A game with perfect test coverage and no playtesting will be correct but possibly lifeless. A game with thorough playtesting and no automated tests will be potentially fun but fragile. The ideal is both: automated verification of everything that can be automated, and human evaluation of everything that cannot.
