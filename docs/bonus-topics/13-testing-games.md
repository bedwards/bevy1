# Testing Games

Software testing is one of those practices that nearly everyone agrees is important and that nearly every game project does poorly. The reasons for this are not laziness or ignorance; they are structural. Games are uniquely difficult to test because so much of what matters about a game, whether it feels good to play, whether the visuals look correct, whether the difficulty curve is satisfying, whether the controls are responsive, is subjective, perceptual, and resistant to automation. You cannot write a unit test that asserts "this game is fun." You cannot write an integration test that verifies "the lighting looks correct in this scene." You cannot write a regression test that catches "the jump feels slightly floaty after the physics refactor." And yet games also contain enormous amounts of logic that is perfectly testable by conventional means: damage calculations, inventory management, pathfinding algorithms, state machine transitions, serialization round-trips, collision detection. The art of testing games lies in understanding which parts of your game fall into which category, and applying the right testing strategy to each.

Let us begin with the most tractable category: unit tests for pure game logic. A unit test isolates a small piece of functionality and verifies that it produces the correct output for a given input. In a game context, this means testing functions and systems that take some game state as input and produce a deterministic result. Damage calculation is a classic example: given an attacker with a certain strength, a weapon with a certain damage modifier, and a defender with a certain armor value, the damage function should return a specific number. This is trivially testable with a standard Rust unit test. You call the function with known inputs and assert that the output matches the expected value. You add test cases for edge conditions: zero armor, maximum armor, critical hits, elemental resistances, damage floors and caps. Each test runs in microseconds and catches regressions immediately.

The challenge with unit testing game logic is that many game calculations are not pure functions. They depend on global state, on the Entity Component System world, on resources, on the results of queries over multiple entities. Testing these calculations requires setting up enough of the ECS world to run the system under test, which is where Bevy-specific testing patterns become essential.

## Testing Entity Component System Systems in Bevy

Bevy's architecture is actually quite friendly to testing once you understand the pattern. A Bevy system is a function that takes system parameters (queries, resources, commands, events) as arguments, and the ECS world is the environment that provides those parameters. To test a system, you create a minimal `World`, populate it with the components and resources the system needs, run the system, and then assert that the world's state has changed in the expected way. This is an integration test in the traditional sense, because you are testing the system's interaction with the ECS, but it is narrowly scoped because you are only setting up the specific entities and components that the system under test cares about.

The pattern looks something like this: you create a new `World`, insert whatever resources the system needs (perhaps a `Time` resource, a game configuration resource, or a custom gameplay resource), spawn entities with the components the system queries for, construct a `Schedule` that contains only the system you want to test, run that schedule against the world, and then query the world to verify that the expected changes occurred. For example, to test a movement system that reads a `Velocity` component and updates a `Transform` component, you would spawn an entity with both a `Velocity` and a `Transform`, run the movement system, and assert that the `Transform` has changed by the expected amount given the velocity and the time step.

This pattern is powerful because it lets you test systems in isolation, without the complexity of the full game. You do not need to set up rendering, input, or audio. You do not need to load assets. You do not need to run the entire Bevy application. You are testing the logic of a single system with precisely controlled inputs, which makes the tests fast, deterministic, and easy to reason about. When a test fails, you know exactly which system broke and what the unexpected behavior was.

There are subtleties to be aware of. Systems that use `Commands` to spawn or despawn entities do not take effect immediately; the commands are buffered and applied when the schedule reaches a synchronization point. In a test, you may need to explicitly apply command buffers (by calling `world.flush()` or by running the schedule, which does this automatically) before asserting on the results. Systems that depend on events need the event to be sent before the system runs, which means either sending the event manually by writing to the `Events<T>` resource or running a system that generates the event as part of the test schedule. Systems that depend on change detection (queries that filter on `Changed<T>` or `Added<T>`) behave differently on the first run (where everything appears as "added") versus subsequent runs, so you may need to run the schedule multiple times to test change-detection-dependent behavior correctly.

A more complete testing approach for a complex game might involve what you could call a "test harness world," a reusable setup function that creates a `World` pre-populated with common resources and configurations that many tests need. Instead of each test setting up the world from scratch, it calls the harness function and then adds only the test-specific entities and resources. This reduces duplication and ensures that tests use a consistent baseline configuration. Some projects take this further and create a custom test application builder that sets up a minimal Bevy app with a subset of plugins (excluding rendering, audio, and windowing, which are not needed for logic tests and would add startup cost and platform dependencies).

## Property-Based Testing and Fuzzing Game Logic

Beyond conventional example-based unit tests, property-based testing offers a powerful approach for game logic. Instead of specifying individual inputs and expected outputs, property-based testing defines properties that should hold for all valid inputs, and then uses a test framework to generate thousands of random inputs and verify that the properties hold. In Rust, the `proptest` and `quickcheck` crates provide property-based testing frameworks.

For game logic, properties might include things like: "the player's health should never be negative after applying damage" (testing that your health floor works correctly), "serializing and deserializing a component should produce an identical component" (testing your save system's round-trip behavior), "applying the inverse of a movement should return the entity to its original position" (testing that your movement math is reversible), or "the pathfinding algorithm should never return a path that passes through a wall" (testing that your navigation respects obstacles). These properties are powerful because they test broad invariants rather than specific cases, and the random input generation often discovers edge cases that you would not think to test manually, inputs that combine in unexpected ways or push values to extreme ranges where overflow, underflow, or floating-point imprecision causes bugs.

Property-based testing is particularly valuable for testing mathematical utility functions (vector operations, matrix transforms, easing functions, noise generators), serialization and deserialization (any struct that implements Serialize and Deserialize should round-trip correctly), state machine transitions (the machine should never enter an invalid state, regardless of the sequence of inputs), and any algorithm that has well-defined invariants (a sorting function should produce sorted output, a collision detection algorithm should be symmetric, a hash function should produce the same output for the same input).

Fuzzing, a related technique, takes random input generation to its extreme by feeding completely random data to a function and watching for crashes, panics, or undefined behavior. In Rust, the `cargo-fuzz` tool integrates with LLVM's libFuzzer to perform coverage-guided fuzzing, where the fuzzer intelligently mutates inputs to explore new code paths. Fuzzing is most useful for testing code that processes untrusted input: asset loaders, network message parsers, save file deserializers, and level file parsers. If any of these can be crashed by malformed input, that is at minimum a denial-of-service vulnerability and at maximum a security exploit, and fuzzing is the most effective way to find such bugs before your players (or attackers) do.

## Visual Tests, Snapshot Testing, and Playtesting

Some aspects of a game can only be verified visually, but that does not mean they must be verified manually. Snapshot testing (also called golden-image testing or visual regression testing) provides a semi-automated approach: you render a known scene, capture the output image, and compare it pixel-by-pixel against a reference image that was previously verified to be correct. If the images match (within a configurable tolerance to account for minor rendering differences across hardware), the test passes. If they differ significantly, the test fails and presents the difference for human review.

Bevy supports headless rendering, which makes snapshot testing feasible without a physical display. You can set up a Bevy app with a known scene, render one or more frames to an off-screen buffer, capture the result as an image, and compare it against a reference. The reference images are stored in the repository alongside the test code, and when a rendering change intentionally alters the visual output, you update the reference images and commit the new ones. This approach catches unintentional visual regressions: a shader change that subtly alters lighting, a material parameter that gets reset to a default, a UI layout that shifts by a few pixels after a framework update.

The limitation of snapshot testing is that it is sensitive to rendering differences across GPU vendors and driver versions, which means a test that passes on one developer's machine might fail on another's. This can be mitigated by running snapshot tests in a standardized continuous integration environment with a known GPU (or a software renderer), by using generous comparison tolerances, or by only running snapshot tests on a specific platform and treating them as advisory rather than mandatory on others. Despite these limitations, snapshot testing provides a level of visual regression detection that no amount of unit testing can achieve, because it tests the actual rendered output rather than the intermediate data that produces it.

All of the testing techniques described so far, unit tests, integration tests, property-based tests, snapshot tests, are forms of automated testing. They verify that the game's code does what the programmer intended. But they cannot verify that what the programmer intended is actually fun, intuitive, balanced, or satisfying. That is the domain of playtesting, where real humans play the game and provide feedback on their experience, and it is an irreplaceable part of game development.

Playtesting comes in several forms. Internal playtesting, where the development team plays their own game, catches the most obvious issues: controls that are confusing, difficulty spikes that are frustrating, features that do not work as intended. The limitation of internal playtesting is that developers are too familiar with their own game to experience it as a new player would. They know the "correct" way to play, they know the mechanics because they built them, and they are blind to the assumptions that a new player would not share. External playtesting, where people outside the team play the game, is essential for catching these blind spots. Watching someone play your game for the first time and struggle with something you thought was obvious is humbling and invaluable.

The most useful playtesting is observational: you watch the player play (or record their screen and inputs) without intervening, and you note where they get confused, frustrated, stuck, or bored. You resist the urge to explain how things work, because if the game needs an explanation, that is a design problem, not a player problem. After the session, you can ask questions about their experience, but the observational data, what they actually did versus what they said they did, is usually more reliable because people are not always accurate in reporting their own behavior and thought processes.

Some aspects of playtesting can be instrumented, even if they cannot be fully automated. You can add analytics to your game that record how long players spend in each area, where they die most frequently, which items they use, which paths they take, and which features they ignore. This telemetry data, collected across many playtest sessions, reveals patterns that individual sessions might not: a particular enemy that kills 80 percent of players on their first encounter (probably too hard), a crafting system that fewer than 5 percent of players engage with (probably not well-introduced or not rewarding enough), a level with an area that no one ever discovers (probably poorly signposted). Analytics do not replace observation, but they provide a statistical complement that helps prioritize which issues to address.

The practical challenge of testing games is not any single technique but the overall discipline of incorporating testing into the development workflow. Game developers are often under intense time pressure, and writing tests feels like it slows down the immediate work of building features. The countervailing truth is that a game without tests becomes progressively harder to change as it grows, because every change risks breaking something else, and without tests, you only discover the breakage when someone happens to exercise the broken functionality during manual playtesting, which might be days or weeks after the change was made. Tests are an investment in future development speed, and the return on that investment grows as the project grows.

A pragmatic approach is to focus testing effort where it provides the most value. Pure game logic (damage formulas, economy calculations, stat systems, crafting recipes, AI behavior trees) is high-value, low-cost to test and benefits enormously from unit tests that catch regressions instantly. Serialization round-trips are high-value because a bug in saving or loading can destroy player progress, which is one of the worst user experiences possible. System integration tests that verify the interaction between a few key systems (the health system correctly triggers the death system when health reaches zero, the inventory system correctly applies item effects) catch the most dangerous class of bugs, where individual systems work correctly in isolation but produce incorrect behavior when combined.

On the other end of the spectrum, testing visual polish (particle effects, screen transitions, UI animations) with automated tests is low-value and high-cost, because the tolerance for "correct" is subjective and the setup is complex. These are best covered by playtesting and visual review. Gameplay feel (does the jump feel good? is the camera smooth enough?) is fundamentally untestable by automated means and should not even be attempted; this is exclusively the domain of human playtesting and iterative tuning.

Building testable systems from the start is much easier than retrofitting tests onto systems that were not designed for testability. In the ECS context, this means keeping systems focused (each system does one thing), avoiding hidden dependencies (systems should get their inputs from components and resources, not from global state or static variables), and designing components to be constructible without needing the full game environment. A system that reads from a `Velocity` component and writes to a `Transform` component is trivially testable. A system that reaches into a global singleton, queries five different component types, spawns entities with runtime-loaded assets, and triggers sound effects is much harder to test. The same design principles that make code testable, loose coupling, clear interfaces, single responsibility, also make code more maintainable and more understandable, which means that the investment in testability pays dividends beyond testing itself.

The goal is not 100 percent test coverage; that is neither practical nor useful for a game. The goal is a testing strategy that catches the bugs that matter most, that gives you confidence to refactor and extend your code, and that complements (rather than replaces) the irreplaceable value of human playtesting. Automated tests verify that the machine does what you told it to do. Playtesting verifies that what you told it to do is the right thing. Both are essential, and neither is sufficient on its own.
