# Common Entity Patterns in Bevy

When you first encounter an Entity Component System, the sheer openness of it can feel paralyzing. In object-oriented programming, you have classes and inheritance hierarchies that tell you exactly where to put your data and behavior. An Enemy class inherits from Character, which inherits from GameObject, and so on up the chain. The path is clear, even if it eventually leads you into a tangled thicket of deep inheritance and fragile base classes. In an Entity Component System like Bevy's, there is no such predetermined structure. An entity is just an identifier, a number really, and you attach whatever components you want to it. This freedom is the system's greatest strength, but it also means that newcomers must develop an intuition for how to organize their game objects, an intuition that only comes from seeing patterns repeated across many projects.

This essay explores the most common entity patterns that have emerged in the Bevy community and in Entity Component System design more broadly. These are not rigid rules handed down from on high but rather conventions that experienced developers have converged on because they solve recurring problems elegantly. Understanding these patterns will help you think in terms of data and composition rather than objects and inheritance, which is the fundamental mental shift that Entity Component Systems demand.

## Marker Components and the Art of Tagging

Perhaps the simplest and most powerful pattern in any Entity Component System is the marker component, sometimes called a tag component. A marker component is a struct that contains no data at all. It exists purely to label an entity, to say "this entity is a thing of this kind." In Rust and Bevy, it looks like this: you define a struct with no fields, derive the Component trait on it, and attach it to entities that should be identified by that label. A Player marker, an Enemy marker, a Projectile marker, a Pickup marker. These zero-sized types take up no memory per entity (Rust is clever enough to optimize them away), yet they give your systems a way to query for specific kinds of entities.

The beauty of marker components is that they are composable in ways that inheritance never could be. An entity can be both a Player and a Poisoned and a Grounded and an InCombat, all at the same time, simply by having all four marker components attached. You do not need to figure out which class in an inheritance hierarchy should own the "poisoned" state. You do not need to create a PoisonedGroundedInCombatPlayer subclass. You just attach the relevant markers, and your systems query for whatever combination they care about. A system that applies poison damage queries for entities with both Poisoned and Health components. A system that prevents jumping queries for entities with both Grounded and Player components. Each system sees exactly the slice of the world it needs, and the markers make that slicing possible.

This pattern extends naturally to what you might call "state markers," where you use the presence or absence of a component to represent a state transition. Rather than having an enum field on some Status component that says whether an entity is alive or dead, you might simply remove the Alive marker and add a Dead marker, or you might remove the entity entirely and spawn a new corpse entity in its place. The component itself becomes the state, and the act of adding or removing it becomes the transition. Systems that only care about living entities just query for the Alive marker and never even see the dead ones. This is a subtle but important distinction from the object-oriented approach, where you would typically check a boolean or enum field on every entity and skip the ones that do not match. In the Entity Component System world, the query itself filters, and the system never wastes time on irrelevant entities.

There is a related pattern worth mentioning here: the "newtype" marker, where you wrap a meaningful value in a dedicated component type to distinguish it from other uses of the same underlying data. You might have a Speed component that wraps a floating-point number, and a separate MaxSpeed component that also wraps a floating-point number. They are both just numbers, but by giving them distinct types, you allow systems to query for one or the other without ambiguity. Rust's type system does the heavy lifting here, ensuring at compile time that you cannot accidentally pass a Speed where a MaxSpeed is expected. This is one of those places where Rust's expressiveness and Bevy's design philosophy align beautifully: types are cheap, and using more of them makes your code safer and more readable.

## Bundles of Meaning and the Spawn Pattern

While marker components let you tag entities with simple labels, most game objects need more than labels. A player character needs a position, a velocity, a sprite, a health value, a collision shape, and probably half a dozen other pieces of data. Attaching all of these components one at a time every time you spawn a player would be tedious and error-prone. You would inevitably forget one, and the resulting bug would be subtle because the entity would mostly work but fail in some system that expected the missing component.

This is where component bundles come in. A bundle in Bevy is simply a Rust struct whose fields are all components (or other bundles), and which derives the Bundle trait. When you spawn an entity with a bundle, Bevy unpacks the struct and attaches each field as a separate component. The bundle itself does not exist as a component on the entity; it is purely a spawn-time convenience, a way of saying "this collection of components belongs together and should always be created as a unit." Think of it as a recipe for an entity archetype rather than a thing the entity carries around.

The pattern that emerges is what I call the "spawn pattern," and it looks roughly like this. You define a bundle struct for each major type of game object: PlayerBundle, EnemyBundle, ProjectileBundle, PickupBundle, TriggerZoneBundle, and so on. Each bundle contains all the components that type of entity needs to function correctly. Then you write spawn functions, either as standalone functions or as methods on the bundle type, that create an instance of the bundle with sensible defaults and spawn it into the world. A spawn_player function might take a position as input and return a PlayerBundle with the player's starting health, default speed, the correct sprite, and so on. A spawn_projectile function might take a position, a direction, and a damage value, and produce a ProjectileBundle with the right velocity, collision shape, and lifetime timer.

This pattern has several virtues. First, it centralizes the definition of each entity type, so when you need to change what components a player has, you change it in one place. Second, it makes spawning entities a one-liner in whatever system triggers the spawn, which keeps those systems focused on their own logic rather than on entity construction. Third, it documents your game's entity types in a way that is both human-readable and compiler-checked. If you add a new required component to PlayerBundle, the compiler will force you to provide a value for it everywhere a PlayerBundle is constructed.

Let me dwell for a moment on the concept of entity archetypes, because it illuminates how Bevy works under the hood and why bundles matter for performance. In Bevy's storage layer, entities are grouped by their archetype, which is the specific set of component types they have. All entities with exactly the same set of components live together in memory, packed tightly in arrays. When a system queries for entities with certain components, it only needs to iterate over the archetypes that include those components, skipping entire groups of entities that could not possibly match. This means that if all your players have exactly the same set of components (because they were all spawned from the same bundle), they all live in the same archetype, and systems that query for players get excellent cache performance because the data is laid out contiguously in memory. If, on the other hand, you spawn players in an ad hoc way, sometimes including a component and sometimes not, you fragment your entities across multiple archetypes, and performance suffers. Bundles help you maintain archetype consistency without even thinking about it.

The common game object archetypes are worth enumerating because they recur in nearly every game, regardless of genre. A player entity typically has transform components for position and orientation, a sprite or mesh for rendering, velocity and acceleration for movement, some representation of health or hit points, a collision shape, and marker components for systems to query against. An enemy entity is structurally similar but usually adds components for artificial intelligence state, patrol paths or target tracking, and perhaps a loot table. A projectile entity tends to be lightweight: a position, a velocity, a damage value, a lifetime timer (so it despawns after a fixed duration or distance), and a collision shape. A pickup entity might have a position, a sprite, a collision shape, and a component describing its effect (heal the player, grant a power-up, add ammunition). A trigger zone is an invisible entity with a position, a collision shape, and a marker indicating what should happen when another entity enters it: load a new level, play a cutscene, spawn a wave of enemies.

None of these archetypes are built into Bevy. They are patterns that you, the game developer, create by composing Bevy's built-in components with your own custom components. Bevy gives you the building blocks (Transform, Sprite, Mesh, Camera, and so on), and you assemble them into meaningful game objects through bundles and marker components. This is the composability that makes Entity Component Systems so flexible, and it is why experienced Bevy developers spend time thinking carefully about their component design before writing systems.

## Deferred Operations and the Command Queue

There is a subtle but critical aspect of entity management in Bevy that trips up nearly everyone the first time they encounter it: you cannot spawn or despawn entities, or add or remove components, inside a system that is currently iterating over the world. This makes sense if you think about it. If a system is walking through all entities with a Health component and you suddenly spawn a new entity with Health in the middle of that iteration, what should happen? Should the new entity be included in the current iteration? Should it be skipped? The answer is ambiguous and potentially dangerous, so Bevy simply forbids it during normal queries.

Instead, Bevy provides the Commands abstraction, which is a deferred operation queue. When a system needs to spawn an entity, despawn an entity, add a component, or remove a component, it writes that operation to the command queue, and the operation is executed later, between system runs, at a synchronization point. This means that when you write commands.spawn(PlayerBundle { ... }) inside a system, the player entity does not actually exist yet when the next line of your system runs. It will exist by the time the next system that queries for players runs, but not within the current system's execution.

This deferred execution model is essential for safety and parallelism. Because commands are just data being appended to a queue, multiple systems can write commands simultaneously without conflicting with each other or with systems that are reading the world. The commands are flushed in a deterministic order at well-defined points in the schedule, so the results are predictable even though the operations are deferred. Bevy guarantees that all commands from one system are applied before the next system begins, provided those systems have an ordering constraint between them. If they do not have an ordering constraint, they might run in parallel, and their commands might be interleaved in some unspecified order, but each individual system's commands will be applied atomically.

Understanding the command queue is important for writing correct entity management code. A common mistake is to spawn an entity and then immediately try to query for it or modify it within the same system. This will not work because the entity does not exist yet. Instead, you have two options. First, you can use the return value of commands.spawn(), which gives you an EntityCommands handle that lets you chain additional operations onto the same deferred spawn, like adding extra components or inserting child entities. This is the right approach when you need to customize an entity at spawn time. Second, you can accept that the entity will not be visible until a later system and design your systems accordingly, using an event or a marker component to signal that a new entity has been created and needs further processing. This second approach is more common in practice because it keeps your systems decoupled: the spawning system does not need to know about the processing system, and vice versa.

The command queue also handles despawning, which is the removal of an entity and all its components from the world. Despawning is likewise deferred, so when you write commands.entity(entity_id).despawn(), the entity continues to exist for the remainder of the current system's execution. Other systems that run later in the same frame will not see it. If you need to prevent a despawned entity from being processed by later systems within the same frame, you can add a marker component like PendingDespawn and have those later systems filter it out.

There is a more advanced pattern called "entity reservation" that some developers use when they need to know an entity's identifier before it actually exists. Bevy allows you to reserve an entity identifier from the world, which gives you a valid Entity value that you can store in components or resources, and then later spawn the actual entity with that reserved identifier. This is useful when you need to set up mutual references between entities, like a parent that knows its child's identifier and a child that knows its parent's identifier, before either entity is fully spawned. In practice, Bevy's built-in parent-child relationship system handles the most common cases, but entity reservation remains a useful escape hatch for complex scenarios.

## Putting It All Together

The patterns described in this essay, marker components for tagging, bundles for spawning, and commands for deferred operations, are not isolated techniques. They are interlocking pieces of a design philosophy that values composition over inheritance, data over behavior, and explicit structure over implicit convention. When you define a marker component, you are making a statement about how your game categorizes its entities. When you define a bundle, you are documenting the data layout of a game object type. When you use commands, you are respecting the parallel, data-driven execution model that makes Bevy fast and safe.

Let me paint a picture of how these patterns come together in a small but complete example. Imagine a top-down shooter. You have a PlayerBundle with transform, sprite, velocity, health, and a Player marker. You have an EnemyBundle with similar components plus an AiState component and an Enemy marker. You have a ProjectileBundle with transform, velocity, damage, a lifetime timer, and markers for which team fired it. A movement system queries for all entities with Transform and Velocity and updates their positions. A collision system queries for entities with Transform and CollisionShape and checks for overlaps. When a projectile overlaps an enemy, the collision system writes a command to deal damage and another command to despawn the projectile. A damage system runs next, reduces the health of damaged entities, and if health reaches zero, writes a command to despawn the entity and spawn an explosion effect in its place. A spawner system watches the enemy count and, when it drops below a threshold, writes commands to spawn new enemies at random positions.

Notice how cleanly these systems compose. The movement system does not know or care whether it is moving players, enemies, or projectiles; it just moves anything with a position and a velocity. The collision system does not know what should happen when things collide; it just detects overlaps and writes events or commands. Each system is small, focused, and testable in isolation. The entity patterns, the markers, the bundles, and the deferred commands, are the glue that holds this architecture together.

These patterns will carry you surprisingly far. They scale from tiny game jam projects to large commercial games because they are rooted in the fundamental strengths of the Entity Component System paradigm: composition, data locality, and separation of concerns. As you gain experience with Bevy, you will develop your own variations and refinements of these patterns, tailored to the specific needs of your games. But the core ideas, tag with markers, spawn with bundles, defer with commands, will remain the bedrock of your entity architecture.

The most important thing to internalize is that entities in Bevy are not objects in the object-oriented sense. They do not have methods. They do not have an identity beyond their numeric identifier. They are defined entirely by the components attached to them, and those components can change at any time. An entity that starts as a player can, in principle, have its Player marker removed and an Enemy marker added, transforming it from one type of game object to another without being destroyed and recreated. This dynamism is unusual if you come from an object-oriented background, but it is the source of remarkable flexibility. Game states that would require complex inheritance hierarchies or visitor patterns in an object-oriented language can be expressed with a few component additions and removals in an Entity Component System. Once you develop an intuition for this style of thinking, you may find it difficult to go back to the old way.
