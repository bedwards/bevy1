# Pathfinding

Imagine you are standing at the entrance to a hedge maze. You can see the first few turns, but the exit is somewhere deep within, hidden behind walls of greenery. How do you find your way out? You might try the right-hand rule, always keeping your right hand on the wall, and eventually you will reach the exit (assuming the maze is simply connected). You might try exploring systematically, marking dead ends as you go. You might climb a tree, get a bird's-eye view, and plan your route before taking a single step. Each of these strategies represents a different approach to a fundamental problem: given a starting point and a destination, how do you find a path between them?

In video games, this problem arises constantly. An enemy soldier needs to chase the player through a building with hallways and rooms. A delivery truck needs to drive from one side of a city to the other, obeying roads and avoiding obstacles. A real-time strategy unit needs to march from its barracks to the enemy base, navigating around cliffs, forests, and other units. A non-player character in a role-playing game needs to walk to the tavern, sit down at a table, and order a drink, all without walking through walls or furniture. The quality of pathfinding in a game has an enormous impact on how intelligent and believable its artificial intelligence agents appear. An enemy that walks into walls or takes absurd detours breaks immersion instantly, while an enemy that navigates smoothly and sensibly enhances the illusion that the game world is populated by thinking beings.

This essay explores pathfinding from its mathematical foundations through the algorithms that power it, arriving at the practical considerations that shape pathfinding systems in real games.

## Graphs and the Representation of Navigable Space

Before you can find a path, you need a representation of the space that can be navigated. The mathematical abstraction that underlies virtually all pathfinding is the graph: a collection of nodes (also called vertices) connected by edges. Each node represents a location in the game world, and each edge represents a direct connection between two locations, meaning an agent can travel from one to the other without obstruction. Edges can be weighted, meaning they have a cost associated with traversal, or unweighted, meaning all connections are considered equal. The pathfinding problem then becomes: find the lowest-cost sequence of edges from a starting node to a goal node.

The first question, then, is how to convert a game world into a graph. There are several common approaches, each with distinct tradeoffs in accuracy, memory usage, and computational cost.

The simplest approach is a grid-based graph, where the game world is divided into a regular grid (often the same grid used for tilemaps) and each walkable cell becomes a node. Edges connect each cell to its neighbors, typically the four cardinal directions or eight directions including diagonals. The cost of each edge might be uniform (one for cardinal, the square root of two for diagonal) or might vary based on terrain type (roads are cheap, swamps are expensive, walls are impassable). Grid-based graphs are easy to construct and understand, and they work well for games that already use a tile-based world. Their main drawback is resolution: the grid must be fine enough to capture the relevant obstacles, which can result in a very large number of nodes for big open worlds. A 1000-by-1000 grid has a million nodes, which is manageable for modern algorithms but not trivial.

A waypoint graph is a hand-placed or procedurally generated set of key positions in the world, connected by edges that represent clear lines of travel. Level designers might place waypoints at doorways, intersections, and strategic points, then connect them to form a navigation network. Waypoint graphs are compact (far fewer nodes than a grid) and give designers explicit control over where agents can go, but they require manual work to create and maintain, and they produce paths that can look unnatural because agents travel in straight lines between waypoints, making sharp turns at each one.

Navigation meshes, often abbreviated as navmeshes, represent navigable space as a set of convex polygons that cover the walkable surface. Agents can travel freely within any polygon and can cross from one polygon to an adjacent polygon through their shared edge. The graph for pathfinding is formed by the polygons (as nodes) and their adjacency relationships (as edges). Navmeshes are the industry standard for three-dimensional game pathfinding because they capture the navigable space precisely, handle complex geometry naturally, and produce smooth, natural-looking paths. The downside is that generating navmeshes automatically from complex three-dimensional geometry is a non-trivial problem (though well-solved by tools like Recast, which is the most widely used navmesh generation library in the game industry). For two-dimensional games, navmeshes are less common because grids work well, but they can still be useful for games with non-rectangular obstacles or irregularly shaped levels.

## Dijkstra, A-Star, and the Art of Searching

With a graph in hand, the next step is to search it for the shortest path. The two algorithms that dominate game pathfinding are Dijkstra's algorithm and A-star, and understanding both is essential because A-star is essentially Dijkstra's algorithm with a clever enhancement.

Dijkstra's algorithm, published by Edsger Dijkstra in 1959, finds the shortest path from a starting node to all other nodes in a weighted graph with non-negative edge weights. It works by maintaining a set of unvisited nodes and a tentative distance for each node (initially zero for the start node and infinity for all others). At each step, it selects the unvisited node with the smallest tentative distance, marks it as visited, and updates the tentative distances of its unvisited neighbors. If traveling through the current node to a neighbor produces a shorter path than the neighbor's current tentative distance, the tentative distance is updated. This process repeats until the goal node is visited (at which point we know the shortest path to it) or until all reachable nodes have been visited.

Dijkstra's algorithm is correct and complete (it always finds the shortest path if one exists), but it has a significant weakness for game pathfinding: it explores nodes in all directions equally. If the start is in the bottom-left corner of the map and the goal is in the top-right, Dijkstra's will spend time exploring nodes to the left and below the start, which are clearly in the wrong direction. It does not "know" where the goal is and cannot prefer nodes that are closer to it.

A-star (A*) fixes this by adding a heuristic: an estimate of the remaining distance from each node to the goal. When selecting which node to explore next, A-star considers not just the known cost from the start (as Dijkstra's does) but also the estimated cost to the goal. The total estimated cost is the sum of these two values, and A-star always explores the node with the lowest total estimated cost. This biases the search toward the goal, dramatically reducing the number of nodes explored in most cases.

The heuristic must satisfy a property called admissibility: it must never overestimate the true remaining cost. If the heuristic is admissible, A-star is guaranteed to find the shortest path. The most common heuristic for grid-based pathfinding is the Euclidean distance (the straight-line distance from the current node to the goal), which is admissible because no path can be shorter than a straight line. For grids that only allow cardinal movement (no diagonals), the Manhattan distance (the sum of the horizontal and vertical distances) is a tighter admissible heuristic that results in even fewer explored nodes.

The performance difference between Dijkstra's and A-star can be dramatic. On a large open grid with few obstacles, A-star might explore a narrow corridor of nodes between the start and goal, while Dijkstra's explores a growing circle that encompasses a much larger area. In the worst case (when the heuristic provides no useful information or the shortest path requires exploring most of the graph), A-star degenerates to Dijkstra's. But in the typical case for game worlds, where the heuristic is informative and the shortest path is roughly in the direction of the goal, A-star is far more efficient.

Both algorithms produce a path as a sequence of nodes, which the game then converts into movement instructions for the agent. On a grid, this is a sequence of tile coordinates. On a navmesh, this is a sequence of polygons, which is then refined into a smooth path through string-pulling (also called the funnel algorithm), which finds the shortest path within the polygon corridor by threading a "string" through the edges, pulling it taut to eliminate unnecessary turns.

## Beyond A-Star: Flow Fields, Hierarchical Methods, and Decision Making

A-star is the workhorse of game pathfinding, but it has limitations that become apparent in certain scenarios. When hundreds or thousands of agents all need to pathfind to the same destination (as in a real-time strategy game where you select a large army and right-click on the enemy base), running a separate A-star for each agent is expensive. Flow fields offer an elegant alternative.

A flow field is a precomputed data structure that, for a given destination, tells every cell in the grid which direction to move. You compute it by running a single Dijkstra-style expansion outward from the destination, calculating the cost to reach the destination from every cell, and then for each cell, determining which neighboring cell has the lowest cost and recording the direction toward it. The result is a vector field that acts like a current: any agent placed anywhere on the grid can simply read the flow direction at its current cell and move that way, and it will follow the shortest path to the destination without any per-agent pathfinding computation. When the destination changes, you recompute the flow field (which is essentially one Dijkstra expansion), and all agents automatically adjust their movement. Flow fields are used extensively in real-time strategy games and tower defense games where many units share a common goal.

Hierarchical pathfinding addresses the problem of large worlds where the graph is too big for A-star to search efficiently in real time. The idea is to create a hierarchy of increasingly coarse representations of the graph. At the finest level, you have the full grid. At the next level, you group grid cells into clusters or regions and create a graph where each region is a node, connected to adjacent regions. At an even coarser level, you might group regions into super-regions. To find a path, you first search the coarsest level to find the general route (which regions to pass through), then refine within each region at the finer level. This dramatically reduces the search space because the coarse-level graphs have far fewer nodes. The Hierarchical Path-Finding A-star (HPA*) algorithm and its variants are the most well-known implementations of this idea, and they are used in commercial games with large open worlds.

There is also the question of how artificial intelligence agents decide where to go in the first place, which is a level of decision-making above pathfinding itself. Pathfinding answers the question "how do I get from here to there," but the agent first needs to answer "where should I go?" This higher-level decision might come from a behavior tree, a finite state machine, a utility system, or a goal-oriented action planner. An enemy soldier's behavior tree might determine that the current goal is to flank the player, which translates into a destination behind the player's position, which is then fed into A-star to produce a path. The pathfinding system is a tool used by the decision-making system, not a replacement for it.

In practice, game pathfinding systems are full of pragmatic compromises and clever tricks. Paths are often computed asynchronously, spread over multiple frames, to avoid frame-rate spikes when many agents request paths simultaneously. Agents often use path smoothing to eliminate unnecessary zigzags that grid-based paths produce. Dynamic obstacles (other agents, moving platforms, destructible terrain) require either periodic path recalculation or local avoidance behaviors (like steering forces) that adjust the agent's movement in real time without recomputing the full path. Local avoidance and global pathfinding are complementary: the global path provides the general direction, and local avoidance handles the moment-to-moment dodging of nearby obstacles and other agents.

The interplay between all these systems, graph representation, search algorithms, hierarchical decomposition, flow fields, behavior trees, and local avoidance, creates the illusion of intelligent movement that players take for granted. When an enemy squad splits up to flank you, when townspeople walk around each other on a busy street, when a real-time strategy army funnels through a narrow bridge in an orderly fashion, pathfinding is working behind the scenes to make it all look natural. It is one of those areas of game development that, when done well, is completely invisible to the player, and when done poorly, is infuriatingly obvious.

The field continues to evolve. Machine learning approaches are beginning to augment traditional pathfinding, using neural networks to predict good heuristics or to learn navigation behaviors from data. Any-angle pathfinding algorithms like Theta-star produce smoother, more natural paths by allowing movement at any angle rather than restricting it to grid directions. And the increasing scale of game worlds, some spanning entire planets or procedurally generated galaxies, pushes researchers and developers to find ever more efficient ways to answer the deceptively simple question: how do I get from here to there?

For developers working in Bevy, pathfinding is typically implemented as a system that reads entity positions and navigation data, computes paths using one of the algorithms described here, and writes the resulting path as a component on the agent entity. Other systems then read that path component and move the agent along it. The Entity Component System architecture is well-suited to this because the pathfinding system is cleanly separated from the movement system, the behavior system, and the rendering system. Each does its job, operating on the components it cares about, and together they produce agents that navigate the world believably. Whether you implement pathfinding yourself, use a community crate, or wait for built-in navigation features in future Bevy versions, the fundamental concepts remain the same, because the mathematics of shortest paths and graph search have not changed since Dijkstra sat down with pencil and paper in 1959, and they remain as relevant and as beautiful as ever.
