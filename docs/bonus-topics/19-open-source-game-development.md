# Open Source Game Development

The history of game engines is, in large part, a history of access. In the early decades of the game industry, every studio built its own engine, or its own technology stack that was functionally equivalent to an engine, because there was nothing else available. The technology was proprietary, closely guarded, and deeply tied to the specific hardware it targeted. When id Software released the source code for the Doom engine in 1997, it was a landmark event not because the code was particularly portable or well-documented (it was written for a very specific era of hardware), but because it demonstrated that opening up engine technology could benefit the entire community without destroying the original creator's business. id had already moved on to newer engines; the old code was more valuable as a learning resource and a foundation for modding than as a trade secret.

Fast forward to today, and the game engine landscape has been transformed by openness, though not uniformly. Epic's Unreal Engine and Unity are available for free to small developers, with source access in Unreal's case, but they are not open source in the meaningful sense: they are proprietary tools offered under permissive licensing terms that can change at the licensor's discretion, as Unity demonstrated dramatically in 2023 with its retroactive pricing change that shook developer trust across the industry. Godot is genuinely open source under the MIT license and has built a passionate community. And then there is Bevy, which represents something newer and, in some ways, more radical: a modern, high-performance game engine built entirely in the open, from the first commit, using Rust, a language that barely existed when the previous generation of engines was being designed.

This essay explores the culture and economics of open source game engine development, how Bevy specifically is funded and governed, what it means to contribute to an open source engine, the complex relationship between open source engines and the commercial games built with them, and the practical implications of the licenses under which these engines are released.

## The Culture and Economics of Open Source Engines

Open source software development is, at its heart, an economic anomaly that should not work but does. People write code and give it away for free. Other people use that code to build commercial products. The original authors receive no direct compensation for the value they create. And yet, some of the most critical infrastructure in the world, the Linux kernel, the Apache web server, the OpenSSL cryptographic library, the PostgreSQL database, runs on open source software that was largely created by volunteers, with some corporate sponsorship layered on top.

Game engines are a particularly interesting case because they are extraordinarily complex pieces of software. A production-quality game engine encompasses rendering (both two-dimensional and three-dimensional), audio, physics, input handling, asset management, scripting or extensibility, user interface, networking, animation, and much more. Building all of this to a commercial standard requires thousands of hours of expert engineering. The fact that open source game engines exist at all is a testament to the power of distributed collaboration and the passion of the game development community.

The motivations of open source engine contributors are varied and worth understanding because they shape the culture of these projects. Some contributors are learning: they want to understand how engines work at a deep level, and contributing to one is the most effective way to learn. Some are building: they need an engine for their own game, and contributing improvements upstream means those improvements are maintained by the community rather than by the contributor alone. Some are idealists: they believe that foundational technology should be freely available and that the game industry benefits when barriers to entry are lowered. Some are professionals: they work at companies that use or depend on the engine, and their contributions are part of their day jobs. Some are seeking reputation: a track record of high-quality open source contributions is a powerful credential in the software industry. Most contributors are motivated by some combination of these factors, and the healthiest open source projects are those that create value for all of these groups simultaneously.

The economics of open source game engines are precarious. Unlike open source infrastructure software (where companies like Red Hat build billion-dollar businesses around support and services), game engines do not have an obvious service model. Game developers using an engine need the engine to work; they do not typically need ongoing consulting from the engine's creators. The Godot model relies on donations and grants from organizations like the Godot Foundation, supplemented by corporate sponsorships. The Bevy model is similar, relying on donations through platforms like GitHub Sponsors and grants, with the engine's creator, Cart (Carter Anderson), supported by the community to work on Bevy full-time. This funding model works, but it is fragile. It depends on the continued generosity of the community and on the engine remaining popular enough to attract donations. There is no recurring revenue stream that scales with the engine's adoption, which means that success (more users, more bug reports, more feature requests) creates demand that the funding model may struggle to meet.

This is an unsolved problem in open source economics more broadly, and it affects game engines acutely because of their complexity. A web framework might be maintained by a handful of people in their spare time, but a game engine that competes with commercially funded alternatives needs sustained, full-time engineering effort. The Bevy community is actively aware of this challenge and experiments with funding models, but there is no silver bullet. What sustains Bevy, ultimately, is the passion and commitment of its contributors, the quality of the engine they are building, and the growing number of developers who find it compelling enough to use and support.

## How Bevy Is Funded and Governed

Bevy's governance structure is relatively informal compared to some larger open source projects, reflecting its age and the size of its community. Carter Anderson created Bevy and remains its benevolent dictator for life (though the community tends to avoid that specific terminology), making final decisions on the engine's direction, architecture, and release cadence. In practice, Bevy's decision-making is collaborative. Major changes are proposed through Request For Comments (RFCs) documents that are discussed publicly by the community, and pull requests go through a review process involving multiple experienced contributors. The project has a growing set of maintainers who have commit access and take responsibility for specific areas of the engine (rendering, audio, the Entity Component System, and so on).

Funding comes primarily through GitHub Sponsors, where individuals and companies can make recurring donations to support Bevy's development. Carter Anderson receives the majority of this funding to support his full-time work on the engine. Some other contributors receive sponsorship for their work as well, though the amounts are generally smaller. The Bevy Foundation (or equivalent organizational structure) manages funds and helps coordinate the project's administrative needs. Corporate sponsors contribute both money and engineering time, particularly companies that are building commercial products with Bevy and want to ensure the engine's continued development.

The governance and funding of an open source project are important because they determine the project's long-term sustainability and its responsiveness to the community's needs. A project that depends on a single maintainer is vulnerable to burnout. A project without adequate funding will lose contributors to paying work. Bevy's community is aware of these risks and is gradually building the organizational infrastructure to mitigate them, but the project is still young enough that much depends on the energy and dedication of a relatively small core team.

## Contributing to Bevy

Contributing to Bevy is, for many developers, their first experience with large-scale open source collaboration, and it is worth describing what the process looks like because it is both more approachable and more rigorous than many people expect.

The first step is usually finding something to work on. Bevy's issue tracker on GitHub is the primary source of tasks, ranging from small bug fixes and documentation improvements to large architectural changes. Issues are labeled by difficulty, area, and priority, and there are labels specifically for issues that are good for newcomers. The Bevy community is notably welcoming to new contributors, which is a cultural choice that the project's leadership actively cultivates. Rudeness, gatekeeping, and dismissiveness are not tolerated, and experienced contributors are encouraged to mentor newcomers rather than simply reviewing their code and pointing out flaws.

The contribution process follows a standard open source workflow. You fork the repository, create a branch, make your changes, and open a pull request. Pull requests are reviewed by one or more maintainers, who check for correctness, code style, test coverage, documentation, and alignment with the project's architectural principles. Reviews can be thorough, sometimes requiring multiple rounds of revision, but they are conducted respectfully and with the goal of improving both the code and the contributor's understanding. Once a pull request is approved and passes continuous integration tests, it is merged into the main branch.

What makes contributing to a game engine uniquely rewarding is the breadth of the work. You might fix a rendering bug one week, improve the Entity Component System's performance the next, and write documentation the week after that. The engine touches so many domains, graphics, audio, physics, networking, user interface, tooling, that there is always something new to learn. And because Bevy is written in Rust, contributors deepen their understanding of one of the most innovative programming languages of the past decade, a skill that is valuable far beyond game development.

## Open Source Engines and Commercial Games

The relationship between open source engines and the commercial games built with them is complex and sometimes uncomfortable. An open source engine is created by a community of volunteers (and some funded contributors), and then a commercial studio uses that engine to ship a game that generates revenue. The studio benefits enormously from the engine, the game could not exist without it, but the engine's contributors do not share in the revenue. Is this fair?

The answer, within the framework of open source licensing, is unequivocally yes. Open source licenses are explicit grants of permission to use the software for any purpose, including commercial purposes, without obligation to share revenue. This is a feature, not a bug. The open source model works because it removes friction: anyone can use the software, modify it, and build on it without negotiating licenses or paying fees. This lowers barriers to entry, increases the number of users (which increases the number of potential contributors), and accelerates the software's improvement. A game studio that ships a commercial game on Bevy benefits the Bevy ecosystem even if it never contributes a line of code, because its game demonstrates the engine's capabilities, attracts new users and contributors, and validates the project's viability.

That said, studios that build commercial products on open source engines have strong pragmatic incentives to contribute back. Bugs they find and fix upstream do not need to be maintained in a private fork. Features they develop and contribute upstream are maintained by the community, reducing the studio's ongoing maintenance burden. Contributing builds goodwill with the community, which can translate into recruitment advantages (developers want to work at studios that contribute to the tools they love). And contributing ensures that the engine continues to develop in directions that serve the studio's needs, because the studio has a voice in the project's direction.

## License Implications: MIT and Apache 2.0

Bevy is dual-licensed under the MIT License and Apache License 2.0, which is the same licensing choice made by the Rust programming language and many prominent Rust libraries. Understanding these licenses is important for anyone building commercial products with Bevy.

The MIT License is one of the most permissive open source licenses in existence. It grants permission to use, copy, modify, merge, publish, distribute, sublicense, and sell copies of the software, with only two conditions: the copyright notice and permission notice must be included in all copies or substantial portions of the software, and the software is provided "as is" without warranty. That is the entire license. You can use MIT-licensed code in commercial, closed-source products. You can modify it without sharing your modifications. You can charge money for products built with it. You just have to include the license text somewhere in your distribution (typically in an acknowledgments file or about screen).

The Apache License 2.0 is similarly permissive but adds a few provisions that the MIT License lacks. Most notably, it includes an explicit patent grant: contributors to Apache-licensed software automatically grant users a license to any patents they hold that cover their contributions. This provides legal protection against patent claims that the MIT License does not address. The Apache License also includes a contribution clause that defines the terms under which contributions are accepted, and it requires that modified files carry a notice stating that they have been modified. These additional provisions make the Apache License slightly more complex but also more protective for both users and contributors.

The dual-licensing approach (MIT or Apache 2.0, at the user's choice) gives maximum flexibility. Users who prefer the simplicity of MIT can use the MIT License. Users who want the patent protection of Apache 2.0 can use that instead. Either way, the practical implications for game developers are the same: you can use Bevy to build commercial games without paying royalties, without sharing your game's source code, and without any licensing friction. The only obligation is to include the appropriate license notices in your distribution.

This licensing model contrasts with engines like Unreal, which uses a custom license that includes a royalty on revenue above a threshold, and Unity, whose license terms have changed over time and generated significant controversy. The permanence and simplicity of open source licenses is a genuine advantage: the terms under which you use Bevy today will not change tomorrow, because the license is a grant that cannot be revoked. Code that has been released under MIT or Apache 2.0 remains available under those terms forever, regardless of what happens to the project, the maintainers, or the community. This permanence provides a level of business certainty that proprietary licenses, which can be changed by the licensor at any time, fundamentally cannot match.

Open source game development is not a panacea. It comes with real challenges: uncertain funding, volunteer burnout, the difficulty of competing with commercially funded engines that can throw hundreds of engineers at problems. But it also comes with real advantages: transparency, community, permanence, and the extraordinary power of distributed collaboration. Bevy is still young, and its ultimate success is not guaranteed. But it represents a genuine and exciting experiment in building a world-class game engine in the open, and the principles it embodies, openness, accessibility, and community ownership of foundational technology, are principles worth believing in and supporting.
