# Save Systems

The ability to save your progress and return to it later is so fundamental to modern gaming that it is easy to forget how revolutionary it once was. Early arcade games had no save mechanism: you inserted a coin, played until you ran out of lives, and started over from the beginning. The first home console games were similar, though some allowed you to pause and leave the console running, which was a save system of sorts, reliant on the fragile hope that no one would bump the power switch. The invention of battery-backed save memory in cartridges (pioneered by The Legend of Zelda in 1986) and the later shift to memory cards and hard drives transformed what games could be. Once players could save and resume, games could be longer, deeper, and more complex. Role-playing games could span dozens of hours. Strategy games could play out over weeks. Open-world games could let players shape vast persistent worlds that accumulated hundreds of hours of meaningful change.

Behind this simple player-facing feature (press a button, your progress is saved) lies one of the more subtle engineering challenges in game development. What exactly do you save? How do you serialize the state of a living, breathing game world into bytes on a disk? How do you load those bytes back and reconstruct a world that feels identical to the one the player left? What happens when you update your game and the save format changes? How do you handle autosaving without disrupting gameplay? These questions have no single right answer; the right approach depends on the game, its complexity, and its design goals. But the principles and patterns are consistent enough to be worth studying.

## What to Save and What to Reconstruct

The most important decision in designing a save system is choosing what to persist and what to reconstruct from scratch. This distinction is the difference between a save system that is elegant and maintainable and one that is bloated, fragile, and a constant source of bugs.

Consider a simple platformer. The player has a position in the world, a current health value, a set of collected items, and has completed certain levels. The world has enemies at various positions, some of which the player has defeated. There are coins scattered around that the player may have collected. The save file could, in theory, capture the complete state of every entity in the world: every enemy's position and health, every coin's collected-or-not status, every particle effect in mid-flight, the exact sub-pixel position of the player's character. But most of this is unnecessary. If the save system records which level the player is on, their health, their collected items, and their position (or just the checkpoint they last reached), the game can reconstruct everything else from its level data. Enemies can be respawned at their default positions. Coins can be reset (or the save can record which ones were collected, if the game's design requires it). Particle effects can be regenerated. The key insight is that the save file only needs to capture the delta between the initial state of the world and the current state, plus any player-specific data that cannot be derived from anything else.

This principle, save the minimum necessary to reconstruct the player's experience, applies across all genres, though the specifics vary dramatically. A role-playing game might need to save the player's character stats, inventory, quest progress, relationship values with non-player characters, and the state of certain world flags (has the bridge been destroyed? is the king alive?). A strategy game might need to save the positions and states of all units, the state of the economy, diplomatic relationships, and the current turn number. A sandbox game like Minecraft essentially saves the entire world because the world is the content, players have modified every block, and nothing can be derived from a template.

The general principle is: anything that the game's initialization code can produce from static data (level layouts, default enemy placements, item databases) does not need to be saved. Anything that the player has changed, or that game systems have evolved in ways the player would notice, should be saved. The boundary between these two categories is a design decision, not a technical one. A game where enemies respawn when you reload might save less than a game where killed enemies stay dead. Both are valid designs, and the save system should reflect the design intent.

One particularly tricky category is dynamic state that is not directly visible to the player but affects gameplay. Random number generator seeds are a classic example: if your game's procedural generation uses a seeded random number generator, saving the seed and the number of times it has been advanced allows you to reconstruct the same random sequence without saving the actual generated content. Physics engine state is another example. You generally do not save the internal state of the physics simulation (contact points, solver iterations, warm-starting data), because physics engines are designed to reach a stable state quickly from approximate initial conditions. You save positions and velocities, and the physics engine figures out the rest in a frame or two. Timer values, animation states, and audio playback positions are similarly transient: you might save them for a seamless resume experience, or you might accept a brief discontinuity when loading and skip them.

## Serialization Formats and Bevy Scenes

Once you have decided what to save, you need to convert it into bytes. This process is called serialization (converting data structures to a byte stream) and deserialization (converting bytes back to data structures). The choice of serialization format affects the save file's size, readability, compatibility, and performance.

JavaScript Object Notation, or JSON, is the most common human-readable serialization format. JSON save files are text files that can be opened in any text editor, which makes them excellent for debugging. When a player reports a bug, having them send their save file (which you can read as plain text) is invaluable. JSON is verbose, which means save files are larger than they need to be, and parsing JSON is slower than parsing binary formats, but for most games the difference is irrelevant. A save file measured in kilobytes or even a few megabytes loads in milliseconds regardless of format.

Binary formats like MessagePack, Bincode (a Rust-native binary format), or Protocol Buffers trade human readability for compactness and speed. Binary save files are typically two to ten times smaller than equivalent JSON files and parse faster. For games with very large save states (open-world games, sandbox games, strategy games with thousands of units), binary formats can make a meaningful difference in save and load times. The downside is that binary files cannot be inspected with a text editor, making debugging harder, and they are more sensitive to format changes (a misaligned byte offset can corrupt the entire file, whereas a JSON file with a missing field usually fails gracefully).

Rust's serde library is the standard tool for serialization and deserialization, and it supports a wide range of formats through a plugin architecture. You derive the Serialize and Deserialize traits on your data structures, and then you can serialize them to JSON, MessagePack, Bincode, TOML, YAML, or any other format that has a serde implementation. This means you can start with JSON during development (for easy debugging) and switch to a binary format for release without changing your data structures, just the serialization call.

Bevy provides a scene system that can serve as the foundation for a save mechanism. A Bevy scene is a serialized representation of a collection of entities and their components. You can snapshot the current state of the world (or a subset of it) into a scene, serialize that scene to a file, and later deserialize and spawn it to recreate the saved state. This approach is appealing because it leverages Bevy's existing infrastructure rather than requiring you to build a parallel serialization system from scratch. You define which components should be saved (by ensuring they implement the necessary traits for reflection and serialization), create a scene containing the entities you want to save, and write the scene to disk.

The scene-based approach works well when your save state maps naturally to entities and components, which it often does in a Bevy game because your game state is already represented as entities and components. However, there are caveats. Not all components are easily serializable. Components that contain handles to assets (like textures, meshes, or audio files) store internal identifiers that are meaningless outside the current engine session. Instead of serializing the handle, you need to serialize the asset's path or identifier and re-load the asset on deserialization. Components that contain function pointers, closures, or other non-data types cannot be serialized at all and must be reconstructed programmatically. Resources (singleton data that is not attached to any entity) need separate handling because they are not part of the entity-component graph.

A practical save system in Bevy typically combines scene serialization for entity data with custom serialization for resources, configuration, and any data that does not fit neatly into the entity-component model. You might define a SaveData struct that contains the serialized scene alongside additional fields for player metadata, game settings, timestamp, and version information. This wrapper struct is what actually gets written to and read from disk.

## Versioning, Autosave, and Checkpoints

Save data versioning is one of those problems that is easy to ignore during development and painful to deal with after release. The issue is simple: your game evolves over time. You add new features, change data structures, fix bugs, and rebalance systems. Each of these changes can affect the save format. A save file created with version 1.0 of your game might be incompatible with version 1.1 if you added a new field to a component, changed a field's type, or restructured your entity hierarchy.

The simplest versioning strategy is to include a version number in the save file and refuse to load save files from incompatible versions. This is safe but hostile to players, who lose their progress every time the game updates. A better approach is to write migration code that can upgrade save files from older versions to the current version. When you load a save file, you check its version number and apply a chain of migration functions: version 1 to version 2 (add the new stamina field with a default value), version 2 to version 3 (rename the "mana" field to "magic_power"), and so on. Each migration function transforms the data from one version to the next, and the chain brings any save file up to the current version. This requires discipline: every time you change the save format, you must write a migration function and increment the version number. But the result is that players never lose their progress, which is a powerful demonstration of respect for their time.

For formats like JSON, migrations can often be done at the level of the raw data (inserting missing fields with default values, renaming keys) without needing to deserialize into the game's current data structures, which is useful when the old structures no longer exist in the codebase. For binary formats, migrations are harder because the data is not self-describing, and you may need to keep old deserialization code around to read old formats.

Autosave systems add another dimension of complexity. An autosave triggers automatically at certain intervals or events (entering a new area, completing a quest, after a boss fight) without the player's explicit input. The challenge is that the save operation must not disrupt gameplay. Serializing the game state takes time (potentially tens of milliseconds for a complex world), and writing to disk is an I/O operation that can stall if the disk is busy. The standard approach is to serialize the save data on the main thread (which requires a brief pause in the game logic, ideally imperceptible) and then write the serialized data to disk on a background thread. Some games display a small save icon during autosave to manage player expectations and to warn them not to quit while the save is in progress.

Checkpoints are a game design concept that intersects with save system engineering. A checkpoint is a predefined point in the game where progress is recorded, and if the player dies or quits, they resume from the last checkpoint rather than from wherever they happened to be. Checkpoint systems can be simpler than full save systems because you only need to capture state at specific, controlled moments rather than at arbitrary points during gameplay. The game designer can ensure that the world is in a "clean" state at each checkpoint (all enemies are in their starting positions, all triggers are reset), which reduces the amount of data that needs to be saved.

Many games use a hybrid approach: checkpoints for the primary progression system (you respawn at the last checkpoint when you die) combined with a full save for session persistence (you can quit and resume from where you left off). The checkpoint handles the game design concern of pacing and difficulty, while the full save handles the practical concern of letting players take a break.

The save system is one of those features that sits at the intersection of engineering and design, where technical decisions have direct implications for the player experience. A save system that is fast and seamless encourages players to experiment, knowing they can always reload. A save system that is slow or unreliable makes players anxious, afraid to try risky strategies because they might lose progress. A save system that respects version upgrades tells players that their time investment in the game is valued and protected. Getting the save system right is not glamorous work, it will never be featured in a trailer or win an award, but it is one of the most important things you can do for the players who invest their time in your game. And in a Bevy context, the engine's emphasis on data-oriented design, where game state is naturally organized into serializable components, gives you a strong foundation to build on. The Entity Component System's explicit representation of state as data, rather than hiding it inside objects and method calls, is exactly what a save system needs: a clear, inspectable, serializable description of the world that can be written to disk and brought back to life.
