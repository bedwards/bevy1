# Data-Oriented Design

In 2014, Mike Acton, then the engine lead at Insomniac Games (the studio behind Ratchet and Clank and Sunset Overdrive, and later Spider-Man), gave a talk at the CppCon conference titled "Data-Oriented Design and C++." The talk was, on its surface, about writing faster C++ code. But it was really about something much deeper: a fundamental challenge to the way most programmers think about software. Acton's central thesis was that software is a transformation of data, that the purpose of a program is to read input data, transform it, and produce output data, and that programmers who lose sight of this basic truth write software that is slower, more complex, and harder to maintain than it needs to be. Object-oriented programming, he argued, encourages programmers to think about abstractions, hierarchies, and encapsulation rather than about the data itself, the shape of the data, how much of it there is, where it lives in memory, and how it flows through the machine. The result is code that is elegant in the abstract but hostile to the hardware it runs on.

This talk became one of the most influential presentations in game development, not because its ideas were entirely new (the game industry had been moving toward data-oriented thinking for years before Acton articulated it so clearly), but because it crystallized a set of principles that had been floating around in oral tradition and tribal knowledge into a coherent philosophy. That philosophy, data-oriented design, is the intellectual foundation on which Entity Component Systems like Bevy's are built. Understanding it deeply will change how you think about software, not just game engines, but any performance-sensitive system.

## The Hardware Reality

To understand why data-oriented design matters, you need to understand the machine your code runs on, not at the level of programming language abstractions but at the level of actual hardware. And the single most important fact about modern hardware is this: the processor is fast, and memory is slow. Absurdly, catastrophically slow, relative to the processor.

A modern Central Processing Unit can execute billions of operations per second. It can add two numbers, compare two values, or shift a bit pattern in a fraction of a nanosecond. But when it needs to read a value from main memory (Random Access Memory, or RAM), the read takes on the order of one hundred nanoseconds. That does not sound like much, but relative to the CPU's operation speed, it is an eternity. In the time the CPU spends waiting for a single memory read, it could have executed hundreds of operations. This disparity, known as the memory wall, has been growing for decades as processor speeds have increased faster than memory speeds, and it is the dominant performance bottleneck in most software today. Your program is not slow because the CPU cannot compute fast enough. It is slow because the CPU spends most of its time waiting for data to arrive from memory.

To mitigate this bottleneck, modern CPUs use a hierarchy of caches: small, fast memory banks that sit between the processor and main memory. The Level 1 cache is the smallest and fastest, typically 32 or 64 kilobytes, with access times of a few nanoseconds. The Level 2 cache is larger and slightly slower, typically 256 kilobytes to a few megabytes. The Level 3 cache is larger still, typically several megabytes to dozens of megabytes, with access times in the tens of nanoseconds. When the CPU needs a piece of data, it checks the Level 1 cache first. If the data is there (a cache hit), it gets it quickly. If not, it checks Level 2, then Level 3, and finally main memory (a cache miss). Each level is roughly an order of magnitude slower than the previous one.

Here is the critical detail that makes this cache hierarchy relevant to software design: when the CPU loads data from memory into a cache, it does not load just the one value it needs. It loads an entire cache line, typically 64 bytes of contiguous memory. This means that if your data is laid out contiguously in memory, accessing the first element effectively pre-loads the next several elements into the cache for free. The second, third, fourth, and subsequent accesses are cache hits, blazingly fast, because the data was brought into the cache as a side effect of the first access. If, on the other hand, your data is scattered across memory (allocated in separate heap objects, spread across different arrays, linked by pointers that jump to random locations), then each access is likely to be a cache miss, requiring a slow trip to main memory. The layout of your data in memory, not the cleverness of your algorithms, is often the primary determinant of your program's performance.

This is the hardware reality that data-oriented design takes as its starting point. It is not an abstraction, not a theory, not a matter of opinion. It is physics: the speed of light limits how fast signals can travel across a chip, and the economics of chip fabrication make large, fast memory prohibitively expensive. These constraints are not going away. If anything, they are intensifying as processors become more parallel (with more cores, wider vector units, and deeper pipelines) while memory latency improves only incrementally.

## Structure of Arrays Versus Array of Structures

The most tangible expression of data-oriented design is a pattern called Structure of Arrays, usually contrasted with the more intuitive Array of Structures pattern. This distinction sounds like a minor implementation detail, but it has profound implications for performance, and it is the reason Entity Component Systems are structured the way they are.

In the Array of Structures pattern, you define a struct (or class, or object) that contains all the data for a single entity, and then you store an array of these structs. Imagine a game with particles, each having a position (x, y, z coordinates), a velocity (x, y, z components), a color, and a lifetime. In the Array of Structures pattern, you would define a Particle struct with all these fields and store an array of Particles. This is the natural, intuitive layout, the one that object-oriented programming leads you to. Each particle is a self-contained unit with all its data in one place.

Now imagine you have a system that updates particle positions based on their velocities. It iterates through the array of particles and, for each particle, reads the position and velocity, computes the new position, and writes it back. The problem is that each Particle struct contains not just the position and velocity (which this system needs) but also the color and lifetime (which it does not). When the CPU loads a cache line to read a particle's position, it also loads the color and lifetime, which are useless to this system. They waste cache space, displacing other particles' position data that could have been in the cache. If the Particle struct is large (many fields, some of them large), this waste becomes severe. The system ends up processing data at a fraction of the theoretical speed because the cache is full of data it does not need.

The Structure of Arrays pattern inverts the layout. Instead of an array of Particle structs, you have separate arrays for each field: an array of positions, an array of velocities, an array of colors, and an array of lifetimes. All the x-coordinates are contiguous in memory, all the y-coordinates are contiguous, all the velocities are contiguous, and so on. Now, when the position-update system iterates through particles, it reads from the position array and the velocity array, and every byte it loads into the cache is useful. There is no wasted cache space on color or lifetime data. The system processes data at near-theoretical speed because the cache is being used optimally.

This is not a small difference. For systems that process thousands or millions of entities (particles, game objects, spatial queries), the Structure of Arrays layout can be two to ten times faster than the Array of Structures layout, depending on the ratio of useful data to total data in each struct. The larger the struct and the fewer fields a given system accesses, the greater the benefit. This is why Entity Component Systems store components in separate arrays grouped by type (the Structure of Arrays pattern) rather than storing all of an entity's components together (the Array of Structures pattern). When a Bevy system queries for entities with Position and Velocity components, it iterates over tightly packed arrays of positions and velocities, getting excellent cache utilization, even if those entities also have Sprite, Health, and dozens of other components that the system does not care about.

Bevy takes this further with its archetype-based storage. Entities with the same set of component types are grouped together into an archetype, and within each archetype, components are stored in the Structure of Arrays layout. This means that a system iterating over all entities with a specific component combination traverses contiguous memory with optimal cache behavior. The archetype grouping also enables Bevy to skip entire groups of entities that could not possibly match a query, reducing the amount of data the system even needs to consider. It is a layered optimization: the archetype system reduces the search space, and the Structure of Arrays layout within each archetype makes the actual iteration fast.

## Thinking in Transformations

The Structure of Arrays pattern is a concrete technique, but data-oriented design as a philosophy goes beyond memory layout. Its deeper lesson is about how to think about software. Instead of thinking about objects with identity, state, and behavior (the noun-centric worldview of object-oriented programming), data-oriented design asks you to think about data flowing through transformations (a verb-centric worldview).

Consider the traditional object-oriented approach to a game update. You have a list of game objects. Each game object has an update method. You iterate through the list and call update on each object. Inside each update method, the object accesses its own state (position, velocity, health, artificial intelligence state, animation state, whatever), possibly reads global state or other objects' state, computes new values, and writes them back. The logic is scattered across dozens of update methods in dozens of classes, and the data access patterns are unpredictable because each object might read any other object's state at any time. The CPU's cache is constantly thrashing as it jumps from one object's memory to another's, and the compiler cannot optimize across method boundaries because each method is a separate compilation unit with unknown side effects.

Data-oriented design reorganizes this into a pipeline of focused transformations. One transformation reads all positions and velocities and writes new positions (movement). Another transformation reads all positions and collision shapes and writes collision events (collision detection). Another reads collision events and health values and writes new health values (damage application). Each transformation accesses a small, well-defined set of data, processes it in a tight loop, and produces well-defined output. The data flows from one transformation to the next in a clear, linear pipeline. The CPU's cache is happy because each transformation accesses contiguous arrays. The compiler is happy because each transformation is a simple loop with no hidden side effects. The programmer is happy (eventually) because each transformation is small, testable, and easy to reason about in isolation.

This is exactly how Bevy systems work. Each system is a transformation that reads certain components, processes them, and writes results (either by modifying components or by issuing commands to spawn, despawn, or modify entities). The Entity Component System's scheduler ensures that systems run in the correct order, respecting data dependencies, and can run systems in parallel when they access non-overlapping data. The entire architecture is designed to facilitate the kind of data-flow thinking that data-oriented design advocates.

There is a subtlety here that is worth appreciating: data-oriented design does not mean you cannot have abstractions. It means that your abstractions should be informed by the data they operate on rather than imposed independently of it. An Entity Component System is itself an abstraction, a very powerful one, but it is an abstraction that is shaped by hardware realities. Components are stored in cache-friendly arrays because that is what the hardware wants. Systems process components in batches because that is what the hardware is good at. Queries filter entities by component type because that allows the storage layer to provide contiguous data access. Every design decision serves the data, and the data serves the hardware.

Mike Acton put it memorably in his talk: "The purpose of all programs, and all parts of those programs, is to transform data from one form to another." This sounds reductive, but it is clarifying. When you are confused about how to structure a piece of code, ask yourself: what is the input data? What is the output data? What is the simplest transformation that produces the output from the input? Where does the data live in memory, and how can I access it efficiently? These questions cut through abstraction and complexity and lead you to solutions that are both fast and clean. They also lead you, quite naturally, to the Entity Component System pattern, which is why data-oriented design and the Entity Component System are so closely intertwined.

The tension between data-oriented design and object-oriented design is not absolute, and presenting them as warring camps misses the nuance. Many programs benefit from object-oriented organization at the large scale (modules, interfaces, encapsulation of complexity) and data-oriented organization at the small scale (cache-friendly data layout, batch processing, Structure of Arrays). Bevy itself demonstrates this hybrid sensibility beautifully: the plugin system uses trait-based interfaces that would be familiar to any object-oriented programmer, while the Entity Component System's storage and scheduling are thoroughly data-oriented. The art is in knowing which paradigm to apply at which level of granularity, and the general rule of thumb is that the closer you are to the hot loop, the code that runs thousands or millions of times per frame, the more data orientation matters.

Data-oriented design is not a silver bullet, and applying it dogmatically can lead to code that is fast but incomprehensible, where the meaning of the data is obscured by its layout optimizations. The goal is not to optimize every data access pattern to cache perfection but to be aware of how data layout affects performance, to use that awareness when making architectural decisions, and to let profiling guide optimization efforts to the places where they matter most. In a Bevy game, the Entity Component System already provides a data-oriented foundation. Your job is to work with that foundation rather than against it, by designing components and systems that align with the engine's storage model. Keep components small and focused. Keep systems narrowly scoped. Prefer iteration over random access. Use marker components for filtering. These are not arbitrary rules but natural consequences of the hardware reality that data-oriented design illuminates.

The legacy of Acton's talk and the data-oriented design movement is visible throughout the modern game industry, not just in Entity Component Systems but in job systems, data-driven rendering architectures, and the general shift toward thinking about software as data transformation rather than object manipulation. For anyone building games with Bevy, understanding this philosophy enriches your understanding of why the engine is designed the way it is and equips you to make better decisions when performance matters. The hardware is not going to change its fundamental nature any time soon. The memory wall is here to stay. The programmers who understand this, and who design their software accordingly, will continue to build the fastest, most responsive, most ambitious software in the world. Data-oriented design is how they do it.
