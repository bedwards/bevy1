# Data-Oriented Design: Thinking in Transformations

There is a way of thinking about software that has been quietly gaining ground in the game development world for over a decade, and it is fundamentally different from the object-oriented approach that most programmers learn first. It does not start with classes, interfaces, or inheritance hierarchies. It does not begin by asking "what objects exist in this system and what can they do?" Instead, it starts with a different question: "what data exists, how is it laid out in memory, and what transformations need to be applied to it?" This approach is called data-oriented design, and it is the philosophical foundation upon which Bevy's Entity Component System (ECS) is built. Understanding data-oriented design is not strictly necessary for using Bevy — you can write components, systems, and queries without thinking about cache lines and memory layouts — but understanding it transforms your relationship with the engine from "following patterns someone else designed" to "understanding why those patterns exist and when to deviate from them."

The central insight of data-oriented design is that software performance is dominated by memory access patterns, not by computational complexity. This is counterintuitive if you learned about performance through algorithm analysis, where the running time of an algorithm is characterized by the number of operations it performs as a function of input size — O(n), O(n log n), O(n²), and so on. Algorithm analysis treats all operations as equal, but on modern hardware, operations are profoundly unequal. An arithmetic operation (addition, multiplication) takes one or two clock cycles. A cache hit (reading data from the processor's fast local memory) takes four to ten cycles. A cache miss (reading data from main memory because it is not in the local cache) takes one hundred to three hundred cycles. This means that a program that performs twice as many operations but keeps all its data in cache will run faster than a program that performs fewer operations but scatters its data across memory, triggering frequent cache misses.

The hardware reason for this disparity is the memory hierarchy that all modern processors use. The CPU has several levels of cache memory, each larger and slower than the previous: L1 cache (tens of kilobytes, about four cycles of latency), L2 cache (hundreds of kilobytes, about ten cycles), L3 cache (megabytes, about forty cycles), and main memory (gigabytes, about two hundred cycles). When the CPU accesses a memory address, it loads not just the requested byte but an entire cache line (typically sixty-four bytes) into L1 cache, on the assumption that the program will likely access nearby memory soon. This assumption — called spatial locality — is what makes cache-friendly memory layouts fast: if your data is packed together in memory, loading one element brings its neighbors into cache for free, and iterating through the data sequentially hits cache almost every time. If your data is scattered, every element is a potential cache miss, and the CPU spends most of its time waiting for memory rather than computing.

Object-oriented programming, as it is traditionally practiced, tends to produce cache-unfriendly memory layouts. Each object is allocated separately on the heap, which means objects that are logically related (all the enemies in a game, all the projectiles) may be scattered across memory depending on when they were allocated, what other allocations happened in between, and how the heap allocator chose to place them. When a system iterates over all enemies to update their positions, each enemy object might be at a completely different memory address, causing a cache miss for each enemy. The data within each object is also mixed: the enemy's position, velocity, health, AI state, animation frame, and references to other objects are all packed together in a single allocation, even though the position update only needs the position and velocity fields. The other fields (health, AI state, animation) occupy space in the cache line but are not used, wasting precious cache capacity.

Data-oriented design inverts this layout. Instead of storing all fields of each entity together (an "array of structures" layout), it stores all values of each field together (a "structure of arrays" layout). All positions are packed in one array. All velocities are packed in another array. All health values are packed in a third array. When the movement system iterates over positions and velocities, it reads from two tightly packed arrays, maximizing cache utilization. The health, AI state, and animation data are not loaded into cache because they are stored elsewhere, leaving more cache capacity for the data that actually matters for this operation. This layout is what Bevy's archetype-based ECS storage provides: entities with the same set of components are grouped into archetypes, and each archetype stores each component type in a contiguous array.

The transformation model of data-oriented design encourages you to think of your program as a series of transformations applied to streams of data, rather than as a collection of objects sending messages to each other. A movement system is a transformation: it takes a stream of (position, velocity) pairs and produces updated positions. A damage system is a transformation: it takes a stream of (health, incoming_damage) pairs and produces updated health values and possibly death events. An AI system is a transformation: it takes a stream of (AI_state, perception_data) tuples and produces updated states and action commands. Each transformation is a simple function that operates on well-defined inputs and produces well-defined outputs, with no hidden state, no side effects on unrelated data, and no dependency on the order of individual elements in the stream. This is why Bevy can automatically parallelize systems: if two transformations operate on disjoint data, they can run simultaneously without coordination.

Mike Acton's influential 2014 talk, "Data-Oriented Design and C++," articulated many of these principles and challenged the game development community to rethink its default programming paradigms. Acton's core argument was not that object-oriented programming is inherently bad but that it leads developers to think about the wrong things: about class hierarchies and interfaces instead of about data layouts and access patterns, about abstraction for its own sake instead of abstraction that serves performance. He proposed a series of principles: the purpose of all programs is to transform data, the most important property of data is its layout in memory, and the most important characteristic of code is its data access pattern. These principles do not replace good software engineering practices — you still need clean code, clear interfaces, and maintainable architecture — but they reframe those practices in terms of data rather than objects.

The tension between data-oriented design and object-oriented design is not absolute. Many programs benefit from object-oriented organization at the large scale (modules, interfaces, encapsulation) and data-oriented organization at the small scale (cache-friendly data layout, batch processing, structure-of-arrays). Bevy itself demonstrates this hybrid: the plugin system is object-oriented (plugins register capabilities with the application through a trait interface), while the ECS is data-oriented (components are stored by type in contiguous arrays, systems process data in batches). The art is in knowing which paradigm to apply at which level of granularity, and the general principle is that the closer you are to the hot loop — the code that runs thousands of times per frame — the more data orientation matters.

For practical Bevy development, the implications of data-oriented thinking are concrete. Keep components small and focused: a component that holds only the data needed for a specific purpose (like Position, Velocity, or Health) participates in fewer archetypes and creates more cache-friendly iteration patterns than a component that holds unrelated data (like PlayerData with position, velocity, health, inventory, quest progress, and settings). Keep systems focused: a system that reads only the components it needs enables maximum parallelism with other systems and touches only the data it operates on. Prefer iteration over random access: iterating over a query processes entities in archetype order, which is sequential in memory, while looking up individual entities by ID requires indirection through the entity-to-archetype mapping, which is slower. Use marker components liberally for filtering: a system that queries `Query<&Transform, With<Enemy>>` iterates only over enemy transforms, not all transforms, and the `With` filter is applied at the archetype level with zero per-entity cost.

Data-oriented design is not a silver bullet, and applying it dogmatically can lead to code that is fast but incomprehensible. The goal is not to optimize every data access pattern to cache perfection but to be aware of how data layout affects performance, to use that awareness when making architectural decisions, and to let profiling guide optimization efforts to the places where they matter most. In a Bevy game, the ECS already provides a data-oriented foundation — your job is to work with that foundation rather than against it, by designing components and systems that align with the engine's storage model rather than fighting it with patterns borrowed from object-oriented traditions. The reward for this alignment is software that is fast, parallel, and maintainable, not because you wrote clever optimization code but because you organized your data in a way that lets the hardware do what it does best.
