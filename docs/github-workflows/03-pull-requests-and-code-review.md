# Pull Requests and Code Review

There is a moment in every developer's career when they stop thinking of code review as an obstacle and start thinking of it as a gift. It usually happens the first time a reviewer catches a bug that would have taken hours to debug in production, or the first time a colleague suggests a simpler approach that makes the code half as long and twice as clear. Code review is one of those practices that seems burdensome when you are new to it, tolerable once you get used to it, and indispensable once you have experienced what happens without it. Pull requests are the mechanism through which code review happens on GitHub, and they are far more than a bureaucratic checkpoint between writing code and shipping it. They are a communication tool, a quality gate, a teaching opportunity, and a historical record, all wrapped in a single interface that has become so central to modern software development that many teams cannot imagine working without it.

A pull request, at its most basic, is a request to merge changes from one branch into another. You create a branch, make your changes, push the branch to GitHub, and open a pull request that says, in effect, "here is what I did, here is why I did it, and I would like your feedback before we incorporate these changes into the main codebase." What happens next, the review process itself, is where the real value lies. Reviewers examine the changes line by line, leave comments where they have questions or suggestions, and ultimately render a verdict: approve, request changes, or comment without a verdict. The author responds to the feedback, makes additional changes if necessary, and the process continues until the reviewers are satisfied and the pull request is merged. This loop of propose, review, revise, and accept is the heartbeat of collaborative software development, and the quality of the final product depends directly on how well the team executes it.

## Writing Pull Requests That Invite Good Reviews

The quality of the review you receive is largely determined by the quality of the pull request you submit, and the most important factor in pull request quality is size. Small, focused pull requests are easier to review, faster to merge, less likely to introduce bugs, and more likely to receive thoughtful feedback than large, sprawling ones. This is not a matter of opinion; it is a consistent finding across decades of software engineering research and practice. A reviewer who is presented with a twenty-line change to a single file can examine every line carefully, think about edge cases, and provide substantive feedback, usually within a few minutes. A reviewer who is presented with a two-thousand-line change spanning forty files will inevitably skim, miss things, and approve with a vague sense of unease that they did not really understand what they were approving. The natural unit of a pull request is one logical change: one bug fix, one feature, one refactoring. If you find yourself writing a pull request description that says "this PR does three things," that is usually a sign that it should be three pull requests.

Keeping pull requests small requires a particular kind of discipline, because the path of least resistance is to keep working on your branch until everything you had in mind is done and then submit it all at once. The alternative, which takes practice but becomes natural with time, is to plan your work in small increments from the beginning. Before you start coding, think about how the feature or fix can be broken into independent pieces that each make sense on their own. Implement the first piece, submit it for review, and start on the next piece while the first is being reviewed. This approach, sometimes called stacked pull requests or incremental delivery, keeps the review cycle short and ensures that you are never more than a day or two of work away from having your code merged. It also reduces the risk of merge conflicts, because each pull request touches a smaller portion of the codebase and spends less time waiting to be merged.

The title and description of a pull request are your opportunity to help the reviewer understand what they are about to read, and they deserve more care than most developers give them. The title should be a concise summary of what the change does, specific enough to be meaningful when scanning a list of pull requests. "Fix rendering bug" is too vague; "Fix sprite flickering during diagonal camera movement" tells the reviewer exactly what to expect. Some teams adopt conventions for the title format, like prefixing it with a tag that indicates the type of change: [Fix], [Feature], [Refactor], [Docs], [Test]. These tags make it easy to scan a list of pull requests and quickly identify the ones that are relevant to you.

The description is where you provide the context that the diff alone cannot convey. A good description explains what the change does, why it is needed, how it works (at a high level, not line by line), and how it was tested. If the change is related to a GitHub Issue, the description should include a reference like "Fixes #42" or "Closes #123," which creates a link between the pull request and the issue and arranges for the issue to be closed automatically when the pull request is merged. If the change involves a design decision that might not be obvious from the code, the description is the place to explain the reasoning so that reviewers do not waste time questioning a choice that was made deliberately. If there are things the reviewer should pay particular attention to, or areas where you are uncertain and would especially appreciate feedback, calling those out in the description makes the review more efficient and more useful.

Pull request templates, defined as Markdown files in the repository's .github directory, can standardize the format of descriptions across the team and ensure that important information is not forgotten. A good template is lightweight enough that it does not feel like filling out a form, but structured enough that it prompts authors to include the information that reviewers need. A typical template might include sections for a summary of the change, the motivation or issue reference, a description of how it was tested, and a checklist of common quality items like "I have run the linter" and "I have updated the documentation." The template should be a starting point, not a straitjacket; authors should feel free to modify or extend it when the standard format does not fit their particular change.

Draft pull requests are an underused feature that can significantly improve the development workflow. A draft pull request is one that is explicitly marked as not ready for review, which signals to your teammates that you are sharing work in progress for early feedback rather than requesting a formal review. This is useful in several situations: when you want to validate your approach before investing time in polishing the details, when you want to show your progress on a large task without implying that you are done, or when you want to trigger Continuous Integration checks on your branch without cluttering anyone's review queue. Converting a draft to a regular pull request when you are ready for review is a single click, and the transition serves as a clear signal to reviewers that it is time to take a look.

## The Review Process and How to Do It Well

Reviewing code is a skill that is distinct from writing code, and like any skill, it improves with practice and intentionality. A good reviewer reads the code with three goals in mind: correctness (does the code do what it is supposed to do?), quality (is the code well-structured, readable, and maintainable?), and context (does the change make sense given the broader architecture of the project?). Checking for correctness means thinking about edge cases, error handling, concurrency issues, and potential regressions. Checking for quality means looking at naming, structure, duplication, and whether the code follows the project's conventions and idioms. Checking for context means considering whether the change fits well with the existing codebase, whether it introduces patterns that will be difficult to maintain or extend, and whether there are alternative approaches that might be simpler or more robust.

GitHub provides three types of review verdicts, and using them correctly is important for clear communication. An approval means "I am satisfied that this change is ready to merge." A request for changes means "I have identified issues that should be addressed before this can merge." A comment without a verdict means "I have observations or questions, but I am not blocking the merge." The distinction between these three levels matters because it tells the author how to respond. An approval requires no further action. A request for changes means the author needs to make revisions and re-request review. A comment means the author should consider the feedback but can use their judgment about whether to act on it. When reviewers use the wrong verdict, either blocking a merge over trivial stylistic preferences or approving despite having significant concerns, it erodes trust in the review process and makes it less effective.

The tone of review comments matters enormously, and this is one of the areas where teams most often struggle. Code review is inherently personal; someone is examining your work and pointing out its flaws, and it is very easy for that process to feel like criticism rather than collaboration. The most effective reviewers frame their comments as questions or suggestions rather than commands, explain the reasoning behind their feedback rather than just stating an opinion, and focus on the code rather than the coder. "Have you considered using a hash map here instead of a vector? The lookup pattern on line 47 would go from O(n) to O(1)" is a much more useful and much less confrontational comment than "This is wrong, use a hash map." Similarly, acknowledging what is good about a change, not as empty flattery but as genuine recognition, creates a positive dynamic that makes the entire review process more pleasant and more productive.

GitHub's suggestion feature, which lets reviewers propose specific code changes that the author can accept with a single click, is a wonderful tool for turning abstract feedback into concrete action. Rather than saying "this variable name is confusing, consider renaming it," a reviewer can write a suggestion that replaces the variable name inline, and the author can apply the change without even switching to their editor. This feature is particularly useful for small improvements, like fixing a typo, adjusting formatting, or simplifying an expression, where the cost of describing the change in words is higher than the cost of just making the change directly. For larger suggestions, it is usually better to describe the approach in a comment and let the author implement it, since the author has more context about how the change fits into the larger picture.

The speed of code review is a perennial source of tension in development teams. Authors want their pull requests reviewed quickly so they can move on to the next task; reviewers have their own work to do and cannot drop everything to review someone else's code on demand. A good guideline is to aim for an initial review within a few hours of the pull request being submitted, with a maximum turnaround of one business day. This does not mean the review needs to be complete in that time, just that the reviewer has taken a first look and provided initial feedback. Longer delays create a cascade of problems: the author context-switches away from the change and has to re-load it when feedback arrives, other changes land in the meantime and create merge conflicts, and the backlog of unreviewed pull requests grows, creating pressure to rush through reviews and reduce their thoroughness. Some teams address this by designating a "review rotation" where one team member is responsible for initial reviews each day, ensuring that no pull request sits unattended for long.

## Merge Strategies and Branch Protection

When a pull request has been approved and all checks have passed, the final step is merging it into the base branch. GitHub offers three merge strategies, and understanding the differences between them is important for maintaining a clean and useful commit history.

The first strategy is the merge commit, which is the default. This creates a new commit on the base branch that has two parents: the tip of the base branch and the tip of the feature branch. All of the individual commits from the feature branch are preserved in the history, and the merge commit itself serves as a marker that shows when and where the branch was integrated. This strategy preserves the most complete history and makes it easy to see which commits were part of which pull request, but it can result in a cluttered history if feature branches contain many small, incremental commits like "fix typo," "address review feedback," and "try another approach."

The second strategy is squash and merge, which combines all of the commits from the feature branch into a single commit on the base branch. This produces a very clean history where each merged pull request corresponds to exactly one commit, making it easy to scan the log and understand what changed at a high level. The downside is that the individual commits from the feature branch are lost, so if you need to understand the step-by-step evolution of a change, you have to look at the pull request rather than the commit history. Squash merging also has a practical complication: if you have a long-running feature branch that you squash and then continue to develop on, you will encounter merge conflicts because Git does not recognize the squashed commit as being related to the original commits.

The third strategy is rebase and merge, which replays the commits from the feature branch onto the tip of the base branch, creating new commits with new hashes but the same content as the originals. This produces a linear history without merge commits, which some people find easier to read and navigate. However, because the commits are recreated with new hashes, the original commits on the feature branch become orphaned, which can be confusing if anyone has references to them. Rebase and merge also requires that the feature branch be up to date with the base branch and free of conflicts, which means the author may need to rebase manually before merging.

There is no universally correct merge strategy; the right choice depends on the team's preferences and the project's needs. Many teams settle on squash and merge as their default because it keeps the main branch history clean and readable, with each commit representing one complete, reviewed change. Teams that value a detailed commit history, or that practice a discipline of writing meaningful, atomic commits on their feature branches, may prefer rebase and merge or even plain merge commits. The important thing is to choose a strategy, apply it consistently, and configure the repository settings to enforce it so that team members do not have to remember the convention each time they merge.

Branch protection rules are the mechanism that ties together all of the quality practices described above and ensures that they are actually followed. At a minimum, most teams configure their main branch to require pull requests (no direct pushes), require at least one approving review, and require that Continuous Integration checks pass before merging. These three rules together guarantee that every change to the main branch has been reviewed by at least one other person and has passed the automated test suite, which is a remarkably strong baseline for code quality. Additional rules can require that the branch be up to date with the base branch before merging (which prevents the "it worked on my branch" problem where two individually-correct changes conflict when combined), require that specific people or teams review changes to specific files (code owners), and require that all review conversations have been resolved.

GitHub has introduced repository rulesets as a more flexible alternative to traditional branch protection rules. Rulesets can apply to multiple branches at once using pattern matching, they can be layered so that multiple rulesets apply to the same branch, and they can be managed at the organization level to enforce consistent policies across many repositories. For a game development studio with dozens of repositories, the ability to define a baseline set of rules at the organization level and then add project-specific rules at the repository level is a significant improvement over configuring branch protection individually for each repository.

The combination of small pull requests, clear descriptions, thoughtful reviews, appropriate merge strategies, and enforced branch protection creates a development workflow that is both rigorous and sustainable. It is rigorous because every change is examined and tested before it reaches the main branch, which catches bugs, improves code quality, and spreads knowledge across the team. It is sustainable because each individual step is lightweight, taking minutes rather than hours, and because the automation handles the mechanical aspects like running tests and enforcing rules, leaving humans free to focus on the judgment calls that only humans can make. The teams that execute this workflow well ship faster, not slower, than teams that skip code review, because the time saved by catching bugs early and maintaining a healthy codebase far exceeds the time spent on the review process itself. This is the counterintuitive truth at the heart of code review: the practice that seems like it should slow you down is actually what lets you go fast.
