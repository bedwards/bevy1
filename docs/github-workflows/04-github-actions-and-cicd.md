# GitHub Actions and Continuous Integration and Continuous Delivery

Software that is not tested is software that is hoped to work, and hope is not a strategy. This is the fundamental insight behind Continuous Integration, the practice of automatically building and testing every proposed change to a codebase, and Continuous Delivery, the practice of keeping the software in a state where it can be deployed to production at any time. Together, these two practices form the backbone of modern software development, and GitHub Actions is the platform's native implementation of both. Actions is not merely a Continuous Integration service bolted onto GitHub; it is a general-purpose automation framework deeply integrated into the repository lifecycle, capable of responding to virtually any event that occurs in or around your code. Understanding how to design, build, and maintain Actions workflows is essential for any team that wants to move fast without breaking things, and it is particularly important for Rust game projects where compile times are long, platform targets are diverse, and the cost of shipping a bug can be measured in frustrated players rather than mere inconvenience.

A GitHub Actions workflow is defined in a YAML file stored in the .github/workflows directory of your repository. The file specifies when the workflow should run (its triggers), what it should do (its jobs and steps), and where it should run (the type of machine, or runner, that executes each job). A simple workflow might run on every push to the main branch and every pull request targeting main, execute a single job that checks out the code, installs dependencies, and runs the test suite, and report the results back to the pull request as a status check. A sophisticated workflow might do all of that plus compile for multiple platforms, run different categories of tests in parallel, build release binaries, upload them as downloadable artifacts, deploy a web build to a staging server, and notify the team in a chat channel. The same YAML-based configuration language handles both cases, and the progression from simple to sophisticated is a matter of adding jobs and steps rather than learning a new tool.

## The Building Blocks of a Workflow

Every workflow is built from three nested layers: the workflow itself, its jobs, and the steps within each job. The workflow is the outermost container. It has a name, a set of triggers that determine when it runs, and a collection of jobs that define what it does. Triggers, specified under the on key in the YAML file, can be repository events like push, pull_request, release, or issue_comment; scheduled events using cron syntax; manual triggers via the workflow_dispatch event; or calls from other workflows via the workflow_call event. The flexibility of the trigger system means you can automate responses to almost anything that happens in your repository, not just code changes but also issue creation, label assignment, discussion posts, and wiki edits.

Jobs are the units of parallelism in a workflow. Each job runs on its own runner, an isolated virtual machine or container, and by default all jobs in a workflow run in parallel. You can create dependencies between jobs using the needs keyword, which tells a job to wait for one or more other jobs to complete before it starts. This allows you to express workflows that have both parallel and sequential phases, like running linting and testing in parallel and then running a deployment step only if both succeed. Each job specifies the type of runner it should use via the runs-on keyword, with common choices including ubuntu-latest for Linux, macos-latest for macOS, and windows-latest for Windows. GitHub provides hosted runners for all three platforms, and you can also configure self-hosted runners if you need custom hardware, specific software, or simply want more control over the execution environment.

Steps are the atomic units of work within a job. Each step either runs a shell command using the run keyword or invokes an action using the uses keyword. Actions are reusable units of automation, published in GitHub repositories, that encapsulate common tasks like checking out code, setting up a programming language runtime, caching dependencies, or uploading artifacts. The GitHub Marketplace contains thousands of community-created actions, and you can also write your own in JavaScript, TypeScript, or as Docker containers. Steps run sequentially within a job, sharing a filesystem and environment variables, so the output of one step is available to the next. This sequential execution within jobs, combined with parallel execution between jobs, gives you fine-grained control over the structure and performance of your workflows.

Matrix builds are one of the most powerful features of GitHub Actions for projects that need to support multiple platforms or configurations. A matrix is a set of variables that the workflow iterates over, creating a separate job instance for each combination of variable values. For a Rust project, you might define a matrix with operating systems (Linux, macOS, Windows) and Rust toolchain versions (stable, beta, nightly), resulting in nine parallel jobs that test every combination. You can include or exclude specific combinations to handle special cases, like skipping nightly on Windows if it is known to be unreliable, or adding an extra variable for a particular combination that requires special flags. Matrix builds are the natural way to express "test this on everything" in a workflow, and they are far more maintainable than writing separate jobs for each platform and version.

## Designing a Continuous Integration Pipeline for a Rust Game Project

Rust projects have a distinctive set of Continuous Integration requirements that stem from the language's characteristics: long compile times, a powerful but slow type checker, a rich ecosystem of linting tools, and cross-compilation to diverse targets including WebAssembly for browser-based deployment. A well-designed Continuous Integration pipeline for a Rust game project built with an engine like Bevy should address all of these concerns while maintaining fast enough feedback that developers are not left waiting for minutes on end to find out whether their change is acceptable.

The first and most impactful optimization is caching. Rust's compilation model means that building dependencies from scratch on every run can easily take ten to fifteen minutes for a project with a significant dependency tree, which is common when using a full-featured game engine. The Swatinem/rust-cache action is the standard solution for this problem. It caches the compiled artifacts in the target directory and the downloaded crate registry, using a cache key derived from the Rust toolchain version, the operating system, the Cargo.lock file, and the cargo configuration. On subsequent runs, the cached artifacts are restored and only the changed crates need to be recompiled, reducing build times from minutes to seconds for incremental changes. The cache is smart enough to handle different jobs in the same workflow independently, so a job that runs clippy with different flags than the job that runs tests will maintain its own cache without interference.

Beyond the rust-cache action, there are additional caching strategies worth considering. The sccache tool, originally developed by Mozilla, wraps the Rust compiler and caches compilation results, which can be particularly effective when multiple jobs or workflows compile the same dependencies with the same flags. The mozilla-actions/sccache-action makes it easy to integrate sccache into a GitHub Actions workflow, using the GitHub Actions cache as its backend storage. For projects with very large dependency trees or very frequent builds, the time savings from sccache can be substantial, though it adds complexity and is not always necessary if the simpler rust-cache approach is already providing acceptable build times.

A typical Continuous Integration workflow for a Rust game project might include the following jobs, some running in parallel and some sequentially.

The formatting check is the fastest job and should run first, or at least in parallel with everything else, because there is no point in running a full build if the code does not meet the project's formatting standards. This job runs cargo fmt --check, which verifies that the code matches the output of rustfmt without actually modifying any files. If the check fails, the developer knows they need to run cargo fmt before their change can be merged. This job typically takes only a few seconds and requires no caching because it does not compile anything.

The linting job runs cargo clippy, Rust's official linting tool, which checks for a wide range of common mistakes, performance issues, and stylistic problems. Clippy is an invaluable tool for maintaining code quality, and running it in Continuous Integration ensures that no linting warnings slip through even if a developer forgets to run it locally. A common configuration is to pass the -D warnings flag, which promotes warnings to errors and causes the job to fail if any warnings are present. This strict approach prevents the gradual accumulation of warnings that, if left unchecked, eventually become so numerous that they obscure new and important warnings. The linting job does require compilation, so it benefits significantly from caching.

The test job runs cargo test, which compiles and executes the project's test suite. For game projects, this often includes unit tests for game logic, integration tests for systems and pipelines, and potentially doc tests that verify code examples in documentation. The test job should set RUSTFLAGS to include -D warnings so that compiler warnings are treated as errors, maintaining the same strict standard as the linting job. For projects that use features to enable optional functionality, the test job might need to run multiple times with different feature combinations to ensure comprehensive coverage.

For game projects that target multiple platforms, additional jobs can compile (and optionally test) the project on macOS and Windows, using a matrix build to keep the workflow definition concise. Testing on macOS and Windows is particularly important for game projects because platform-specific bugs in areas like windowing, input handling, audio, and graphics are common and can be difficult to reproduce if you only test on one platform. If the project targets WebAssembly for browser deployment, a job that compiles to the wasm32-unknown-unknown target verifies that the web build is not broken, even if running the full test suite in a browser environment is not practical.

The Bevy engine itself provides an excellent reference implementation for Continuous Integration in a large Rust project. Its workflow validates code changes across Linux, macOS, and Windows; tests against stable, beta, nightly, and minimum supported Rust version toolchains; verifies compilation for WebAssembly and no_std targets; and maintains a hierarchical caching strategy that builds and refreshes caches daily to ensure fast feedback on pull requests. The Bevy team has also developed a custom Continuous Integration tool at tools/ci/ that encapsulates common test and lint commands, providing a consistent interface for running checks locally and in the workflow. This pattern, where the workflow calls a project-specific script rather than embedding complex command sequences directly in the YAML, is a best practice for any project where the Continuous Integration commands are non-trivial, because it makes it easy to run the same checks locally and keeps the workflow file readable.

## Secrets, Artifacts, and Deployment

Continuous Integration is the first half of the pipeline; the second half is Continuous Delivery, the process of getting the built and tested software into the hands of users. GitHub Actions supports this through secrets management, artifact storage, and deployment workflows.

Secrets are sensitive values like API keys, deployment credentials, and signing certificates that your workflows need but that should never appear in your code or workflow files. GitHub provides an encrypted secret store at both the repository and organization level, and secrets stored there are available to workflows as environment variables or action inputs. Secrets are masked in workflow logs so they are not accidentally exposed, and they are not available to workflows triggered by pull requests from forks, which prevents a common attack vector where a malicious contributor opens a pull request that exfiltrates secrets. Best practice is to grant secrets the narrowest possible scope, using repository-level secrets for values that only one repository needs and organization-level secrets for values shared across multiple repositories, and to rotate them regularly.

Artifacts are files produced by a workflow that you want to preserve after the workflow completes. The most common use case is build artifacts, the compiled binaries, packages, or archives that represent a releasable version of your software. The actions/upload-artifact and actions/download-artifact actions let you save files from one job and retrieve them in another, or download them from the workflow run page after the workflow completes. For a game project, artifacts might include release builds for each platform, a WebAssembly bundle for web deployment, or an archive of profiling data from performance tests. Artifacts are retained for a configurable period (ninety days by default) and can be downloaded by anyone with access to the repository.

Deployment workflows tie together the build, test, and release phases into an end-to-end pipeline. A typical deployment workflow for a game project might be triggered by the creation of a new release tag, build release binaries for all target platforms, sign the binaries if required, upload them as release assets on the GitHub release page, deploy the web build to a hosting service, and notify the team that the release is complete. GitHub's environment feature lets you define deployment targets (like staging and production) with their own secrets and protection rules, including required reviewers who must approve a deployment before it proceeds. This gives you a controlled, auditable deployment process that is both automated and gated by human judgment at the appropriate points.

## Reusable Workflows and Organizational Patterns

As teams and organizations grow, the challenge of maintaining consistent Continuous Integration practices across many repositories becomes significant. If each repository has its own independently-maintained workflow files, they will inevitably drift apart over time as different developers make different changes, leading to inconsistencies in testing standards, build configurations, and deployment processes. Reusable workflows and composite actions are the mechanisms that GitHub provides for addressing this problem.

A reusable workflow is a workflow file that can be called from other workflows using the workflow_call trigger. The reusable workflow defines its inputs and secrets, and the calling workflow provides values for them. This lets you define a standard Continuous Integration pipeline once in a central repository and reference it from every project that should use it. When you need to update the pipeline, you change it in one place and all projects that reference it automatically get the update on their next run. This pattern is analogous to extracting a shared library in application code, and it provides the same benefits of reduced duplication, improved consistency, and easier maintenance.

Composite actions serve a similar purpose at a smaller granularity. A composite action is a sequence of steps bundled together into a single action that can be used in any workflow. Where a reusable workflow replaces an entire workflow, a composite action replaces a sequence of steps within a job. This is useful for standardizing common setup tasks, like installing a specific set of tools, configuring environment variables, and restoring caches, that need to happen in every job but are not complex enough to warrant a full reusable workflow.

For organizations, the combination of reusable workflows and composite actions enables a layered architecture where the organization defines the standards, shared workflows and actions provide the implementation of those standards, and individual repositories customize the details. The organization might require that all Rust projects run formatting checks, linting, and tests on every pull request. The shared workflow implements these requirements with appropriate caching and configuration. Each repository calls the shared workflow and adds any project-specific jobs, like building documentation or running game-specific integration tests, alongside it. This approach balances consistency with flexibility, ensuring that every project meets the organization's quality standards while allowing each project to address its unique needs.

## Security Considerations

The security of your Continuous Integration pipeline deserves careful attention because a compromised workflow can do significant damage. Workflows run with access to your repository's code, secrets, and the GITHUB_TOKEN, which grants permissions to interact with the GitHub API on behalf of the repository. If an attacker can manipulate what a workflow executes, they can exfiltrate secrets, modify code, create releases, or take other destructive actions.

The most important security practice is to pin actions to specific commit SHAs rather than branch names or tags. When you reference an action by branch or tag, like uses: actions/checkout@v4, you are trusting that the code at that reference will not change in a way that harms you. But branches and tags can be moved to point at different commits, which means a compromised upstream action could silently change its behavior without your workflow file changing at all. Pinning to a SHA, like uses: actions/checkout@abc123def456, ensures that the exact code you reviewed is the exact code that runs, regardless of what happens to the upstream repository after you pin it. GitHub's 2026 security roadmap has introduced a dependencies section in workflow YAML that extends this pinning to transitive dependencies, locking the entire dependency tree by SHA in a manner similar to Go's module system.

The GITHUB_TOKEN should be configured with the minimum permissions necessary for the workflow to function. By default, the token has broad permissions that most workflows do not need, so setting the default to read-only at the repository or organization level and then granting specific permissions in individual workflows is a significant security improvement. Each workflow file can specify its required permissions using the permissions key, and you should do so explicitly rather than relying on the default.

Pull requests from forks present a particular security challenge because the fork's code is untrusted. Workflows triggered by pull_request from a fork run with read-only permissions and do not have access to secrets, which is a sensible default that prevents malicious contributors from exfiltrating sensitive information. However, some workflows legitimately need to comment on pull requests or perform other actions that require write permissions, and handling this safely requires careful design, typically involving a two-workflow pattern where the first workflow runs the tests with restricted permissions and the second workflow, triggered by the completion of the first, performs the privileged actions.

Timeouts are another important safety measure. A workflow that hangs indefinitely, whether due to a bug, an infinite loop, or a network issue, consumes runner resources and can delay other workflows. Setting an explicit timeout-minutes on each job, with thirty minutes being a reasonable default for most Continuous Integration jobs, ensures that hung jobs are terminated automatically. For Rust projects in particular, where compilation can be slow but should not be unbounded, a timeout provides a safety net against pathological cases where a dependency update or configuration change causes compilation to take much longer than expected.

The investment in designing a robust, secure, well-optimized Continuous Integration pipeline pays for itself many times over across the life of a project. Every pull request that passes the pipeline is a change that has been automatically verified to compile, to pass its tests, to meet its linting standards, and to build for all target platforms, without any human having to remember to do those checks manually. Every bug that the pipeline catches before it reaches the main branch is a bug that does not have to be debugged in production, does not have to be hot-fixed under pressure, and does not frustrate a user who expected the software to work. The pipeline is not free, it takes time to build, time to maintain, and runner minutes to execute, but compared to the cost of the problems it prevents, it is one of the best investments a team can make.
