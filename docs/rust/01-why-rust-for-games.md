# Why Rust for Games

The choice of programming language for a game engine is one of those decisions that echoes through every line of code written on top of it, every debugging session spent tracking down a crash, every late night spent optimizing a frame that takes just a little too long to render. For most of the history of video games, the choice was effectively made for you: you wrote in C or C++, because those were the languages that gave you the control over memory and hardware that real-time performance demands. Other languages existed, of course, and many successful games have been written in C#, Java, Lua, and even Python, but these languages run on runtimes that sit between your code and the hardware, adding layers of abstraction that cost performance in ways that are sometimes acceptable and sometimes catastrophic. The question of why Bevy chose Rust, and why a growing number of game developers are making the same choice, is really a question about what has changed in the landscape of systems programming languages since C++ was designed in the early 1980s, and whether those changes matter enough to justify learning something new.

The core argument for Rust in game development rests on three pillars, and understanding each of them requires understanding the specific pain points that C and C++ impose on game developers. The first pillar is memory safety without garbage collection. The second is fearless concurrency. The third is a modern type system with powerful abstractions that cost nothing at runtime. Each of these addresses a category of bug that has plagued game developers for decades, and each does so in a way that no previous systems programming language has managed.

Memory safety is the most frequently cited advantage of Rust, and for good reason. In C and C++, the programmer is responsible for managing memory manually: allocating it when needed, freeing it when done, and ensuring that no code ever reads from memory that has been freed, writes to memory that belongs to something else, or forgets to free memory that is no longer needed. These are not exotic edge cases — they are the most common source of crashes, security vulnerabilities, and hard-to-reproduce bugs in C and C++ programs. Use-after-free bugs, where code accesses memory that has already been returned to the system, can cause crashes that happen only under specific timing conditions, making them fiendishly difficult to reproduce and debug. Buffer overflows, where code writes past the end of an allocated block, can silently corrupt unrelated data structures, causing symptoms that appear far from the actual bug. Memory leaks, where allocated memory is never freed, cause the game to consume more and more system resources until it slows to a crawl or crashes.

Rust eliminates all of these bugs through its ownership system, a set of compile-time rules that track who owns each piece of memory and when that memory should be freed. Every value in Rust has exactly one owner at any time. When the owner goes out of scope, the value is automatically freed. References to values must follow strict borrowing rules: you can have many simultaneous read-only references or exactly one mutable reference, but never both at the same time. These rules are enforced by the compiler, not by a runtime garbage collector, which means they impose zero overhead at execution time. The game runs as fast as equivalent C code, but entire categories of memory bugs simply cannot exist in safe Rust code. For game developers who have spent hours or days tracking down a use-after-free bug that only manifests when the frame rate drops below thirty on a specific hardware configuration, this guarantee is transformative.

The alternative approach to memory safety, used by languages like Java, C#, Go, and JavaScript, is garbage collection: the runtime periodically scans memory to find and free objects that are no longer referenced by any live code. This works — garbage-collected languages are memory-safe — but the periodic scanning introduces unpredictable pauses that are anathema to real-time applications. A garbage collection pause of even a few milliseconds can cause a visible stutter in a game running at sixty frames per second, because a single frame's budget is only sixteen milliseconds. Some garbage collectors are more predictable than others, and modern garbage collectors have gotten remarkably good at minimizing pause times, but "minimizing" is not "eliminating," and game developers operating at the edge of their frame budget cannot tolerate unpredictable pauses of any duration. Rust's ownership system provides memory safety with deterministic, zero-cost cleanup, threading the needle between the unsafety of C++ and the unpredictability of garbage collection.

The second pillar, fearless concurrency, addresses the reality that modern hardware is parallel. A gaming PC in 2026 typically has eight to sixteen CPU cores, a current-generation console has eight cores, and even a mobile phone has six or more. A game that runs on a single core leaves the vast majority of the hardware's capability unused, and as core counts continue to increase while single-core speeds plateau, the ability to distribute work across cores becomes less of an optimization and more of a necessity. But concurrent programming in C++ is notoriously dangerous. Data races — where two threads access the same memory simultaneously and at least one of them writes — cause undefined behavior, which in practice means intermittent crashes, corrupted state, and bugs that appear and disappear depending on timing, CPU load, and phase of the moon. Traditional approaches to preventing data races involve locks (mutexes, semaphores), but locks are difficult to use correctly, easy to use incorrectly, and introduce their own category of bugs: deadlocks, where two threads each wait for a lock the other holds, freezing the program permanently.

Rust's borrowing rules prevent data races at compile time. If the compiler can verify that two threads never hold mutable access to the same data simultaneously — which is the same property enforced for single-threaded code, just extended across threads — then data races are structurally impossible. This means that when a Rust program compiles, you know it is free of data races, without having written a single lock or synchronized block. Bevy exploits this guarantee extensively in its scheduler, which automatically runs systems in parallel when their data access patterns do not conflict. The scheduler does not need to insert runtime checks or locks because the type system already guarantees safety. This is what "fearless concurrency" means in practice: not that concurrent programming is easy (it remains conceptually challenging), but that an entire category of catastrophic concurrent bugs is eliminated by the language itself, freeing the developer to think about the design of their parallel systems rather than worrying about whether those systems will corrupt each other's data.

The third pillar is Rust's type system and its zero-cost abstractions, which together enable a style of programming where high-level, expressive code compiles to machine code that is just as fast as hand-written low-level code. The phrase "zero-cost abstraction" means that you pay no runtime penalty for using an abstraction compared to implementing the same behavior manually. Rust's generics, traits, iterators, and pattern matching all follow this principle: they make code more readable and more correct, and the compiler optimizes them down to the same machine instructions you would write if you were coding at the level of individual memory operations. This contrasts with C++, where many abstractions do have hidden costs (virtual function dispatch, exception handling overhead, implicit copies), and with higher-level languages, where abstraction and performance are in constant tension.

For game development specifically, zero-cost abstractions mean that you can write clean, modular code — small components, focused systems, well-named types — without worrying that the modularity is costing you frame time. In Bevy, a system that queries two components and iterates over them compiles to a tight loop over contiguous memory, regardless of how many layers of abstraction are involved in the query construction. The ergonomic Rust code and the optimal machine code are the same thing, which eliminates the traditional game developer's dilemma of choosing between code that is readable and code that is fast.

Beyond these three pillars, Rust offers practical advantages that matter for the day-to-day experience of game development. The Cargo build system and package manager is widely regarded as one of the best in any language ecosystem, handling dependency resolution, building, testing, and documentation generation with minimal configuration. The crates.io ecosystem provides thousands of community libraries for everything from linear algebra to networking to serialization, and the quality bar for Rust crates tends to be high because the language's strictness filters out many categories of bugs before publication. The documentation culture is strong, with most crates providing extensive doc comments that are compiled into browsable HTML documentation automatically. Error messages from the Rust compiler are famously helpful, often suggesting the exact fix for the problem they report, which makes the learning curve less steep than the language's reputation might suggest.

There are genuine costs to choosing Rust, and intellectual honesty requires acknowledging them. Compile times are longer than C for equivalently sized projects, particularly for large dependency trees like Bevy's, and while incremental compilation mitigates this for small changes, a full rebuild of a Bevy game can take several minutes. The learning curve is real: ownership, borrowing, and lifetimes are concepts that most programmers have never encountered before, and internalizing them takes weeks or months of practice. The ecosystem, while growing rapidly, is younger and smaller than the C++ game development ecosystem, which means that some libraries and tools are less mature, less documented, or simply nonexistent. And the language's strictness, while it prevents bugs, also prevents some patterns that experienced C++ developers rely on, requiring rethinking of familiar approaches.

Whether these costs are worth bearing depends on the specific project, the team's experience, and the time horizon. For a game jam project that needs to be finished in forty-eight hours, the learning curve cost is probably prohibitive. For a long-lived project where reliability, performance, and maintainability matter — which describes most commercial games and most engines — the upfront investment in learning Rust pays compound interest in reduced debugging time, fewer production crashes, and safer concurrent code. Bevy's existence and rapid growth as an engine built entirely in Rust is itself evidence that the game development community has increasingly concluded that the benefits outweigh the costs, and the trajectory of the Rust ecosystem suggests that the costs will continue to decrease as tooling matures, compile times improve, and the pool of experienced Rust developers grows.

The deeper argument for Rust in games is not about any single feature but about the alignment between the language's design philosophy and the needs of real-time interactive software. Games need to be fast, correct, concurrent, and maintainable. Rust's ownership model makes them correct and concurrent by construction. Rust's zero-cost abstractions make them fast without sacrificing readability. Rust's strong type system and the Cargo ecosystem make them maintainable as they grow. No language is perfect for every purpose, but the overlap between what Rust provides and what games require is unusually large, and it is this alignment — not any individual feature taken in isolation — that explains why Rust has become the language of choice for the next generation of game engines.
