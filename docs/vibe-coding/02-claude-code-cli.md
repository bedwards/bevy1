# Claude Code: The Command-Line Interface

Claude Code is Anthropic's agentic coding tool, a command-line program that reads your entire codebase, edits files, runs commands, and executes multi-step tasks through natural language conversation. Unlike editor extensions that offer inline completions as you type, Claude Code operates at the project level, reasoning across files and directories to accomplish work that might span dozens of changes across the codebase. You install it with a single command, `npm install -g @anthropic-ai/claude-code`, authenticate through your browser, navigate to your project directory, type `claude`, and begin describing what you want. Within seconds the agent is reading your code, forming a plan, and taking action. The tool runs on your local machine, in your terminal, with access to your filesystem and your shell, which means it can do anything you could do at a command prompt: run tests, start servers, inspect logs, install dependencies, manage git branches, and more. This architecture makes it extraordinarily capable but also demands a thoughtful approach to permissions and trust, a tension that runs through every aspect of its design.

The current default model powering Claude Code is Opus 4.6, available with a one-million-token context window on Max, Team, and Enterprise plans. That context window matters because it allows the agent to hold an entire medium-sized codebase in memory simultaneously, reasoning about relationships between modules, understanding import chains, and proposing changes that account for ripple effects across the project. When the context is smaller, the agent must make more frequent decisions about what to read and what to discard, which introduces a class of errors that arise not from reasoning failures but from incomplete information. The large context window does not eliminate the need for good project structure and clear conventions, but it removes an entire category of failure that plagued earlier generations of coding assistants.

## The Tool System

Every action Claude Code takes is composed of calls to a small set of built-in tools, and understanding those tools is the key to understanding what the agent can and cannot do, how it reasons, and why it sometimes behaves in ways that seem surprising.

The Read tool takes a file path and returns the contents with line numbers. It handles text files of any size, paginating automatically when a file is too large to fit in a single response. It can also render images visually, extract text from PDF documents with page-range selection, and display Jupyter notebooks with all their cells and outputs. When you ask Claude Code to understand your project, the first thing it does is read entry-point files like README, package.json, or Cargo.toml, and then fans out to read whatever the task requires. The tool is read-only and safe, which is why it never requires permission approval.

The Write tool creates or completely overwrites a file. It is used when generating new files from scratch, such as a new module, a configuration file, or a test suite. Because it replaces the entire file, the agent must have read the file first if it exists, ensuring it does not accidentally destroy content it did not know about. The Edit tool is the more surgical counterpart: it performs exact string replacements within an existing file, sending only the diff rather than the entire file contents. The agent reads the file, identifies the exact text to replace, and specifies the replacement, and the tool applies the change only if the old text matches uniquely. This design prevents a large class of accidental edits and ensures that changes are precise and reviewable. In practice, the vast majority of code modifications go through Edit rather than Write, because most tasks involve changing part of a file rather than rewriting it entirely.

The Bash tool runs a shell command and returns the output. Each command executes in a separate process, and while the working directory persists between calls, shell state like environment variables does not. This tool is what gives Claude Code its extraordinary reach: through Bash, the agent can run test suites, compile projects, start and stop servers, query databases, interact with APIs through curl, manage git operations, install packages, and perform any other operation available from the command line. The power of Bash is also the primary reason permissions exist, because a sufficiently creative shell command can do almost anything to your system, from deleting files to exfiltrating data to installing malware. Claude Code maintains an auto-approval allowlist of read-only commands like ls, git status, lsof, pgrep, and others that are safe to run without asking, but any command that could modify state requires explicit approval unless the user has configured a more permissive mode.

The Glob tool performs fast file pattern matching, accepting patterns like `**/*.js` or `src/**/*.ts` and returning matching file paths sorted by modification time. The Grep tool searches file contents using regular expressions, supporting full regex syntax, file-type filtering, and multiple output modes including matched lines, file paths only, and match counts. Together, Glob and Grep allow the agent to navigate unfamiliar codebases efficiently, finding relevant files and code passages without reading every file from top to bottom. When you ask Claude Code to find all the places where a particular function is called, or to locate every configuration file in the project, it composes Glob and Grep calls to narrow down the search space before reading the specific files it needs. This search-then-read pattern is fundamental to how the agent works and is one of the reasons it can operate effectively even in very large repositories.

These six tools, Read, Write, Edit, Bash, Grep, and Glob, form the complete set of primitive operations from which all of Claude Code's behavior is composed. Every task, no matter how complex, reduces to a sequence of reads, searches, edits, writes, and shell commands. The agent's intelligence lies not in the tools themselves, which are simple, but in the reasoning that determines which tool to call, with what arguments, in what order, and what to do with the result. This architecture, sometimes called the ReAct pattern for Reasoning and Acting, drives the agent through an iterative loop: it receives a task, reasons about the next step, acts by calling a tool, observes the result, updates its understanding, and repeats until the task is complete or it needs human input.

## Permission Modes and Trust

Claude Code supports several permission modes that control how aggressively the agent can act without asking for confirmation, and choosing the right mode is one of the most important decisions a user makes.

The default mode, normal mode, asks for confirmation before every sensitive operation. File edits, shell commands, and other state-modifying actions all require the user to press a key to approve. This is the safest mode and the right choice when you are unfamiliar with the agent's behavior, working on a sensitive codebase, or learning the tool for the first time. The friction of confirmation prompts is real, but it ensures you see and approve every change before it happens.

Accept edits mode bypasses permission prompts for file modifications but still asks for confirmation before running shell commands. This mode suits developers who trust the agent's code edits but want to maintain a checkpoint before any command that could have side effects beyond the filesystem.

Plan mode sits at the opposite extreme from autonomous execution: it prevents tool execution entirely. Claude Code can analyze your code, form plans, and explain its reasoning, but it cannot make changes. This mode is useful for architectural exploration, when you want to discuss approaches with the agent before committing to any of them.

The bypass permissions mode, invoked with the flag `--dangerously-skip-permissions`, is what the community sometimes calls "YOLO mode." It suppresses all confirmation prompts, allowing the agent to read, write, edit, and execute commands without any human approval. Anthropic designed this flag with a deliberately alarming name to signal that it should only be used in isolated environments like containers, virtual machines, or continuous integration pipelines where the agent cannot cause lasting damage to your real system. Using it on your primary development machine with access to your credentials, your ssh keys, and your production infrastructure is precisely as dangerous as the flag's name suggests.

Between these extremes sits what Anthropic calls auto mode, a newer addition that aims to be a safer alternative to full bypass. Auto mode uses a set of heuristics and rules to auto-approve operations it considers safe while still blocking those it considers risky, and users can customize the approval rules through their settings. The don't ask mode takes the inverse approach: it auto-denies all tool usage unless the tool is explicitly pre-approved through the permissions allow list, which means Claude will never prompt you for confirmation but will silently skip anything you have not whitelisted. This mode is useful for highly constrained automation scenarios where you want the agent to do exactly a defined set of things and nothing else.

## Hooks

Hooks are user-defined actions that execute automatically at specific points in Claude Code's lifecycle, and they represent one of the most powerful extension mechanisms the tool offers. If you have used git hooks, the concept is identical: you define what should happen at a particular event, and the system guarantees it will execute every time that event occurs.

The hook system supports four types of actions. Command hooks run a shell command. HTTP hooks send the event's JSON payload as a POST request to a URL endpoint. Prompt hooks send a prompt to a lightweight Claude model for single-turn evaluation. Agent hooks spawn a subagent that can use tools to verify conditions before returning a decision. Each hook configuration specifies an event, an optional regex matcher that narrows when the hook fires, and the action to take.

The most commonly used events are PreToolUse, which fires before Claude performs an action like writing a file or running a command, and PostToolUse, which fires after an action completes. A PreToolUse hook can block an operation entirely, making it suitable for guardrails like preventing writes to environment files, blocking commands that would modify production databases, or enforcing naming conventions. A PostToolUse hook is ideal for cleanup and formatting tasks: running a code formatter after every file edit, triggering a linter after every write, or updating a log. The Stop event fires whenever Claude finishes responding, and if your hook returns a blocking signal, Claude is forced to continue working. Combining this with a prompt hook where a small, fast model evaluates whether all tasks were completed and all tests were run creates a self-verifying agent that checks its own work before declaring victory.

The critical property of hooks, the thing that distinguishes them from instructions in a CLAUDE.md file or a system prompt, is that they are guaranteed to execute. An instruction in natural language is a suggestion the model can choose to follow or ignore. A hook is a deterministic mechanism that runs every time its trigger condition is met, making it suitable for security-critical and compliance-critical workflows where "usually" is not good enough.

## Model Context Protocol Servers

The Model Context Protocol, usually abbreviated MCP, is an open standard that Anthropic released in November 2024 to let AI models connect to external tools and data sources. Within a year it became the default way to extend what a large language model can do, and Claude Code has deep support for it. An MCP server acts as a bridge between Claude Code and an external system: a database, an API, a file system, a cloud service, a monitoring dashboard, or anything else that exposes functionality through the protocol. When you configure an MCP server in Claude Code, the tools it provides become available to the agent just like the built-in tools, appearing in the tool list and callable through the same reasoning loop.

Configuration supports multiple transport mechanisms including HTTP, stdio, and Server-Sent Events, and servers can be defined at the project level in `.claude/settings.json` or globally in `~/.claude/settings.json`. Popular MCP servers in the ecosystem include the GitHub server for interacting with pull requests, issues, and repositories; database servers for querying PostgreSQL, MySQL, or SQLite; and API documentation servers that give the agent access to your API specs so it can generate more accurate client code. Claude Code can also run as an MCP server itself, exposing its file editing and command execution tools to other AI systems through the protocol, which enables multi-agent architectures where one agent delegates specialized tasks to another.

## Slash Commands and Skills

Slash commands are shortcuts that begin with a forward slash and trigger predefined actions within a Claude Code session. Built-in commands include things like `/status`, which shows active settings and configuration sources, and `/voice`, which activates voice mode for spoken interaction. But the real power lies in custom commands, which are simply markdown files stored in a `.claude/commands` directory at the project level or in `~/.claude/commands` for global availability. Each file contains a natural-language prompt that Claude executes when you type the corresponding slash command, and you can include the `$ARGUMENTS` placeholder to pass dynamic input.

In recent versions, Anthropic merged the slash command system into a broader concept called skills, which are reusable capabilities that can be invoked by the user, triggered by events, or composed together. A skill can be a simple prompt file, like the old slash commands, but it can also include frontmatter that specifies configuration like tool permissions, isolation requirements, or scheduling intervals. The `/loop` skill, for instance, sets up a recurring task that fires at a specified interval, essentially turning Claude Code into a session-scoped cron scheduler. You might use `/loop 5m check if the staging server is responding and report the HTTP status code` to create a lightweight monitor that runs every five minutes as long as your session is active. The `/schedule` skill extends this further by creating persistent scheduled tasks that can survive session restarts.

## Settings and CLAUDE.md

Claude Code's behavior is governed by two complementary systems. Settings files, written in JSON and stored at `~/.claude/settings.json` for global configuration or `.claude/settings.json` for project-level configuration, control what the agent can do: which tools are allowed, which permission mode is active, which environment variables are set, which MCP servers are connected. CLAUDE.md files, written in markdown, control what the agent should know: project conventions, architectural decisions, coding standards, preferred libraries, testing strategies, deployment procedures, and any other context that helps the agent make better decisions.

CLAUDE.md files are hierarchical. You can place one at the project root, another in a subdirectory, and another at the global level in `~/.claude/CLAUDE.md`. The agent reads all of them at startup and prioritizes the most specific, most nested file when instructions conflict. This hierarchy lets you establish global conventions, like always using four-space indentation and always running tests before committing, while allowing individual projects or directories to override those defaults.

The CLAUDE.md file is arguably the single most important configuration surface in the entire tool. A well-written CLAUDE.md transforms Claude Code from a generic coding assistant into a context-aware collaborator that understands your project's architecture, follows your team's conventions, and avoids the mistakes that generic code generation tends to make. Without it, every session starts from zero context and the agent must rediscover your patterns by reading your code. With it, the agent begins every session already aligned with your expectations, which dramatically reduces the number of corrections needed and accelerates the feedback loop from prompt to acceptable output.

During a session, pressing the `#` key opens an instruction prompt that automatically updates your CLAUDE.md file, creating a self-maintaining system where improvements you discover during coding sessions become permanent enhancements to your workflow. This mechanism embodies a broader principle of the tool's design: Claude Code is not meant to be configured once and left alone, but rather tuned continuously as you learn what works and what does not. Every correction you make, every convention you clarify, every edge case you document becomes part of the institutional memory that makes future sessions more productive. The most effective Claude Code users treat their CLAUDE.md files the way experienced developers treat their shell configurations: as living documents that accumulate wisdom over time and pay dividends on every subsequent interaction.
