# Effective Prompting for Code Generation

The single most common mistake people make when working with an AI coding assistant is treating the prompt as a wish. They type something like "make a login page" or "add authentication" and expect the model to read their mind about the framework, the styling, the validation rules, the error handling, the session management, the redirect behavior, and every other decision that a real implementation requires. The model, faced with ambiguity, makes choices. Sometimes those choices happen to align with what the developer wanted, and the experience feels magical. More often the choices diverge in ways that require rework, and the developer concludes the tool is unreliable when in fact the tool was simply underinformed. Prompting well is not about learning tricks or memorizing templates; it is about understanding that a large language model is a function that maps input to output, and the quality of the output is bounded by the quality of the input. If you provide vague input, you get plausible but generic output. If you provide precise, context-rich input, you get output that fits your specific situation. This essay is about how to provide that precision without turning every prompt into a specification document, because the goal is not to write more but to write the right things.

The foundation of effective prompting is context, and by context I mean everything the model needs to know in order to make the same decisions you would make if you were writing the code yourself. Context includes the programming language and its version, the framework and its version, the coding conventions your team follows, the architectural patterns already established in the codebase, the data shapes involved, the API contracts being consumed or produced, the error handling strategy, the testing approach, and any non-functional requirements like performance constraints, security considerations, or accessibility standards. You do not need to enumerate all of these in every prompt, because a well-configured CLAUDE.md file and a readable codebase provide most of this context implicitly. The agent reads your existing code before making changes, and if your code is consistent and well-organized, the model will infer patterns from what it reads. But when context is missing or ambiguous, and especially when you are starting a new module or working in an area of the codebase the agent has not yet explored, explicitly stating your expectations saves an enormous amount of back-and-forth.

Consider the difference between "add a user profile endpoint" and "add a GET /api/users/:id endpoint that returns the user's display name, email, and avatar URL as JSON, using the existing UserRepository class to fetch data, following the same error handling pattern as the existing /api/posts endpoints, and returning a 404 with our standard error envelope if the user is not found." The first prompt requires the model to guess the URL structure, the HTTP method, the response shape, the data access layer, the error handling approach, and the response format. The second prompt eliminates all those guesses by pointing to existing patterns and specifying the decisions that matter. The second prompt is longer, but it is not harder to write, because the developer already knows all of these things and is simply saying them out loud rather than leaving them implicit. The time spent typing the extra sentences is repaid many times over by the elimination of review-and-revise cycles.

## Showing Rather Than Telling

One of the most powerful techniques in prompting is providing examples, because examples act as implicit constraints that guide the model toward the format, style, and behavior you expect without requiring you to articulate every rule explicitly. When you tell the model "follow our coding conventions," the model has to infer what those conventions are from whatever code it has read, and it may get some of them wrong. When you show the model a specific function that exemplifies your conventions and say "write the new function in the same style as this one," the model has a concrete reference that anchors its output. The difference is the difference between giving someone driving directions using only compass headings versus giving them a route they have already driven and saying "go the same way but turn left at the library instead of the school."

Examples are especially valuable for output format. If you want test files structured in a particular way, with a specific import order, a specific describe-block nesting, a specific assertion style, pasting an example test file and saying "generate tests for the PaymentService following this exact structure" produces dramatically better results than describing the structure in words. The model excels at pattern matching, and a concrete example provides a much richer pattern than an abstract description. This principle extends beyond code to any structured output: if you want a database migration in a specific format, show an existing migration. If you want a commit message in a specific style, show three recent commits. If you want an API response shaped in a particular way, paste the schema or an example response.

There is a subtlety here worth exploring, which is the distinction between asking the model to do something and describing the problem you are trying to solve. When you ask for a specific implementation, like "add a debounce wrapper around the search input handler," you are prescribing the solution and the model will execute it, even if a different approach would serve your actual goal better. When you describe the problem, like "users are complaining that the search results flicker as they type because the API is called on every keystroke, and we need to reduce the number of API calls without adding noticeable delay to the perceived responsiveness," you give the model the freedom to propose the best solution, which might be debouncing, or might be throttling, or might be a cancellation pattern, or might be a combination of approaches you had not considered. The model has read thousands of implementations of search-as-you-type patterns and may be aware of trade-offs you have not thought about. By describing the problem rather than prescribing the solution, you tap into that breadth of knowledge while retaining the ability to evaluate and choose among the options the model suggests.

This is not to say you should never be prescriptive. Sometimes you know exactly what you want and the fastest path is to say so directly. The skill is in recognizing when your certainty about the solution is warranted and when it might be premature. A useful heuristic: if you could write the code yourself in a few minutes but simply want the AI to save you the typing, be prescriptive. If the problem is complex enough that you would need to think about it for a while before deciding on an approach, describe the problem and let the model think with you.

## Specifying Constraints and Non-Functional Requirements

Generated code tends to be correct in the happy path and weak in the edges, because the model optimizes for the most likely interpretation of your prompt, and the most likely interpretation rarely includes the full set of constraints that production code must satisfy. If you do not mention error handling, the model may or may not include it, depending on the patterns it has seen in similar code. If you do not mention performance, the model will choose the most readable implementation, which may not be the most efficient. If you do not mention security, the model will not add input validation, rate limiting, or authorization checks unless the context strongly implies them. This is not a flaw in the model; it is a consequence of the fact that the model generates code that matches the specification it was given, and a specification that omits constraints produces code that omits the corresponding safeguards.

The most effective prompters develop a habit of stating their non-functional requirements alongside their functional ones. "This function will be called thousands of times per second, so avoid allocations in the hot path." "This endpoint is publicly accessible, so validate and sanitize all input parameters." "This component will be used in a screen reader context, so ensure proper ARIA attributes and keyboard navigation." "This data pipeline processes personally identifiable information, so log the operation but never log the data values." These constraints are not implementation details that you should leave to the model's judgment; they are requirements that the model cannot infer from the functional description alone, and omitting them is the single most reliable way to get code that works in testing and fails in production.

A related practice is stating what you do not want. Models have tendencies, patterns they have seen so frequently that they reproduce them even when the context does not call for them. If you are working in a codebase that deliberately avoids a popular library, say so: "Do not use lodash; use native array methods." If you have a preference for a particular control flow style, say so: "Use early returns for error cases rather than nested if-else blocks." If there is a pattern the model tends to generate that you find problematic, preempting it explicitly is more effective than correcting it after the fact, because each correction costs a round trip and each round trip costs context window space and human attention.

## Iterating on Output

Even with excellent prompting, the first output is rarely the final output, and expecting it to be sets you up for frustration. The most productive workflow treats the initial generation as a draft and the subsequent conversation as a collaborative revision process. This is where the interactive nature of a tool like Claude Code, as opposed to a one-shot generation tool, provides its greatest advantage: you can look at the output, identify what you like and what you do not, and provide targeted feedback that steers the next iteration. "The overall structure is good, but the error handling should use our custom AppError type instead of generic Error, and the database query should use a parameterized query instead of string interpolation" is the kind of feedback that produces a dramatically improved second draft, because it tells the model exactly what to change without requiring it to regenerate from scratch.

The art of iteration is knowing what level of feedback to give. At the broadest level, you can redirect the entire approach: "Actually, let's use a different architecture for this. Instead of polling the API on an interval, let's use WebSockets." At an intermediate level, you can adjust specific implementation decisions: "The caching logic is right, but use a Least Recently Used eviction policy instead of time-based expiration." At the finest level, you can correct individual lines: "On line 47, the comparison should be less-than-or-equal, not strictly less-than." Matching the granularity of your feedback to the granularity of the problem avoids wasting time on overly detailed instructions when the broad strokes need adjustment, and avoids losing precision when the broad strokes are right and only a detail needs fixing.

There is also value in asking the model to explain its choices before you decide whether to accept them. "Why did you use a recursive approach here instead of an iterative one?" or "What are the trade-offs of this caching strategy compared to the alternatives you considered?" These questions serve two purposes. First, they surface reasoning that might reveal a legitimate consideration you had not thought of, potentially changing your mind about what the right approach is. Second, they test the model's understanding: if it cannot articulate a coherent reason for its choice, that is evidence the choice was arbitrary rather than principled, and you should probably override it with your own judgment.

## The Compound Effect of Good Prompting Habits

The difference between a developer who prompts well and one who prompts poorly is not visible in any single interaction. In a single prompt-response cycle, the well-prompted version might save ten minutes of revision compared to the poorly-prompted version, a modest improvement. But across a full day of work, those ten-minute savings compound, because each successful generation builds on the previous one, each avoided misunderstanding preserves context window space for the work that matters, and each precise specification teaches the model more about your preferences and patterns, improving subsequent generations within the same session. Over a week, the compound effect is dramatic: the well-prompted developer ships features faster, with fewer bugs, with more consistent code, and with less fatigue, because they spend their cognitive energy on the hard problems, the architectural decisions, the edge cases, the user experience considerations, rather than on correcting the model's misinterpretations of ambiguous instructions.

The best prompting advice is also the simplest: say what you mean. State the language, the framework, the pattern, the constraints, the edge cases, the examples, the non-functional requirements. Point to existing code that exemplifies what you want. Describe the problem when you are not sure of the solution. Prescribe the solution when you are. Ask the model to explain its choices when you are unsure. Give targeted feedback when the output is close but not right. Treat the interaction as a conversation with a very fast, very knowledgeable, but very literal colleague who will do exactly what you ask and nothing more. The model does not read between the lines, does not share your aesthetic preferences by default, does not know your team's conventions unless you tell it or show it, and does not handle ambiguity the way a human collaborator would, by asking clarifying questions. It resolves ambiguity by guessing, and it guesses based on the statistical distribution of code it has seen, which may or may not match your specific situation. Your job as the prompter is to eliminate as much ambiguity as possible so that the model's guesses converge with your intentions. That convergence is not a matter of luck; it is a skill, and like all skills, it improves with practice, attention, and a willingness to reflect on what worked and what did not.

There is one more dimension to effective prompting that deserves attention, and it concerns the relationship between the prompt and the CLAUDE.md file. Everything I have described in this essay, the conventions, the patterns, the constraints, the examples, the non-functional requirements, can be encoded once in your CLAUDE.md file and then inherited by every prompt in every session. Developers who invest time in writing a thorough CLAUDE.md find that their prompts get shorter and simpler over time, because the persistent context file carries the load that individual prompts would otherwise need to bear. "Add a GET endpoint for user profiles" becomes a sufficient prompt when the CLAUDE.md file already specifies the URL pattern, the response envelope format, the error handling strategy, the data access layer, and the testing expectations. The prompt and the CLAUDE.md file form a system, and the most effective practitioners design that system deliberately, placing stable, reusable context in the file and reserving the prompt for the specific, variable aspects of each task. This division of labor between persistent context and per-task instruction is the architectural principle that makes AI-assisted development scale from individual prompts to sustained, productive working sessions.
