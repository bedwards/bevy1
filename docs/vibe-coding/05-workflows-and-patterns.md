# Practical Workflows and Patterns for AI-Assisted Development

The previous essays in this series addressed the conceptual foundations of vibe coding, the tools available in Claude Code, the art of prompting, and the dynamics of human-AI collaboration. This essay turns to the concrete: the specific workflows and patterns that developers use in practice to get productive work done with AI coding agents. These patterns are not mutually exclusive; most experienced practitioners mix and match them depending on the task at hand, the state of the codebase, and the level of confidence they have in the agent's ability to handle the work autonomously. What they share is a deliberate structure, a way of organizing the interaction between human and machine that makes the collaboration more predictable, more verifiable, and more effective than unstructured prompting.

The workflows described here have emerged from the collective experience of thousands of developers working with tools like Claude Code, Cursor, GitHub Copilot, and Aider over the past two years. Some of them are documented in official guides and tutorials; others are folk practices passed through blog posts, conference talks, and team Slack channels. All of them are living patterns that continue to evolve as the tools improve and the community discovers new ways to use them.

## Plan Then Execute

The most fundamental workflow pattern is to separate planning from execution, and while this sounds obvious, the temptation to skip the planning phase is strong because the tools make execution so fast. When you can go from prompt to working code in thirty seconds, spending five minutes thinking about what to build feels like waste. It is not. The planning phase is where you avoid the kind of mistakes that are cheap to prevent and expensive to fix: wrong architectural choices, missing requirements, incorrect assumptions about data shapes, and misalignment with the rest of the codebase.

In practice, the plan-then-execute workflow begins with a conversation in which you describe what you want to build and ask the agent to propose an approach before writing any code. You might say something like "I need to add real-time notification support to the application. Before you write any code, analyze the existing architecture, identify the best integration points, and propose a plan including which files you would modify, what new files you would create, and what patterns you would follow." The agent reads the codebase, reasons about the architecture, and presents a plan. You review the plan, ask questions, suggest modifications, and reach agreement on the approach. Only then do you tell the agent to proceed with implementation.

This workflow has several advantages that compound over the life of a project. First, it surfaces misunderstandings early, when they are free to correct, rather than late, when they are embedded in code that must be revised or discarded. If the agent misunderstands which notification system you mean, or proposes an approach that conflicts with an architectural decision it was not aware of, the planning conversation reveals that discrepancy before any code is written. Second, the plan serves as documentation of the decision, a record of what was considered, what was chosen, and why. Third, the plan provides a checklist against which you can verify the implementation: once the agent has finished coding, you can compare the result against the plan and confirm that every item was addressed.

A refinement of this pattern that has gained traction in the community is what some teams call spec-driven development, where the plan takes the form of a structured specification document, often a product requirements document or a technical design document, that the AI helps you write before any implementation begins. The specification becomes the authoritative statement of intent, and all subsequent implementation and testing is evaluated against it. This approach is particularly valuable for features that span multiple files, services, or team boundaries, where a clear written specification prevents the kind of drift that occurs when the intent exists only in someone's head and must be reconstructed from code after the fact.

The plan-then-execute pattern also maps naturally to Claude Code's plan mode, which allows the agent to analyze the codebase and propose changes without executing any of them. You can start a session in plan mode, have the agent read the relevant code and propose its approach, switch to normal or accept-edits mode once you approve the plan, and proceed with implementation knowing that the agent and you share a common understanding of the goal.

## Test-Driven AI Development

Test-driven development has always been one of the most effective ways to produce reliable software, and it turns out to be even more valuable in the context of AI-assisted development, because tests provide a machine-readable specification that constrains the agent's output in ways that natural-language prompts alone cannot. The workflow inverts the typical vibe coding sequence: instead of generating implementation code first and tests afterward (or not at all), you generate tests first and implementation second, so the agent writes code to pass tests rather than writing code to match a prose description.

The practical mechanics of test-driven AI development look something like this. You begin by describing the feature in terms of its expected behavior: "When a user submits a payment with a valid card, the system should create a charge, record the transaction, send a confirmation email, and return a success response. When the card is declined, the system should not create a charge, should record the failed attempt, and should return an error with a specific code." You ask the agent to generate test cases that cover these scenarios, including edge cases you specify: expired cards, network timeouts, duplicate submission attempts, amounts exceeding the daily limit. You review the tests, adjust them as needed, and confirm that they accurately represent the intended behavior. Then, and only then, you ask the agent to write the implementation that makes all the tests pass.

This workflow provides a verification layer that is both automatic and precise. If the agent's implementation passes all the tests, you have high confidence that it correctly handles every scenario you specified. If it fails a test, the failure points you directly to the gap between intent and implementation. And because the tests were written (or at least reviewed) before the implementation, they are not contaminated by the implementation's assumptions; they are independent statements of expected behavior against which any implementation can be evaluated.

The teams that report the highest productivity with AI coding agents, the ones who describe their agents as able to "fly" through a project, almost universally have strong test suites. The test suite serves as a safety net that lets the agent take bold action, making large changes, refactoring aggressively, trying novel approaches, without the risk that those actions introduce undetected regressions. An agent working in a codebase with no tests must be conservative, because every change is a gamble; an agent working in a codebase with comprehensive tests can be ambitious, because every change is immediately validated.

A variant of this pattern that works well for bug fixes is what you might call reproduce-then-fix: you first write a test that reproduces the bug, verify that it fails, then ask the agent to make the test pass without breaking any existing tests. This workflow ensures that the fix actually addresses the reported problem and does not introduce new problems, and it leaves behind a regression test that prevents the bug from recurring.

## Exploratory Prototyping

Not every coding task is well-defined enough for plan-then-execute or test-driven development. Sometimes you are exploring a new technology, evaluating a library, investigating a possible approach, or trying to understand a problem well enough to formulate a plan. In these situations, the most productive workflow is unstructured exploration, where you use the AI as a thinking partner and a rapid prototyping tool, generating throwaway code to build understanding rather than production artifacts.

Exploratory prototyping with an AI agent might look like this: "I'm considering using WebSockets for real-time updates instead of our current polling approach. Can you set up a minimal proof of concept with a WebSocket server and a simple client, so I can see how the connection lifecycle works and test the latency?" The agent produces a prototype in minutes, you experiment with it, discover questions you did not know to ask, and feed those questions back into the conversation. "What happens when the connection drops and reconnects? Show me how to handle that." "How would we authenticate WebSocket connections with our existing JWT tokens?" "What's the memory footprint of ten thousand concurrent connections?" Each question generates a new experiment, and each experiment deepens your understanding.

The key to making exploratory prototyping productive is maintaining clear boundaries between exploration and production code. The prototypes generated during exploration are disposable; their purpose is to build your understanding, not to ship. When you have learned enough to make a decision, you discard the prototypes and use the plan-then-execute or test-driven workflow to build the production implementation, now informed by the knowledge you gained during exploration. Developers who blur this boundary, who let exploratory prototypes evolve into production code through incremental patching, end up with systems that carry the technical debt of code that was never designed for the purpose it now serves.

## The Loop Skill and Recurring Tasks

One of the more distinctive workflow patterns enabled by Claude Code is the use of the `/loop` skill to create recurring tasks that run at specified intervals within a session. The basic syntax is straightforward: `/loop 5m check if the staging deployment at port 3000 is responding and report the HTTP status code` creates a task that runs every five minutes, and the agent executes the specified check each time the interval elapses. The interval can be specified in seconds, minutes, hours, or days, and defaults to ten minutes if omitted.

This capability enables a category of workflows that were previously awkward to accomplish without leaving the development environment. Monitoring a deployment while you continue working on other tasks, polling a continuous integration pipeline for completion, watching a log file for specific error patterns, periodically checking whether a long-running data migration has finished: all of these become simple one-line commands that run in the background of your Claude Code session. The tasks fire only while Claude Code is running and idle, and they automatically expire after three days to prevent forgotten loops from consuming resources indefinitely. A session can hold up to fifty scheduled tasks, which is more than enough for any reasonable development workflow.

The loop skill is particularly powerful when combined with hooks. You might set up a loop that runs your test suite every fifteen minutes and a hook that sends you a notification if the tests fail. Or a loop that checks the health of a set of microservices and a hook that automatically files an issue if one of them goes down. These combinations create a lightweight monitoring and automation layer that lives entirely within your development session, no separate infrastructure required.

## Background Agents and Subagents

Beyond the loop skill's session-scoped scheduling, Claude Code supports a more powerful form of parallelism through background agents and subagents. A subagent is a separate Claude Code instance that runs a specific task independently, potentially with its own tool permissions, its own context, and its own isolation boundaries. You can configure subagents in your project's `.claude/agents` directory, defining each one's capabilities, instructions, and constraints through a markdown file with frontmatter.

The most compelling use case for subagents is parallel task execution. When you have multiple independent tasks that do not share state, running them sequentially wastes time because each task waits for the previous one to finish. Subagents let you dispatch multiple tasks simultaneously: one agent writes the implementation, another writes the tests, a third updates the documentation. The parent agent coordinates the work, dispatches the tasks, and reconciles the results when the subagents complete.

The Claude Agent SDK, which is the programmatic interface that powers Claude Code and is available as a Python and TypeScript library, enables even more sophisticated multi-agent architectures. Spotify, for example, used the agent SDK to build a system that automated code migrations across hundreds of repositories, with each agent operating independently on its assigned repository, running the migration, validating the result, and opening a pull request. This kind of large-scale, parallel, automated code modification is impractical with manual development but straightforward when agents can be spawned, configured, and coordinated programmatically.

For individual developers rather than platform teams, the most accessible form of background agents is simply running multiple Claude Code instances in separate terminal windows or tabs, each working on a different task. This approach works but introduces the risk of file conflicts when two agents modify the same file simultaneously, which is where worktrees become essential.

## Worktrees for Parallel Work

Git worktrees are a feature of git that allows you to check out multiple branches of the same repository simultaneously, each in its own directory with its own working tree and index. Claude Code has built-in support for worktrees through the `--worktree` flag (shortened to `-w`), and this support transforms parallel AI development from a risky juggling act into a safe, structured workflow.

When you start Claude Code with the worktree flag, the tool creates an isolated worktree in a subdirectory of your repository, typically under `.claude/worktrees/`, checks out a new branch there, and scopes the entire session to that directory. The key property is isolation: changes made in one worktree are invisible to changes made in another, because each worktree has its own file state and its own branch. You can run three Claude Code instances simultaneously, one building a new feature in one worktree, another fixing a bug in a second worktree, and a third refactoring a module in a third worktree, with no risk of the agents interfering with each other.

The practical workflow for worktree-based parallel development starts with identifying tasks that are genuinely independent, meaning they do not modify the same files or depend on each other's changes. You open a terminal for each task, start Claude Code with the worktree flag and a descriptive name, and give each instance its instructions. The instances run simultaneously, each in its own isolated branch, and when they finish, you review the results independently and merge the branches through your normal git workflow, resolving any conflicts that arise from the parallel work.

Worktrees also solve a problem that arises with long-running agent sessions. If an agent is working on a complex task that takes twenty minutes, you are not blocked from doing other work; you simply open a new worktree and continue with a different task while the first agent finishes. This is the same principle that makes branching valuable in traditional development, the ability to maintain multiple lines of work simultaneously, but applied to the specific challenge of AI-assisted development where an agent might occupy a branch for an extended period.

Subagents can be configured to use worktree isolation automatically by adding `isolation: worktree` to the agent's frontmatter, which means you can build multi-agent workflows where each agent operates in its own branch without any manual worktree management. The parent agent dispatches work, each subagent creates its own worktree, performs its task, and the results are collected and merged. This pattern scales from two parallel tasks on a developer laptop to dozens of parallel tasks in a continuous integration environment.

The combination of all these patterns, planning before execution, testing before implementation, exploring before committing, looping for monitoring, spawning subagents for parallelism, and isolating with worktrees for safety, constitutes the toolkit of workflows that define productive AI-assisted development in 2026. No single pattern is sufficient for every situation, but together they cover the full range of tasks a developer encounters, from greenfield prototyping to production maintenance, from solo weekend projects to coordinated team efforts. The developers who thrive are not the ones who have memorized the most patterns but the ones who have internalized the principles behind them: separate planning from execution so mistakes are caught early, use tests as machine-readable specifications so correctness is verifiable, isolate parallel work so agents do not interfere with each other, and maintain verification layers at every stage so the speed of generation does not outrun the capacity for quality. These principles are not specific to any particular tool or model; they are the general principles of engineering applied to a new and rapidly evolving mode of work, and they will remain relevant long after the specific tools and commands described here have been superseded by their successors.
